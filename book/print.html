AI 시대의 프론트엔드 개발자

학습 개요

이 책은 AI 시대의 프론트엔드 개발자가 무엇을 준비해야 하는지 다룬다. 단순히 최신 프레임워크를 소개하거나 AI 코딩 도구 사용법을 나열하는 책이 아니다. 이 책의 관심사는 더 좁고 더 깊다. AI가 코드를 생성하는 시대에도 인간 개발자가 어떻게 시스템을 이해하고, 설계를 유지하고, 자동생성 코드에 대한 통제권을 잃지 않을 수 있는가다.

프론트엔드 개발은 이미 여러 번 중심축이 바뀌었다. jQuery와 DOM 조작 중심의 시대, SPA와 컴포넌트 중심의 시대, 서버 렌더링과 정적 생성이 다시 중요해진 시대, 그리고 지금의 AI 보조 개발 시대가 이어졌다. 하지만 모든 변화 뒤에도 브라우저는 여전히 코드를 실행하고, 네트워크는 여전히 지연되며, 상태는 여전히 꼬이고, 사용자는 여전히 깨진 화면 앞에서 기다리지 않는다.

따라서 이 책은 세 가지 층으로 구성된다.

첫째, 현재의 프론트엔드 지형을 이해한다. React, Next.js, Vue, Nuxt, Svelte, SvelteKit, Astro, Solid, Angular 같은 도구들이 어떤 문제를 해결하려고 하는지 살펴본다. 중요한 것은 어떤 프레임워크가 이기는지 맞히는 일이 아니라, 프레임워크들이 공통으로 향하는 방향을 읽는 일이다.

둘째, 변하지 않는 기본기를 복원한다. 브라우저 렌더링, 이벤트 루프, HTTP, 캐시, CORS, 쿠키, 보안, 접근성, 성능 같은 주제는 유행과 별개로 프론트엔드 개발자의 판단력을 만든다. AI가 코드를 작성해도 버그의 원인을 설명하고 고칠 책임은 여전히 개발자에게 남는다.

셋째, AI와 공생하는 작업 방식을 익힌다. AI에게 일을 맡기는 능력은 중요하지만, 더 중요한 것은 일을 작게 자르고, 결과를 읽고, 검증하고, 기존 코드베이스의 질서 안에 통합하는 능력이다. 이 책은 프롬프트 작성법만이 아니라 코드 리뷰, 테스트, 타입, 설계 경계, 회귀 체크를 함께 다룬다.

넷째, AI 시대의 일정 산정을 다시 생각한다. AI가 코드 초안을 빠르게 만들수록 일정 산정은 단순 구현 시간 예측에서 벗어나 요구사항의 불확실성, 검증 범위, 통합 비용, 자동생성 코드에 대한 책임을 함께 다루는 활동이 된다. 빠르게 만들 수 있는 것과 책임질 수 있는 것은 다르다.

이 책의 최종 목표는 독자가 “AI를 쓰는 프론트엔드 개발자“를 넘어 “AI가 만든 코드도 자기 코드처럼 책임질 수 있는 프론트엔드 개발자“가 되도록 돕는 것이다.

이 책의 독자

  • AI 코딩 도구를 써보고 있지만 결과물을 완전히 신뢰하기 어려운 프론트엔드 개발자
  • React나 Next.js 같은 프레임워크는 쓰지만 브라우저 내부 동작에 자신이 없는 개발자
  • 큰 코드베이스를 읽고 변경하는 데 어려움을 느끼는 개발자
  • 팀 차원에서 AI 생성 코드를 안전하게 도입하고 싶은 리드 개발자
  • 앞으로의 프론트엔드 커리어에서 무엇을 공부해야 할지 다시 정리하고 싶은 개발자

이 책의 관점

AI 시대에는 코드를 빨리 쓰는 능력의 희소성이 낮아진다. 반대로 문제를 정확히 정의하는 능력, 좋은 구조와 나쁜 구조를 구분하는 능력, 실행 환경을 이해하는 능력, 코드를 읽고 검증하는 능력은 더 중요해진다.

이 책은 프론트엔드 개발자의 역할을 다음과 같이 본다.

  • 구현자: 요구사항을 작동하는 UI로 바꾼다.
  • 해석자: 사용자, 디자이너, 백엔드, 브라우저, AI 사이의 언어를 번역한다.
  • 설계자: 시간이 지나도 변경 가능한 구조를 만든다.
  • 검증자: 생성된 코드가 실제로 맞는지 확인한다.
  • 운영자: 코드가 배포된 뒤에도 성능, 접근성, 보안, 유지보수를 책임진다.
  • 산정자: 빠른 초안과 책임 가능한 제품 코드 사이의 불확실성을 설명한다.

1. 프론트엔드 개발자는 무엇이 달라졌는가

AI 보조 개발 도구가 널리 쓰이기 시작하면서 프론트엔드 개발의 병목은 바뀌고 있다. 예전에는 화면을 만들기 위해 직접 코드를 타이핑하는 시간이 큰 비중을 차지했다. 지금은 컴포넌트 초안, CSS, 테스트 코드, 리팩터링 후보, 문서화, 심지어 작은 기능의 전체 구현까지 AI가 빠르게 제안한다. 속도만 보면 개발자는 이전보다 훨씬 많은 코드를 다룰 수 있게 되었다.

하지만 병목이 사라진 것은 아니다. 병목은 작성에서 판단으로 이동했다. 무엇을 만들지 정확히 정의하는 일, 기존 코드베이스의 질서를 읽는 일, 생성된 코드가 실제 요구사항을 만족하는지 확인하는 일, 운영 중 문제가 생겼을 때 원인을 설명하는 일은 여전히 사람에게 남아 있다. 오히려 생성 속도가 빨라졌기 때문에 판단의 품질은 더 중요해졌다.

프론트엔드 개발자는 이제 두 종류의 코드를 동시에 다룬다. 하나는 사람이 직접 의도를 쌓아가며 작성한 코드다. 다른 하나는 AI가 빠르게 생성했지만 의도와 맥락이 충분히 설명되지 않은 코드다. 둘 다 저장소에 들어오면 같은 제품을 구성한다. 사용자는 어느 코드가 사람이 쓴 것인지 AI가 쓴 것인지 알지 못한다. 장애가 나도 구분하지 않는다. 따라서 자동생성 코드도 결국 팀의 코드이고, 개발자가 책임져야 하는 코드다.

작성자의 역할에서 편집자의 역할로

AI 이전의 개발자는 주로 작성자였다. 요구사항을 읽고, 파일을 만들고, 함수를 작성하고, 컴포넌트를 나누며 코드의 대부분을 직접 생산했다. 물론 코드 리뷰와 유지보수도 있었지만, 코드가 생겨나는 과정에서 개발자의 손과 머리는 밀접하게 붙어 있었다.

AI 이후의 개발자는 작성자이면서 편집자다. 여기서 편집자는 단순히 오탈자를 고치는 사람이 아니다. 편집자는 글의 의도, 구조, 독자, 흐름, 사실성을 판단한다. 프론트엔드 개발에서 편집자 역할은 다음과 같다.

  • AI가 만든 코드가 요구사항을 정확히 해석했는지 확인한다.
  • 기존 아키텍처와 이름 짓기, 상태 관리 방식에 맞는지 본다.
  • 지나치게 큰 컴포넌트, 불필요한 추상화, 중복 상태를 줄인다.
  • 테스트, 타입, 접근성, 성능, 보안 기준으로 검증한다.
  • 변경 이유와 남은 위험을 팀이 이해할 수 있게 기록한다.

이 역할은 더 높은 수준의 역량을 요구한다. 코드 한 줄을 쓰는 능력보다 코드가 들어갈 자리를 판단하는 능력이 중요해진다. AI가 빠르게 만든 초안을 그대로 병합하는 개발자는 코드 생산량은 늘릴 수 있지만 시스템 이해도는 잃을 수 있다. 반대로 AI 초안을 읽고 좋은 구조로 다듬을 수 있는 개발자는 더 넓은 범위의 일을 다룰 수 있다.

AI가 잘하는 일

AI는 패턴이 명확하고 맥락이 충분히 주어진 작업에서 강하다. 프론트엔드에서는 특히 다음 작업에 유용하다.

  • 반복적인 컴포넌트 초안 생성
  • 이미 정해진 디자인 토큰이나 UI 라이브러리를 활용한 마크업 작성
  • 기존 코드와 비슷한 형태의 CRUD 화면 작성
  • 테스트 케이스 초안 작성
  • 타입 정의와 인터페이스 정리
  • 작은 리팩터링 후보 제안
  • 문서화와 주석 정리
  • 에러 메시지, 빈 상태 문구, 접근성 라벨 초안 작성

이런 작업의 공통점은 정답의 형태가 비교적 분명하다는 것이다. 예를 들어 이미 팀에 Button, Modal, Table, FormField 컴포넌트가 있고, 비슷한 화면이 여러 개 있다면 AI는 기존 패턴을 참고해 꽤 괜찮은 초안을 만들 수 있다. 사람이 처음부터 모든 코드를 쓰는 것보다 훨씬 빠르다.

AI는 코드 읽기 보조에도 유용하다. 낯선 코드베이스에서 진입점, 주요 타입, 데이터 흐름, 테스트 위치를 요약하게 할 수 있다. 다만 이 요약은 지도 초안일 뿐이다. 실제 지도는 파일을 열고, 타입을 확인하고, 테스트와 실행 결과를 대조하면서 사람이 완성해야 한다.

AI가 자주 틀리는 일

AI는 그럴듯함을 잘 만든다. 이것은 장점이면서 위험이다. 특히 프론트엔드에서는 눈에 보이는 UI가 만들어지면 내부 구조가 괜찮다고 착각하기 쉽다. 하지만 UI가 보인다는 것과 제품 코드로 들어갈 수 있다는 것은 다르다.

AI가 자주 틀리는 영역은 다음과 같다.

  • 비즈니스 규칙의 미묘한 예외
  • 서버 상태와 클라이언트 상태의 경계
  • 인증, 권한, 개인정보 같은 보안 맥락
  • 접근성 요구사항
  • 브라우저 렌더링 비용과 성능 병목
  • 오래된 응답이 최신 상태를 덮어쓰는 race condition
  • 기존 코드베이스의 암묵적 규칙
  • 팀이 의도적으로 피하고 있는 라이브러리나 패턴

예를 들어 AI는 검색 화면을 만들 때 useState를 여러 개 추가하고, API 호출을 컴포넌트 안에 직접 넣고, 로딩과 에러 상태를 간단히 처리할 수 있다. 작은 데모에서는 잘 작동한다. 하지만 실제 서비스에서는 검색 조건이 URL과 동기화되어야 하고, 뒤로 가기와 새로고침이 동작해야 하며, 느린 네트워크에서 오래된 응답을 버려야 하고, 권한 없는 응답과 빈 결과를 다르게 처리해야 한다. 이 차이를 판단하는 것은 프론트엔드 개발자의 일이다.

속도는 품질이 아니다

AI 도구를 쓰면 기능을 빠르게 만들 수 있다. 그러나 빠른 생성은 품질을 보장하지 않는다. 오히려 생성 속도가 빨라질수록 품질을 확인하는 루프가 더 중요해진다.

프론트엔드 품질은 여러 층으로 구성된다.

  • 기능: 요구사항대로 동작하는가?
  • 구조: 변경 가능한 형태인가?
  • 타입: 데이터 계약이 명확한가?
  • 테스트: 중요한 흐름과 실패 모드가 검증되는가?
  • 접근성: 키보드와 보조 기술로 사용할 수 있는가?
  • 성능: 느린 기기와 네트워크에서도 견딜 수 있는가?
  • 보안: 민감한 값과 사용자 입력을 안전하게 다루는가?
  • 운영성: 문제가 생겼을 때 원인을 추적할 수 있는가?

AI가 기능 초안을 빠르게 만들수록 개발자는 이 층들을 의식적으로 확인해야 한다. 예전에는 코드를 작성하는 동안 자연스럽게 고민하던 것들이, 이제는 리뷰와 검증 단계에서 명시적으로 다뤄져야 한다.

프론트엔드 개발자의 새로운 기본 자세

AI 시대의 프론트엔드 개발자는 AI를 거부할 필요가 없다. 생산성을 높여주는 도구를 쓰지 않을 이유는 없다. 하지만 AI에게 판단까지 넘기면 안 된다.

좋은 기본 자세는 다음과 같다.

  • AI에게 일을 맡기되, 작업 범위를 작게 자른다.
  • 결과를 받기 전에 완료 조건을 정의한다.
  • 생성된 코드를 읽고 설명할 수 있어야 한다.
  • 이해하지 못한 코드는 병합하지 않는다.
  • 테스트와 브라우저 검증을 생략하지 않는다.
  • 기존 코드베이스의 패턴을 새 코드보다 우선한다.
  • 자동생성 코드도 운영 중 장애를 일으킬 수 있다는 사실을 잊지 않는다.

이 책 전체는 이 자세를 구체화하는 과정이다. 프레임워크를 이해하고, 브라우저를 이해하고, 상태와 설계를 이해하고, 코드를 읽고, AI가 만든 결과물을 검증하는 방법을 배운다. AI 시대에 프론트엔드 개발자는 사라지는 것이 아니라 더 선명해진다. 빠른 생성과 엄격한 판단 사이의 균형을 잡는 사람이 필요하기 때문이다.

2. 2020년대 중반의 프레임워크 지형도

프론트엔드 프레임워크를 배울 때 가장 흔한 질문은 “무엇이 가장 인기 있는가“다. 이 질문이 완전히 틀린 것은 아니다. 인기는 생태계, 채용 시장, 문서, 라이브러리, 커뮤니티 규모와 연결된다. 하지만 이 질문만으로는 오래 버티는 판단을 만들 수 없다. 더 좋은 질문은 “이 도구는 어떤 문제를 해결하려고 하는가“다.

프레임워크는 개발자가 브라우저 위에서 애플리케이션을 만들 때 반복해서 마주치는 문제에 대한 답이다. UI를 어떻게 조각낼 것인가, 상태를 어떻게 변경할 것인가, 서버에서 데이터를 언제 가져올 것인가, HTML을 어디서 만들 것인가, 사용자가 상호작용하기 전에 무엇을 미리 준비할 것인가, 팀이 같은 규칙으로 일하게 하려면 어떤 제약을 둘 것인가. 프레임워크의 차이는 결국 이 질문들에 대한 답의 차이다.

AI 시대에는 이 관점이 더 중요하다. AI는 특정 프레임워크의 문법을 빠르게 흉내 낼 수 있다. 하지만 어떤 렌더링 모델이 현재 제품에 맞는지, 어떤 상태 관리 방식이 변경을 견디는지, 어떤 아키텍처가 팀의 속도와 안정성을 동시에 지키는지는 여전히 사람이 판단해야 한다.

React와 Next.js: 생태계와 풀스택 경계

React는 오랫동안 프론트엔드 생태계의 중심에 있었다. React의 힘은 단순히 컴포넌트 문법이 아니라 거대한 생태계, 채용 시장, 디자인 시스템과 상태 관리 도구, 테스트 도구, 문서와 사례의 축적에서 나온다.

Next.js는 React를 기반으로 라우팅, 서버 렌더링, 정적 생성, API route, 서버 컴포넌트, streaming 같은 기능을 통합해 애플리케이션 프레임워크로 확장했다. 이 흐름은 프론트엔드와 백엔드의 경계를 다시 생각하게 만든다. 화면은 브라우저에서만 만들어지는 것이 아니라 서버에서도 만들어지고, 데이터 패칭은 클라이언트 effect 안에서만 일어나지 않는다.

React와 Next.js를 배울 때 중요한 질문은 다음과 같다.

  • 이 컴포넌트는 서버에서 렌더링되어도 되는가?
  • 클라이언트 상태가 필요한 부분은 어디인가?
  • 데이터 패칭은 route 단위인가, 컴포넌트 단위인가?
  • hydration 비용은 어느 정도인가?
  • 서버와 클라이언트 경계에서 어떤 값이 직렬화되는가?

AI에게 Next.js 코드를 생성하게 할 때는 이 경계를 명확히 설명해야 한다. 그렇지 않으면 불필요하게 클라이언트 컴포넌트를 만들거나, 서버에서만 가능한 작업을 브라우저 코드에 넣거나, 캐싱 정책을 놓칠 수 있다.

Vue와 Nuxt: 접근성과 균형

Vue는 템플릿 기반 문법과 반응성 시스템을 통해 많은 개발자에게 접근성이 좋은 프레임워크로 자리 잡았다. Vue의 장점은 HTML과 가까운 템플릿, 명확한 single-file component 구조, 점진적 도입 가능성이다. Nuxt는 Vue 생태계에서 서버 렌더링과 파일 기반 라우팅, 데이터 패칭, 모듈 시스템을 제공한다.

Vue와 Nuxt는 팀이 명확한 구조를 선호하면서도 지나치게 무거운 규율을 원하지 않을 때 좋은 선택지가 될 수 있다. 특히 템플릿, 스타일, 로직이 한 파일 안에서 분명하게 나뉘는 구조는 AI가 코드를 읽고 수정할 때도 장점이 될 수 있다.

다만 어떤 도구든 마찬가지로, 접근성이 좋다는 것이 설계가 자동으로 좋아진다는 뜻은 아니다. Vue에서도 서버 상태와 UI 상태를 섞을 수 있고, 컴포넌트를 지나치게 크게 만들 수 있으며, composable이 무분별하게 늘어날 수 있다.

Svelte와 SvelteKit: 컴파일러 중심 사고

Svelte는 런타임보다 컴파일러에 더 많은 일을 맡기는 접근으로 주목받았다. 개발자는 비교적 간결한 문법으로 컴포넌트를 작성하고, Svelte는 이를 빌드 시점에 효율적인 JavaScript로 변환한다. SvelteKit은 여기에 라우팅, 서버 렌더링, 데이터 로딩, form action 같은 애플리케이션 기능을 제공한다.

Svelte 계열을 이해할 때 중요한 것은 “프레임워크가 브라우저에서 얼마나 많은 일을 하는가“라는 질문이다. 모든 프레임워크는 추상화를 제공하지만, 그 추상화가 런타임에서 작동하는지 빌드 시점에 정리되는지에 따라 성능과 디버깅 경험이 달라진다.

AI 시대에는 간결한 문법이 초안 생성에 유리할 수 있다. 그러나 컴파일러가 제공하는 반응성 규칙을 제대로 이해하지 못하면 생성된 코드가 미묘하게 잘못될 수 있다. 문법이 짧을수록 규칙을 더 정확히 알아야 한다.

Astro와 아일랜드 아키텍처

Astro는 콘텐츠 중심 사이트와 정적 사이트, 문서, 블로그, 마케팅 페이지에서 강점을 보인다. 핵심 아이디어는 모든 페이지를 거대한 클라이언트 애플리케이션으로 만들지 않고, 필요한 상호작용 영역만 선택적으로 hydration하는 것이다. 이를 흔히 아일랜드 아키텍처라고 부른다.

Astro가 던지는 질문은 중요하다. “이 페이지 전체가 정말 JavaScript 애플리케이션이어야 하는가?” 많은 웹 페이지는 대부분 정적 콘텐츠이고, 일부만 상호작용이 필요하다. 이런 경우 불필요한 JavaScript를 줄이는 것이 성능과 사용자 경험에 직접적인 이득을 준다.

프론트엔드 개발자는 SPA를 기본값으로 생각하기 쉽다. 하지만 AI 시대에는 더 많은 코드가 더 빨리 만들어지기 때문에, 오히려 “이 코드는 필요 없는 것 아닌가“라는 질문이 중요해진다.

Angular, Solid, 그리고 다른 선택지

Angular는 명확한 프레임워크 규율과 큰 조직에서의 운영성을 강점으로 한다. 라우팅, 폼, HTTP, 의존성 주입, 빌드 도구 등 많은 요소가 프레임워크 안에 들어 있다. 이는 자유도를 줄이는 대신 팀 단위 일관성을 높인다.

Solid는 세밀한 반응성과 효율적인 업데이트 모델을 강조한다. React와 비슷한 JSX 형태를 사용하면서도 내부 모델은 다르다. 이런 도구는 프론트엔드 생태계가 단일한 방향으로만 가지 않는다는 사실을 보여준다.

이 밖에도 Qwik, Remix, htmx 같은 도구들은 각자 다른 문제의식을 갖고 있다. 모든 도구를 깊이 배울 필요는 없지만, 이들이 던지는 질문은 배울 가치가 있다. hydration을 줄일 수 있는가, 서버와 클라이언트의 협업을 단순화할 수 있는가, HTML의 힘을 더 활용할 수 있는가, 복잡한 SPA가 정말 필요한가.

프레임워크를 비교하는 여섯 가지 축

프레임워크 선택은 취향만으로 할 수 없다. 다음 여섯 가지 축으로 비교하면 더 차분한 판단을 할 수 있다.

첫째, 렌더링 모델이다. CSR, SSR, SSG, ISR, streaming, partial hydration 중 어떤 방식을 지원하고, 기본값은 무엇인가.

둘째, 데이터 패칭 모델이다. 데이터를 route에서 가져오는가, 컴포넌트에서 가져오는가, 서버에서 가져오는가, 클라이언트에서 가져오는가. 캐시와 재검증은 어떻게 표현하는가.

셋째, 상태 관리 모델이다. 지역 상태, 서버 상태, URL 상태, 전역 상태를 어떻게 나누는가. 프레임워크가 제공하는 기본 도구가 있는가.

넷째, 타입 친화성이다. TypeScript와 자연스럽게 결합되는가. 타입 추론이 좋은가. API와 route, form의 계약을 표현하기 쉬운가.

다섯째, 팀 운영성이다. 문서, 채용, 생태계, 업그레이드 비용, 린트와 테스트 도구, 디자인 시스템과의 결합을 고려해야 한다.

여섯째, AI 친화성이다. 코드 패턴이 일관적인가. 타입과 테스트가 충분한가. 파일 구조가 예측 가능한가. AI가 변경할 때 실수해도 검증 루프가 잡아낼 수 있는가.

유행보다 중요한 것

프레임워크는 계속 바뀐다. 하지만 프레임워크가 해결하려는 문제는 오래 남는다. 렌더링을 어디서 할 것인가, 데이터를 언제 가져올 것인가, 상태를 어디에 둘 것인가, 사용자가 상호작용하기 전까지 어떤 비용을 지불할 것인가, 팀이 같은 방식으로 코드를 읽게 하려면 어떤 규칙이 필요한가.

AI 시대의 프론트엔드 개발자는 프레임워크 이름보다 이 질문들을 익혀야 한다. 그래야 React를 쓰든 Vue를 쓰든 Svelte를 쓰든, 새로운 도구가 나오든, 도구의 표면 아래에 있는 설계 선택을 읽을 수 있다.

3. AI 보조 개발의 실제 워크플로

AI를 개발에 활용한다는 말은 너무 넓다. 어떤 사람에게는 자동완성이고, 어떤 사람에게는 챗봇에게 질문하는 일이며, 어떤 사람에게는 에이전트가 파일을 수정하고 테스트까지 실행하는 일이다. 이 책에서는 AI 보조 개발을 “개발자가 목표와 경계를 정의하고, AI가 탐색과 생성과 수정의 일부를 맡고, 개발자가 결과를 검증해 제품 코드에 통합하는 작업 방식“으로 본다.

중요한 점은 AI가 개발 루프 안으로 들어왔다는 것이다. 예전에는 개발자가 문서를 읽고, 코드를 작성하고, 테스트하고, 리뷰를 받았다. 이제는 그 사이사이에 AI가 들어온다. 요구사항을 정리할 때, 기존 코드를 읽을 때, 초안을 만들 때, 테스트를 추가할 때, 리뷰 체크리스트를 만들 때 AI가 도울 수 있다.

하지만 도구가 들어왔다고 루프의 본질이 바뀌지는 않는다. 좋은 개발은 여전히 문제 정의, 설계, 구현, 검증, 기록의 반복이다. AI는 이 반복을 빠르게 만들 수 있지만, 반복 자체를 없애지는 못한다.

나쁜 워크플로: 큰 요청과 무검증 병합

AI를 처음 도입할 때 흔한 방식은 큰 요청을 한 번에 던지는 것이다.

관리자 대시보드를 만들어줘.

이런 요청은 데모를 만들 때는 효과적일 수 있다. AI는 카드, 표, 그래프, 필터, 버튼이 있는 화면을 빠르게 만들어낸다. 하지만 실제 코드베이스에서는 문제가 된다. 어떤 디자인 시스템을 써야 하는지, 데이터는 어디서 오는지, 권한은 어떻게 처리하는지, URL 상태가 필요한지, 테스트는 어떤 수준이어야 하는지, 기존 컴포넌트와 어떻게 연결되는지 알 수 없기 때문이다.

더 위험한 것은 생성된 코드를 충분히 읽지 않고 병합하는 것이다. 빠르게 만들어진 코드는 빠르게 기술 부채가 된다. 특히 프론트엔드에서는 UI가 눈에 보이기 때문에 “대충 된 것 같다“는 착각이 쉽다.

좋은 워크플로: 작은 작업과 명시적 완료 조건

좋은 AI 보조 개발 워크플로는 작업을 작게 자른다.

상품 목록 페이지의 필터 상태를 URL query parameter와 동기화하려고 한다.
기존 `useProductSearchParams` 훅과 `ProductFilterPanel` 컴포넌트를 먼저 읽고,
현재 중복 상태가 있는지 설명한 뒤,
기존 패턴을 유지하면서 변경안을 제안해줘.
완료 조건은 뒤로 가기/새로고침 후 필터 유지, 빈 값 제거, 기존 테스트 통과다.

이 요청은 길지만 좋은 요청이다. 목표, 변경 범위, 읽어야 할 코드, 유지해야 할 패턴, 완료 조건이 들어 있다. AI가 더 좋은 결과를 내기 쉬울 뿐 아니라, 사람이 결과를 검토하기도 쉽다.

좋은 워크플로는 보통 다음 순서를 따른다.

  1. 작업 헌장을 쓴다.
  2. 기존 코드 탐색을 먼저 시킨다.
  3. 변경 계획을 받는다.
  4. 작은 단위로 구현한다.
  5. 생성된 코드를 사람이 읽는다.
  6. 테스트와 브라우저로 검증한다.
  7. 변경 요약과 남은 위험을 기록한다.

작업 헌장 작성하기

작업 헌장은 AI에게 주는 작은 명세다. 복잡할 필요는 없지만 다음 항목은 들어가면 좋다.

목표:
사용자 관점에서 무엇이 가능해져야 하는가?

변경 범위:
어떤 화면, 컴포넌트, 파일, API를 다룰 수 있는가?

기존 패턴:
따라야 할 컴포넌트, 훅, 타입, 테스트 스타일은 무엇인가?

완료 조건:
어떤 동작이 확인되어야 끝났다고 볼 수 있는가?

검증 방법:
어떤 테스트, 타입 검사, 브라우저 확인을 해야 하는가?

제약:
새 의존성 추가 금지, 특정 파일 수정 금지, API 변경 금지 같은 조건은 무엇인가?

작업 헌장은 AI를 위한 문서이면서 사람을 위한 문서다. 작업을 시작하기 전에 이 정도를 쓰지 못한다면, 사실 사람도 아직 문제를 충분히 이해하지 못한 것이다.

탐색과 구현을 분리하기

AI에게 바로 코드를 고치게 하기보다 먼저 탐색을 맡기는 것이 좋다. 예를 들어 다음처럼 요청할 수 있다.

아직 코드는 수정하지 말고, 이 기능과 관련된 파일과 데이터 흐름을 찾아줘.
어떤 컴포넌트가 진입점인지, 상태가 어디에 있는지, 테스트가 어디에 있는지 요약해줘.

탐색과 구현을 분리하면 두 가지 이점이 있다. 첫째, AI가 잘못된 파일을 수정할 가능성이 줄어든다. 둘째, 개발자가 코드베이스를 함께 이해하게 된다. AI의 요약이 틀렸다면 구현 전에 바로잡을 수 있다.

탐색 결과는 항상 실제 파일과 대조해야 한다. AI가 “이 훅이 서버 상태를 관리한다“고 말했더라도, 코드를 열어보면 단순한 UI 상태일 수 있다. 요약은 출발점이지 증거가 아니다.

AI에게 초안을 맡기기 좋은 작업

AI에게 맡기기 좋은 작업은 다음과 같다.

  • 기존 패턴을 따르는 컴포넌트 초안
  • 타입과 인터페이스 정리
  • 테스트 케이스 초안
  • Storybook story 또는 사용 예시 작성
  • 반복적인 스타일 변환
  • 간단한 리팩터링
  • 문서화와 체크리스트 작성

이 작업들은 사람이 검토하기 쉽고, 실패했을 때 되돌리기도 쉽다. 특히 기존 코드 패턴이 명확하면 AI는 더 안정적인 결과를 낸다. 따라서 AI 친화적인 코드베이스를 만들고 싶다면, 먼저 사람이 읽기 좋은 코드베이스를 만들어야 한다. 명확한 이름, 작은 컴포넌트, 일관된 파일 구조, 좋은 타입, 테스트가 AI에게도 좋은 컨텍스트가 된다.

사람이 반드시 붙잡아야 할 작업

반대로 다음 작업은 사람이 최종 판단을 해야 한다.

  • 제품 요구사항의 우선순위 결정
  • 사용자 경험의 미묘한 판단
  • 인증, 권한, 개인정보 처리
  • 아키텍처 경계 변경
  • 새 의존성 도입
  • 성능과 접근성의 최종 승인
  • 배포와 운영 리스크 판단

AI는 선택지를 제안할 수 있다. 하지만 “우리 제품에서 이 선택이 맞는가“는 제품과 팀의 맥락을 아는 사람이 판단해야 한다. AI는 저장소의 모든 암묵적 이유를 알지 못한다. 어떤 코드는 과거 장애 때문에 일부러 그렇게 되어 있을 수 있고, 어떤 패턴은 팀의 역량과 운영 방식 때문에 선택되었을 수 있다.

검증 루프를 기본값으로 만들기

AI 보조 개발에서 가장 중요한 습관은 검증 루프를 기본값으로 만드는 것이다. 생성이 끝이 아니라 검증의 시작이어야 한다.

프론트엔드 작업의 검증 루프는 보통 다음을 포함한다.

  • 타입 검사
  • 린트
  • 단위 또는 컴포넌트 테스트
  • 브라우저에서 실제 사용자 흐름 확인
  • 접근성 기본 점검
  • 성능에 영향을 줄 수 있는 변경 확인
  • diff 리뷰

항상 모든 검증을 다 할 수는 없다. 하지만 어떤 검증을 생략했는지 알고 있어야 한다. “테스트는 돌리지 못했지만 타입 검사는 통과했고, 브라우저에서 주요 흐름은 확인했다“와 “그냥 AI가 해줬다“는 완전히 다르다.

결과 보고 형식

AI와 협업할 때는 마지막 요약도 중요하다. 좋은 결과 보고는 다음 내용을 포함한다.

  • 무엇을 변경했는가
  • 왜 그렇게 변경했는가
  • 어떤 검증을 했는가
  • 어떤 검증을 하지 못했는가
  • 남은 위험은 무엇인가

이 형식은 사람 개발자에게도 좋다. 코드 리뷰어는 변경의 의도를 빠르게 이해할 수 있고, 나중에 문제가 생겼을 때 추적하기 쉽다.

4. 브라우저 렌더링 파이프라인

프론트엔드 개발자가 브라우저를 모르면 화면 버그를 설명할 수 없다. 프레임워크는 개발자에게 편리한 추상화를 제공하지만, 실제 화면은 여전히 브라우저가 만든다. React 컴포넌트든 Vue 템플릿이든 Svelte 컴포넌트든, 마지막에는 HTML, CSS, JavaScript가 브라우저 안에서 해석되고 픽셀로 바뀐다.

AI가 만든 코드도 이 규칙을 벗어나지 않는다. AI는 그럴듯한 컴포넌트를 빠르게 만들 수 있지만, 그 컴포넌트가 레이아웃을 얼마나 자주 다시 계산하게 하는지, 이미지를 로드하면서 화면을 얼마나 밀어내는지, 애니메이션이 메인 스레드를 막는지는 자동으로 보장하지 않는다. 화면이 느리거나 흔들리거나 깜박이는 문제를 해결하려면 브라우저의 렌더링 파이프라인을 알아야 한다.

HTML에서 픽셀까지

브라우저가 화면을 만드는 과정은 단순히 HTML을 위에서 아래로 그리는 일이 아니다. 대략 다음 흐름을 거친다.

  1. HTML을 파싱해 DOM을 만든다.
  2. CSS를 파싱해 CSSOM을 만든다.
  3. DOM과 CSSOM을 결합해 렌더 트리를 만든다.
  4. 각 요소의 위치와 크기를 계산한다.
  5. 요소의 색, 글자, 테두리, 그림자 등을 페인트한다.
  6. 여러 레이어를 합성해 최종 화면을 만든다.

이 흐름을 정확한 내부 구현 수준까지 외울 필요는 없다. 하지만 어떤 변경이 어느 단계에 영향을 주는지는 알아야 한다. 예를 들어 텍스트 색을 바꾸는 것은 대체로 레이아웃을 다시 계산할 필요가 없지만, 요소의 너비를 바꾸면 주변 요소의 위치까지 다시 계산해야 할 수 있다. transform으로 이동시키는 애니메이션과 top이나 left를 계속 바꾸는 애니메이션은 브라우저가 부담하는 비용이 다르다.

DOM과 CSSOM

DOM은 HTML 문서의 구조를 표현한다. CSSOM은 CSS 규칙을 구조화한 것이다. 브라우저는 이 둘을 함께 사용해 실제로 보이는 요소를 결정한다. display: none인 요소는 DOM에는 있을 수 있지만 렌더 트리에는 포함되지 않는다. 반대로 visibility: hidden인 요소는 보이지 않지만 공간은 차지한다.

이 차이는 UI 버그를 설명할 때 중요하다. 예를 들어 AI가 모달을 만들면서 닫힌 상태의 모달을 단순히 투명하게만 처리했다면, 화면에는 보이지 않아도 포커스나 클릭 흐름에 영향을 줄 수 있다. 접근성과 레이아웃을 함께 고려해야 한다.

CSS도 단순한 장식이 아니다. CSS 선택자의 범위, cascade, specificity, layout model, stacking context는 모두 실제 화면의 동작에 영향을 준다. AI가 만든 스타일이 기존 디자인 시스템의 규칙을 덮어쓰거나, 의도치 않은 stacking context를 만들어 dropdown이 뒤에 깔리는 문제도 자주 생긴다.

레이아웃: 위치와 크기를 계산하는 비용

레이아웃은 각 요소가 어디에 있고 얼마나 큰지 계산하는 단계다. 이 단계는 비쌀 수 있다. 특히 어떤 JavaScript 코드가 DOM의 크기를 읽고 곧바로 스타일을 쓰는 일을 반복하면 브라우저는 최신 값을 알기 위해 레이아웃을 강제로 계산해야 할 수 있다.

예를 들어 다음과 같은 패턴은 조심해야 한다.

items.forEach((item) => {
  const height = item.getBoundingClientRect().height;
  item.style.marginTop = `${height / 2}px`;
});

이 코드는 읽기와 쓰기가 반복된다. 브라우저는 스타일 변경이 다음 측정값에 영향을 줄 수 있기 때문에 불필요한 레이아웃 계산을 하게 될 수 있다. 더 나은 방식은 먼저 필요한 값을 모두 읽고, 그다음 스타일을 한꺼번에 쓰는 것이다.

AI가 스크롤 위치 계산, 드래그 앤 드롭, 가상 목록, 툴팁 위치 조정 코드를 만들 때 이런 문제가 생기기 쉽다. 코드가 맞아 보여도 성능이 나쁠 수 있다.

페인트와 컴포지팅

페인트는 요소의 시각적 표현을 그리는 단계다. 배경색, 텍스트, 테두리, 그림자, 이미지 등이 여기에 포함된다. 컴포지팅은 여러 레이어를 합쳐 최종 화면을 만드는 단계다.

애니메이션을 만들 때는 어떤 속성을 바꾸는지가 중요하다. 대체로 transform과 opacity는 레이아웃과 페인트 비용을 줄이기 쉬운 속성이다. 반면 width, height, top, left, box-shadow 같은 속성을 매 프레임 바꾸면 더 많은 비용이 들 수 있다.

물론 “항상 transform만 써라” 같은 절대 규칙은 위험하다. 모든 최적화는 실제 측정과 함께 판단해야 한다. 하지만 최소한 AI가 만든 애니메이션 코드가 레이아웃을 계속 흔드는지, 브라우저가 매 프레임 많은 일을 하게 만드는지는 확인해야 한다.

이미지와 폰트가 만드는 흔들림

프론트엔드에서 흔한 성능 문제 중 하나는 레이아웃 이동이다. 이미지의 크기를 지정하지 않으면 브라우저는 이미지가 로드되기 전까지 공간을 얼마나 확보해야 할지 모른다. 뒤늦게 이미지 크기가 결정되면 아래 콘텐츠가 밀려난다. 사용자는 버튼을 누르려다가 다른 요소를 누를 수 있다.

폰트도 비슷하다. 웹 폰트가 늦게 로드되면 fallback font로 먼저 렌더링한 뒤 글자 폭이 달라져 레이아웃이 바뀔 수 있다. 이런 문제는 “화면은 결국 보인다“는 관점에서는 작아 보이지만, 실제 사용자 경험에서는 매우 거슬린다.

AI가 카드 목록을 만들 때 img 태그에 width와 height를 지정하지 않거나, 반응형 컨테이너의 aspect ratio를 잡지 않는 경우가 있다. 이때 개발자는 단순히 화면이 예쁜지보다 레이아웃이 안정적인지 봐야 한다.

JavaScript와 렌더링의 관계

JavaScript는 브라우저의 메인 스레드에서 많은 일을 한다. 긴 JavaScript 실행은 렌더링과 사용자 입력 처리를 지연시킬 수 있다. 사용자가 버튼을 눌렀는데 반응이 늦다면, 네트워크만 문제가 아닐 수 있다. JavaScript가 너무 오래 실행 중일 수 있다.

프론트엔드 프레임워크도 결국 JavaScript 위에서 동작한다. 상태가 바뀌면 컴포넌트가 다시 계산되고, DOM 변경이 만들어지며, 브라우저가 렌더링을 갱신한다. 불필요한 상태 변경이나 큰 컴포넌트 트리의 잦은 재계산은 사용자 경험에 영향을 준다.

AI가 만든 코드는 종종 편의를 위해 모든 값을 한 컴포넌트에 몰아넣거나, 렌더링마다 비싼 계산을 다시 수행하거나, 배열 전체를 매번 새로 만드는 식의 코드를 낸다. 작은 예제에서는 문제 없어 보이지만, 데이터가 커지면 문제가 된다.

Core Web Vitals를 읽는 이유

Core Web Vitals는 사용자의 경험을 성능 지표로 바라보려는 시도다. LCP는 주요 콘텐츠가 얼마나 빨리 보이는지, INP는 상호작용에 얼마나 빨리 반응하는지, CLS는 화면이 얼마나 안정적인지를 본다.

이 지표를 숫자 암기 대상으로 볼 필요는 없다. 중요한 것은 각 지표가 어떤 사용자 경험을 나타내는지 이해하는 것이다.

  • LCP가 나쁘다면 주요 이미지, 서버 응답, 렌더링 차단 리소스, 클라이언트 렌더링 비용을 의심한다.
  • INP가 나쁘다면 긴 JavaScript 작업, 무거운 이벤트 핸들러, 불필요한 렌더링을 의심한다.
  • CLS가 나쁘다면 이미지 크기, 광고/동적 콘텐츠 삽입, 폰트 로딩, layout placeholder를 의심한다.

AI에게 “성능을 개선해줘“라고 요청하는 것보다 “이 화면의 LCP를 나쁘게 만들 수 있는 요소를 찾아줘“라고 요청하는 편이 낫다. 성능도 구체적인 질문으로 바꿔야 다룰 수 있다.

AI 생성 코드 리뷰 질문

브라우저 렌더링 관점에서 AI가 만든 UI를 리뷰할 때는 다음 질문을 던진다.

  • 이미지나 비디오의 크기, 비율, placeholder가 안정적으로 잡혀 있는가?
  • 애니메이션이 layout을 계속 바꾸는 속성을 사용하지 않는가?
  • 스크롤, resize, mousemove 같은 빈번한 이벤트에서 비싼 작업을 하지 않는가?
  • DOM 측정과 스타일 쓰기가 반복적으로 섞여 있지 않은가?
  • 닫힌 modal, dropdown, tooltip이 포커스와 stacking context를 망치지 않는가?
  • 렌더링마다 큰 배열 변환이나 비싼 계산을 다시 하지 않는가?
  • 실제 브라우저 성능 도구로 확인할 수 있는가?

이 질문들은 프론트엔드 개발자에게 방어력을 준다. AI가 코드를 빠르게 만들더라도, 브라우저가 그 코드를 어떻게 실행하는지 설명할 수 있으면 통제권을 잃지 않는다.

5. 자바스크립트 런타임과 이벤트 루프

프론트엔드 버그의 상당수는 비동기 흐름을 잘못 이해해서 생긴다. 사용자는 버튼을 클릭하고, 네트워크 요청은 지연되고, 상태는 여러 번 바뀌며, 렌더링은 그 사이에 일어난다. 이 과정을 이해하려면 자바스크립트 런타임과 이벤트 루프를 알아야 한다.

AI가 만든 코드는 문법적으로는 맞아도 시간의 흐름을 틀릴 수 있다. 요청 A가 먼저 시작했지만 요청 B보다 늦게 끝날 수 있다. 컴포넌트가 사라진 뒤에도 비동기 작업이 상태를 바꾸려 할 수 있다. 사용자가 버튼을 두 번 눌러 같은 요청이 중복 전송될 수 있다. 이런 문제는 타입만으로 잡히지 않는다.

콜 스택과 실행의 기본

자바스크립트는 한 번에 하나의 실행 흐름을 처리한다. 함수가 호출되면 콜 스택에 쌓이고, 함수 실행이 끝나면 스택에서 빠진다. 이 모델은 단순하지만 브라우저 환경에서는 사용자 입력, 네트워크, 타이머, 렌더링 같은 일이 함께 일어나기 때문에 비동기 모델이 필요하다.

다음 코드를 보자.

console.log("A");

setTimeout(() => {
  console.log("B");
}, 0);

Promise.resolve().then(() => {
  console.log("C");
});

console.log("D");

출력 순서는 A, D, C, B다. setTimeout의 지연 시간이 0이어도 즉시 실행되는 것은 아니다. Promise의 .then은 마이크로태스크로 처리되고, 타이머 콜백은 태스크 큐에서 처리된다. 이 차이를 모르면 UI 업데이트와 비동기 콜백의 순서를 잘못 예상하게 된다.

태스크와 마이크로태스크

브라우저는 이벤트 루프를 통해 여러 작업을 조율한다. 사용자 클릭, 타이머, 네트워크 콜백 같은 작업은 태스크로 처리된다. Promise callback, queueMicrotask 같은 작업은 마이크로태스크로 처리된다. 보통 하나의 태스크가 끝나면 마이크로태스크 큐가 비워지고, 그다음 브라우저가 렌더링할 기회를 얻는다.

이 모델은 프론트엔드 코드에서 중요하다. 마이크로태스크를 너무 많이 이어붙이면 브라우저가 렌더링할 기회를 늦게 얻을 수 있다. 반대로 렌더링 이후에 해야 하는 작업을 너무 일찍 실행하면 DOM이 기대한 상태가 아닐 수 있다.

프레임워크는 이런 세부 사항을 일부 감춰준다. 하지만 감춰준다는 것은 사라졌다는 뜻이 아니다. React의 상태 업데이트, Vue의 nextTick, Svelte의 tick 같은 API도 결국 렌더링 타이밍을 다루기 위한 도구다.

렌더링과 requestAnimationFrame

requestAnimationFrame은 브라우저가 다음 페인트 전에 실행할 콜백을 등록하는 API다. 애니메이션이나 화면 측정이 필요한 작업에서 유용하다. 예를 들어 요소를 DOM에 추가한 뒤 실제 크기를 측정하고, 그 측정값으로 애니메이션을 시작해야 할 때 렌더링 타이밍을 고려해야 한다.

AI가 애니메이션이나 스크롤 동작을 구현할 때 단순히 setTimeout을 사용하는 경우가 있다. 때로는 충분하지만, 화면 갱신과 맞물린 작업에서는 requestAnimationFrame이 더 적절할 수 있다. 중요한 것은 API 이름을 외우는 것이 아니라 “이 작업은 브라우저의 렌더링 타이밍과 어떤 관계가 있는가“를 묻는 것이다.

비동기 요청과 race condition

검색 자동완성을 생각해보자. 사용자가 r, re, rea, react를 빠르게 입력하면 여러 요청이 순서대로 시작될 수 있다. 하지만 응답은 시작한 순서대로 돌아오지 않는다. rea 요청이 react 요청보다 늦게 끝나면 오래된 결과가 최신 화면을 덮어쓸 수 있다.

이 문제를 race condition이라고 부른다. AI가 만든 검색 코드는 종종 다음처럼 단순하다.

async function onSearch(keyword) {
  setLoading(true);
  const result = await searchProducts(keyword);
  setItems(result.items);
  setLoading(false);
}

이 코드는 성공 케이스에서는 잘 작동한다. 그러나 요청 취소, 오래된 응답 무시, 컴포넌트 unmount, 에러 상태를 고려하지 않는다. 실무에서는 다음 중 하나 이상의 전략이 필요하다.

  • AbortController로 이전 요청을 취소한다.
  • 요청 id를 기록해 최신 요청의 응답만 반영한다.
  • TanStack Query나 SWR 같은 서버 상태 도구에 캐시와 동기화를 맡긴다.
  • 입력값 변경과 실제 요청 사이에 debounce를 둔다.

AI에게 검색 기능을 만들게 할 때는 “빠른 입력에서 오래된 응답이 최신 결과를 덮어쓰지 않게 해줘“라고 명시해야 한다.

stale closure

프론트엔드에서 자주 생기는 또 다른 문제는 stale closure다. 함수가 만들어진 시점의 값을 기억하고, 나중에 실행될 때 최신 값을 보지 못하는 상황이다. React 훅을 사용하는 코드에서 특히 자주 마주친다.

예를 들어 interval 안에서 상태를 읽는 코드가 처음 렌더링의 값을 계속 참조할 수 있다. 이벤트 리스너를 등록하면서 최신 props나 state를 반영하지 못할 수도 있다. AI가 effect dependency를 대충 비워두거나, lint 경고를 피하려고 의존성을 제거하면 이런 문제가 생긴다.

stale closure를 피하려면 다음 질문을 해야 한다.

  • 이 콜백은 언제 만들어지고 언제 실행되는가?
  • 콜백 안에서 읽는 값은 최신이어야 하는가?
  • 의존성 배열이 실제 의존성을 반영하는가?
  • ref를 써야 하는 상황인가, 아니면 구조를 바꿔야 하는가?

AI가 만든 훅 코드는 반드시 effect dependency와 cleanup을 확인해야 한다.

중복 제출과 사용자 입력

폼 제출도 비동기 흐름의 전형적인 문제다. 사용자가 저장 버튼을 두 번 누르면 요청이 두 번 나갈 수 있다. 첫 번째 요청은 실패하고 두 번째 요청은 성공할 수도 있다. 낙관적 업데이트를 했다가 실패했을 때 UI를 되돌리지 않을 수도 있다.

좋은 폼 제출 흐름은 다음 상태를 명시적으로 다룬다.

  • idle
  • validating
  • submitting
  • success
  • error

상태를 단순히 isLoading 하나로 끝내면 에러 메시지, 중복 클릭 방지, 재시도, 제출 후 이동 같은 흐름이 뒤엉킨다. AI가 만든 코드가 try/catch와 setLoading(false)만으로 끝난다면, 실제 사용자 흐름을 충분히 다루는지 확인해야 한다.

취소와 cleanup

프론트엔드 코드는 컴포넌트 생명주기와 함께 움직인다. 사용자가 페이지를 떠나면 진행 중인 요청은 더 이상 필요 없을 수 있다. 모달이 닫히면 이벤트 리스너를 제거해야 한다. 타이머를 만들었다면 cleanup해야 한다.

AbortController는 fetch 요청을 취소할 때 유용하다. effect cleanup은 이벤트 리스너, interval, observer를 정리할 때 필요하다. AI는 종종 “추가하는 코드“는 잘 만들지만 “정리하는 코드“를 빼먹는다. 프론트엔드에서는 빼먹은 cleanup이 메모리 누수, 중복 이벤트, 이상한 상태 업데이트로 이어질 수 있다.

상태 전이를 명시하기

비동기 UI는 시간에 따라 상태가 바뀐다. 이 상태 전이를 코드에서 명시하면 버그를 줄일 수 있다. 단순한 기능은 boolean 몇 개로 충분하지만, 복잡한 기능은 union type이나 state machine이 더 적합할 수 있다.

예를 들어 다음 타입은 loading과 error가 동시에 true가 되는 모순을 막는다.

type AsyncState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: Error };

AI에게 “로딩과 에러 상태를 처리해줘“라고 말하면 여러 boolean을 만들 수 있다. “상태가 서로 배타적인 union type으로 표현되게 해줘“라고 말하면 더 안전한 구조를 얻기 쉽다.

AI 생성 코드 리뷰 질문

자바스크립트 런타임과 이벤트 루프 관점에서 AI가 만든 코드를 리뷰할 때는 다음을 확인한다.

  • 요청이 취소 가능하거나 오래된 응답을 무시하는가?
  • 컴포넌트가 사라진 뒤 상태를 업데이트하지 않는가?
  • 이벤트 리스너, 타이머, observer가 cleanup되는가?
  • effect dependency가 실제 의존성을 반영하는가?
  • 중복 클릭과 중복 제출을 막는가?
  • 로딩, 성공, 실패, 취소 상태가 구분되는가?
  • 렌더링마다 비싼 비동기 작업을 새로 시작하지 않는가?
  • Promise rejection이 조용히 사라지지 않는가?

비동기 버그는 재현이 어렵고, 사용자의 기기와 네트워크 상태에 따라 다르게 나타난다. 그래서 더더욱 처음부터 시간의 흐름을 설계해야 한다. AI는 코드를 빠르게 만들지만, 시간의 의미는 개발자가 붙잡아야 한다.

6. 네트워크, 보안, 스토리지 기본기

프론트엔드는 더 이상 “화면만 만드는 영역“이 아니다. 인증, 세션, 토큰, 쿠키, 캐시, API 에러, 보안 헤더, 개인정보 노출은 모두 프론트엔드 코드와 연결된다. AI가 만든 코드가 겉으로는 편리해 보여도, 토큰을 localStorage에 무심코 저장하거나 민감한 값을 클라이언트 번들에 노출하면 심각한 문제가 된다.

이 장의 목표는 프론트엔드 개발자가 보안 전문가가 되는 것이 아니다. 최소한 어떤 질문을 해야 하는지 아는 것이다. 이 값은 브라우저에 저장해도 되는가. 이 API 응답은 캐시되어도 되는가. 이 요청은 인증 정보를 포함하는가. 사용자가 입력한 값은 안전하게 렌더링되는가. 에러 메시지가 민감한 정보를 노출하지 않는가. 이런 질문이 있어야 AI가 만든 코드를 검토할 수 있다.

HTTP는 UI의 일부다

프론트엔드 개발자는 HTTP를 화면 밖의 일로 생각하기 쉽다. 하지만 HTTP는 UI의 일부다. API가 느리면 로딩 상태가 필요하다. 401 응답은 로그인 흐름과 연결된다. 403 응답은 권한 안내와 연결된다. 404 응답은 빈 상태와 다르다. 500 응답은 재시도와 장애 안내가 필요하다.

상태 코드를 단순히 성공과 실패로만 나누면 사용자 경험이 거칠어진다.

  • 200: 정상 응답
  • 201: 생성 완료
  • 204: 응답 본문 없는 성공
  • 400: 잘못된 요청
  • 401: 인증 필요
  • 403: 권한 없음
  • 404: 리소스 없음
  • 409: 충돌
  • 422: 유효성 검증 실패
  • 500: 서버 오류

AI가 만든 API 호출 코드는 종종 catch에서 “오류가 발생했습니다“만 보여준다. 실제 제품에서는 응답의 의미를 구분해야 한다. 특히 폼에서는 422 응답이 필드별 에러로 연결되어야 하고, 401은 로그인 갱신이나 재인증으로 이어질 수 있다.

캐시는 성능이면서 정확성 문제다

캐시는 빠른 UI를 만드는 데 중요하다. 하지만 잘못된 캐시는 틀린 정보를 보여준다. 프론트엔드 개발자는 브라우저 캐시, HTTP cache header, 서비스 워커 캐시, 서버 상태 라이브러리의 메모리 캐시를 구분할 수 있어야 한다.

예를 들어 상품 목록은 짧은 시간 캐시해도 괜찮을 수 있다. 반면 계정 권한, 결제 상태, 개인정보는 오래 캐시하면 위험할 수 있다. 검색 결과는 query parameter와 함께 캐시 키가 만들어져야 한다. 필터가 바뀌었는데 같은 캐시 키를 사용하면 이전 결과가 계속 보일 수 있다.

AI에게 서버 상태 코드를 만들게 할 때는 캐시 키와 재검증 조건을 명시해야 한다.

검색어, 카테고리, 가격 범위, 정렬, 페이지가 모두 query key에 포함되어야 한다.
즐겨찾기 변경 후 현재 목록 query를 무효화하거나 optimistic update를 일관되게 처리해야 한다.

캐시는 성능 최적화가 아니라 데이터 정확성의 일부다.

CORS는 보안 기능이지 장애물이 아니다

CORS는 브라우저가 다른 origin으로 요청할 때 적용하는 보안 모델이다. 개발 중에는 CORS 에러가 귀찮은 장벽처럼 느껴질 수 있다. 그래서 AI에게 물어보면 “서버에서 모든 origin을 허용하세요” 같은 답을 받을 수도 있다. 그러나 실무에서는 위험한 해결책이다.

CORS를 볼 때는 다음을 확인한다.

  • 어떤 origin을 허용해야 하는가?
  • credential을 포함하는 요청인가?
  • preflight 요청이 필요한가?
  • 허용할 method와 header는 무엇인가?
  • 운영 환경과 개발 환경의 설정이 분리되어 있는가?

프론트엔드 개발자가 CORS 설정을 직접 담당하지 않더라도, 문제를 정확히 설명할 수 있어야 백엔드나 인프라 담당자와 협업할 수 있다. “CORS가 안 돼요“보다 “프론트엔드 origin은 https://app.example.com이고, credential 포함 요청에서 preflight가 실패하고 있습니다“가 훨씬 좋은 설명이다.

쿠키와 토큰 저장

인증 정보를 어디에 저장할지는 중요한 결정이다. localStorage는 사용하기 쉽지만 JavaScript로 접근 가능하다. XSS가 발생하면 저장된 토큰을 읽을 수 있다. HttpOnly 쿠키는 JavaScript에서 읽을 수 없기 때문에 토큰 탈취 위험을 줄일 수 있지만, CSRF와 SameSite 정책을 함께 고려해야 한다.

쿠키를 볼 때 중요한 속성은 다음과 같다.

  • HttpOnly: JavaScript에서 접근할 수 없게 한다.
  • Secure: HTTPS에서만 전송한다.
  • SameSite: cross-site 요청에서 쿠키 전송 정책을 제어한다.
  • Path와 Domain: 쿠키가 전송되는 범위를 정한다.
  • Expires와 Max-Age: 만료 시점을 정한다.

AI가 인증 구현을 제안할 때는 저장 위치를 반드시 확인해야 한다. “JWT를 localStorage에 저장하고 요청마다 Authorization header에 넣는다“는 패턴은 많은 예제에서 보이지만, 모든 서비스에 적합한 답은 아니다. 보안 요구사항, 백엔드 구조, 배포 환경, CSRF 방어 전략에 따라 달라진다.

브라우저 스토리지의 차이

브라우저에는 여러 저장소가 있다. 각각의 용도가 다르다.

  • localStorage: 문자열 기반, 동기 API, 탭을 닫아도 유지된다.
  • sessionStorage: 탭 세션 동안 유지된다.
  • IndexedDB: 더 큰 데이터와 구조화된 저장에 적합하다.
  • Cookie: 요청과 함께 서버로 전송될 수 있다.
  • Cache Storage: 서비스 워커와 함께 리소스 캐싱에 쓰인다.

AI가 “브라우저에 저장하면 된다“고 제안할 때는 무엇을 어디에 저장하는지 물어야 한다. UI 선호도나 임시 draft는 localStorage에 둘 수 있다. 하지만 access token, 주민등록번호, 결제 정보 같은 민감한 값은 그렇게 다루면 안 된다.

또한 localStorage는 동기 API다. 큰 데이터를 자주 읽고 쓰면 메인 스레드에 영향을 줄 수 있다. 저장소 선택은 보안뿐 아니라 성능과 데이터 모델의 문제이기도 하다.

XSS와 사용자 입력

XSS는 사용자가 입력한 값이나 외부 데이터를 안전하지 않게 HTML로 렌더링할 때 발생할 수 있다. 현대 프레임워크는 기본적으로 문자열을 escape해주기 때문에 많은 위험을 줄여준다. 하지만 dangerouslySetInnerHTML, 직접 DOM 조작, markdown 렌더링, 외부 HTML 삽입, URL 생성에서는 여전히 조심해야 한다.

AI는 사용자의 요구가 “HTML을 렌더링해줘“라면 너무 쉽게 raw HTML 삽입 코드를 제안할 수 있다. 이때 개발자는 다음을 확인해야 한다.

  • 이 HTML의 출처는 신뢰할 수 있는가?
  • sanitize를 거쳤는가?
  • 링크의 target="_blank"에 rel="noopener noreferrer"가 필요한가?
  • URL scheme이 javascript: 같은 위험한 값이 될 수 있는가?
  • markdown 렌더러가 raw HTML을 허용하는가?

사용자 입력은 단순한 문자열이 아니다. 브라우저에서 실행될 수 있는 컨텍스트에 들어가면 보안 문제가 된다.

CSRF와 credential 요청

CSRF는 사용자가 로그인한 상태를 악용해 원치 않는 요청을 보내게 만드는 공격이다. 쿠키 기반 인증을 사용할 때 특히 고려해야 한다. SameSite 쿠키 정책은 CSRF 위험을 줄이는 데 도움이 되지만, 모든 상황을 자동으로 해결하지는 않는다. 중요한 변경 요청에는 CSRF token, origin 검증, 적절한 SameSite 정책이 함께 필요할 수 있다.

프론트엔드 개발자는 credential 요청을 만들 때 다음을 확인해야 한다.

  • 요청에 쿠키가 포함되는가?
  • cross-site 요청인가?
  • 서버가 어떤 CSRF 방어를 기대하는가?
  • CORS 설정과 credential 정책이 일관되는가?

AI가 fetch 코드에 credentials: "include"를 추가했다면, 그것이 왜 필요한지 확인해야 한다. 인증이 되게 하려고 넣은 한 줄이 CORS와 CSRF 문제를 함께 끌고 올 수 있다.

클라이언트 번들과 환경 변수

프론트엔드 빌드에서 환경 변수는 조심해야 한다. 클라이언트 번들에 들어간 값은 사용자가 볼 수 있다고 생각해야 한다. 이름이 SECRET이어도 브라우저에 포함되면 더 이상 secret이 아니다.

프론트엔드에 들어가도 되는 값은 public API base URL, 공개 가능한 feature flag, 분석 도구의 public key처럼 노출을 전제로 한 값이다. 서버 전용 secret, private API key, 관리자 token은 클라이언트 코드에 들어가면 안 된다.

AI가 .env 설정이나 SDK 초기화 코드를 만들 때는 이 경계를 확인해야 한다. 특히 “이 API key를 넣어주세요“라는 예제 코드는 브라우저용 public key인지 서버용 secret인지 구분해야 한다.

AI 생성 코드 리뷰 질문

네트워크, 보안, 스토리지 관점에서 AI가 만든 코드를 리뷰할 때는 다음 질문을 던진다.

  • 인증 정보나 민감한 값이 브라우저 저장소에 들어가지 않는가?
  • 클라이언트 번들에 secret이 포함되지 않는가?
  • HTTP 상태 코드별 사용자 경험이 구분되는가?
  • 캐시 키가 모든 검색 조건과 사용자 맥락을 포함하는가?
  • mutation 이후 캐시 무효화나 optimistic rollback이 있는가?
  • CORS 문제를 무작정 wildcard로 해결하려 하지 않는가?
  • credential 요청과 CSRF 방어가 일관되는가?
  • 사용자 입력이나 외부 HTML이 안전하게 렌더링되는가?
  • 에러 로그나 에러 메시지가 개인정보를 노출하지 않는가?

프론트엔드 개발자는 보안의 최종 책임자가 아닐 수 있다. 하지만 브라우저에서 어떤 값이 저장되고, 어떤 요청이 나가고, 어떤 데이터가 화면에 들어오는지 가장 가까이에서 보는 사람이다. AI 시대에는 이 감각이 더 중요해진다. 자동생성 코드가 편해 보일수록, 브라우저와 네트워크의 경계를 더 분명히 봐야 한다.

7. UI 상태와 데이터 흐름 설계

복잡한 프론트엔드 애플리케이션은 상태 관리에서 무너진다. 상태가 너무 멀리 있거나, 너무 가까이 있거나, 서로 다른 출처의 상태가 섞이면 작은 변경도 위험해진다. AI가 코드를 생성할 때도 상태의 경계를 정확히 설명하지 않으면 중복 상태와 임시 플래그가 빠르게 늘어난다.

상태 관리는 라이브러리 선택 문제가 아니다. Redux를 쓰면 해결되는 문제도 있고, TanStack Query를 쓰면 해결되는 문제도 있고, 단순한 useState 하나가 가장 좋은 답인 문제도 있다. 하지만 어떤 상태가 어떤 성격인지 구분하지 못하면 어떤 도구를 써도 복잡도는 줄지 않는다.

상태를 분류하는 이유

상태를 분류하는 이유는 저장 위치를 결정하기 위해서다. 모든 값을 전역 store에 넣으면 접근은 쉬워지지만 변경 이유를 추적하기 어려워진다. 모든 값을 지역 state로 두면 컴포넌트는 독립적으로 보이지만 화면 간 동기화가 깨진다. 서버에서 온 데이터를 클라이언트 state로 복사하면 캐시, 재검증, stale data 문제가 생긴다.

AI가 만든 UI 코드에서 흔한 문제는 “필요해 보이는 값을 일단 state로 만든다“는 것이다. 예를 들어 필터된 목록, 선택된 개수, 다음 페이지 가능 여부처럼 기존 데이터로 계산할 수 있는 값을 별도 state로 저장한다. 처음에는 편해 보이지만, 원본 상태가 바뀔 때 파생 상태도 함께 갱신해야 하므로 버그가 생긴다.

상태를 분류하면 이런 질문을 던질 수 있다.

  • 이 값은 사용자가 직접 바꾸는가?
  • 서버에서 온 값인가?
  • URL에 남아야 하는가?
  • 여러 화면에서 공유되어야 하는가?
  • 기존 값으로 계산할 수 있는가?
  • 시간이 지나면 자동으로 stale해지는가?

이 질문이 상태 설계의 출발점이다.

지역 상태

지역 상태는 특정 컴포넌트 안에서만 의미 있는 값이다. 예를 들어 dropdown이 열렸는지, 탭 내부의 임시 선택값, hover된 항목, 모달 안의 아직 제출하지 않은 입력값이 여기에 해당한다.

지역 상태의 장점은 단순함이다. 가까운 곳에 있으니 이해하기 쉽고, 컴포넌트가 사라지면 상태도 함께 사라진다. 문제는 지역 상태가 여러 컴포넌트에서 필요해질 때다. 이때 무작정 부모로 끌어올리거나 전역 store에 넣기 전에, 그 상태가 정말 공유되어야 하는지 확인해야 한다.

예를 들어 상품 카드의 hover 상태는 전역 상태가 될 이유가 없다. 반면 사이드바의 열림 여부가 여러 레이아웃 컴포넌트와 연결되어 있고 사용자 설정으로 저장되어야 한다면 더 넓은 범위의 상태가 될 수 있다.

AI에게 지역 상태가 충분한 작업을 맡길 때는 “이 상태는 이 컴포넌트 밖으로 노출하지 말아줘“라고 명시하는 것도 좋다. 작은 상태를 불필요하게 전역화하면 나중에 변경 비용이 커진다.

공유 UI 상태

공유 UI 상태는 서버 데이터는 아니지만 여러 컴포넌트가 함께 봐야 하는 화면 상태다. 예를 들어 현재 선택된 organization, 열려 있는 command palette, 전역 toast, layout density, theme, side panel 상태가 있다.

공유 UI 상태는 Context, store, route layout, framework의 전역 상태 도구로 관리할 수 있다. 중요한 것은 범위다. 전체 앱에서 필요한 상태인지, 특정 feature 안에서만 필요한 상태인지 구분해야 한다.

AI가 생성한 코드는 종종 편의를 위해 Context를 새로 만들거나 전역 store에 값을 추가한다. 하지만 전역 상태는 쉽게 늘어나고 어렵게 줄어든다. 공유 UI 상태를 만들기 전에 다음 질문을 한다.

  • 이 상태를 읽는 컴포넌트는 몇 개인가?
  • 이 상태를 변경하는 주체는 몇 개인가?
  • 이 상태가 앱을 벗어나 URL이나 저장소에 남아야 하는가?
  • feature 단위 store로 충분한가?
  • 전역화했을 때 테스트와 디버깅이 쉬워지는가?

공유 상태는 편의가 아니라 책임이다.

서버 상태

서버 상태는 서버에서 온 데이터와 그 데이터의 캐시, stale 여부, 재검증, mutation 결과를 포함한다. 상품 목록, 사용자 프로필, 알림 목록, 권한 정보, 결제 상태 같은 값은 서버 상태다.

서버 상태를 일반 클라이언트 상태처럼 다루면 문제가 생긴다. 서버 데이터는 시간이 지나면 낡는다. 여러 화면에서 같은 데이터를 사용할 수 있다. mutation 후에는 관련 데이터를 다시 가져오거나 캐시를 업데이트해야 한다. 네트워크 실패, 재시도, optimistic update도 필요할 수 있다.

그래서 TanStack Query, SWR 같은 서버 상태 도구가 유용하다. 이 도구들은 단순히 fetch를 감싸는 라이브러리가 아니다. 캐시 키, stale time, refetch, mutation, optimistic update, error state 같은 서버 상태의 문제를 모델링한다.

AI에게 서버 데이터를 다루는 코드를 요청할 때는 다음을 명시해야 한다.

  • query key에 어떤 값이 포함되어야 하는가?
  • 언제 stale하다고 볼 것인가?
  • mutation 후 어떤 query를 무효화할 것인가?
  • optimistic update를 할 것인가?
  • 실패 시 rollback이 필요한가?

서버 상태는 “가져온 뒤 화면에 뿌리는 값“이 아니라 동기화 대상이다.

URL 상태

URL 상태는 주소에 남아야 하는 상태다. 검색어, 필터, 정렬, 페이지 번호, 선택된 탭, 공유 가능한 view mode가 여기에 해당한다. URL 상태의 기준은 간단하다. 사용자가 새로고침하거나 링크를 공유했을 때 유지되어야 하는가?

검색 화면에서 필터를 컴포넌트 state로만 관리하면 새로고침 때 조건이 사라진다. 뒤로 가기도 기대대로 동작하지 않는다. 반대로 모든 UI 상태를 URL에 넣으면 주소가 불필요하게 복잡해진다.

URL 상태는 다음 원칙으로 다룬다.

  • 공유 가능한 검색 조건은 URL에 둔다.
  • 기본값은 query parameter에서 생략한다.
  • 빈 문자열과 undefined를 구분한다.
  • 필터가 바뀌면 페이지 번호를 초기화한다.
  • URL 파싱과 직렬화는 한 곳에 모은다.

AI가 필터 화면을 만들 때 가장 자주 놓치는 것이 URL 상태다. “필터는 새로고침과 뒤로 가기에서도 유지되어야 한다“는 요구를 명시하지 않으면 단순한 지역 state로 구현할 가능성이 높다.

파생 상태

파생 상태는 기존 값으로 계산할 수 있는 값이다. 예를 들어 장바구니 총액, 선택된 항목 개수, 필터 적용 여부, 다음 페이지 가능 여부, 에러 배너 표시 여부가 있다. 파생 상태는 가능하면 저장하지 않고 계산한다.

나쁜 예는 다음과 같다.

const [items, setItems] = useState<Product[]>([]);
const [filteredItems, setFilteredItems] = useState<Product[]>([]);
const [count, setCount] = useState(0);

filteredItems와 count는 items와 필터 조건으로 계산할 수 있다. 별도 state로 저장하면 items가 바뀔 때마다 동기화해야 한다. 동기화가 빠지는 순간 UI는 거짓말을 한다.

더 나은 방식은 다음과 같다.

const filteredItems = useMemo(
  () => items.filter((item) => matchesFilter(item, filter)),
  [items, filter],
);

const count = filteredItems.length;

물론 모든 계산에 useMemo가 필요한 것은 아니다. 중요한 것은 “이 값이 원본인가, 계산 결과인가“를 구분하는 것이다. AI가 state를 많이 만들수록 파생 상태가 섞였는지 확인해야 한다.

상태 위치를 결정하는 절차

새 기능을 만들 때 다음 절차를 사용할 수 있다.

  1. 화면에서 보이는 값을 모두 적는다.
  2. 각 값이 사용자 입력인지, 서버 데이터인지, URL 상태인지, 파생 상태인지 표시한다.
  3. 서버 데이터는 서버 상태 도구나 data loader에 연결한다.
  4. 공유 가능한 검색/탭/페이지 상태는 URL로 보낸다.
  5. 컴포넌트 안에서만 필요한 값은 지역 상태로 둔다.
  6. 기존 값으로 계산 가능한 것은 저장하지 않는다.
  7. 여러 컴포넌트가 읽고 쓰는 UI 상태만 제한적으로 공유한다.

이 절차는 AI에게도 그대로 줄 수 있다. “상태를 먼저 분류하고 구현 계획을 제안해줘“라고 요청하면, 코드를 만들기 전에 설계 대화를 할 수 있다.

데이터 흐름은 한 방향으로 읽혀야 한다

복잡한 UI에서 데이터 흐름은 가능한 한 한 방향으로 읽혀야 한다. 사용자의 행동이 상태를 바꾸고, 상태가 화면을 바꾸고, 필요한 경우 서버와 동기화한다. 이 흐름이 뒤엉키면 디버깅이 어려워진다.

예를 들어 자식 컴포넌트가 부모의 상태를 암묵적으로 바꾸고, 동시에 URL도 바꾸고, 서버 요청도 직접 실행한다면 변경 이유를 추적하기 어렵다. 반대로 이벤트는 위로 올리고, 상태 변경은 정해진 위치에서 일어나고, 서버 동기화는 query/mutation 계층에서 처리되면 코드가 읽힌다.

AI가 만든 컴포넌트가 너무 많은 책임을 갖고 있다면 다음처럼 나눌 수 있다.

  • UI 컴포넌트: props를 받아 표시하고 사용자 이벤트를 알린다.
  • 상태 훅: URL 상태와 지역 상태를 관리한다.
  • 데이터 훅: 서버 query와 mutation을 관리한다.
  • adapter: API 응답을 UI 모델로 변환한다.

이 분리는 재사용보다 이해 가능성을 위해 한다.

AI 생성 코드 리뷰 질문

상태와 데이터 흐름 관점에서 AI가 만든 코드를 리뷰할 때는 다음 질문을 던진다.

  • 서버 상태를 클라이언트 state로 불필요하게 복사하지 않았는가?
  • URL에 남아야 할 검색 조건을 지역 state로만 관리하지 않는가?
  • 기존 값으로 계산 가능한 파생 상태를 별도 state로 저장하지 않는가?
  • 전역 상태가 정말 전역이어야 하는가?
  • query key가 상태의 모든 입력을 포함하는가?
  • mutation 후 캐시 무효화나 업데이트가 명확한가?
  • 상태 변경이 한 방향으로 읽히는가?
  • 컴포넌트 하나가 UI, 데이터 요청, URL 동기화, 비즈니스 규칙을 모두 맡고 있지 않은가?

상태 설계는 프론트엔드의 중심이다. AI가 코드를 빠르게 만들수록 상태의 위치를 먼저 결정하는 습관이 더 중요해진다.

8. 복잡도를 낮추는 디자인 패턴

디자인 패턴은 이름을 외우기 위한 것이 아니라 복잡도를 다루기 위한 언어다. 프론트엔드에서 패턴은 컴포넌트 합성, 컨테이너와 프레젠테이션 분리, 커스텀 훅, compound components, adapter, facade, state machine, command, strategy 같은 형태로 나타난다.

AI 시대에는 패턴을 더 조심해서 써야 한다. AI는 “그럴듯한 추상화“를 빠르게 만들 수 있다. 하지만 그 추상화가 실제 문제를 줄이지 않으면 코드만 더 복잡해진다. 좋은 패턴은 변경을 견디게 하지만, 나쁜 패턴은 단순한 코드를 설명하기 어려운 코드로 바꾼다.

패턴은 문제 뒤에 온다

패턴을 먼저 정하고 코드를 맞추면 과설계가 된다. 반대로 문제를 충분히 관찰한 뒤 패턴을 도입하면 복잡도를 줄일 수 있다.

예를 들어 버튼 하나를 만들기 위해 compound component 패턴을 도입할 필요는 없다. 하지만 탭, 메뉴, select, accordion처럼 여러 하위 요소가 공유 상태와 접근성 관계를 가져야 하는 컴포넌트라면 compound component가 유용할 수 있다.

마찬가지로 단순한 API 호출에는 adapter가 필요 없을 수 있다. 하지만 서버 응답 구조가 UI에 그대로 맞지 않거나, 여러 API 버전을 다뤄야 하거나, 외부 서비스 데이터를 내부 도메인 모델로 바꿔야 한다면 adapter가 필요하다.

좋은 질문은 “이 패턴을 쓰면 멋져 보이는가“가 아니라 “이 패턴이 변경 비용을 줄이는가“다.

컴포넌트 합성

프론트엔드의 가장 기본적인 패턴은 컴포넌트 합성이다. 큰 화면을 작은 컴포넌트로 나누고, 각 컴포넌트가 명확한 책임을 갖게 하는 것이다. 그러나 컴포넌트를 많이 나눈다고 항상 좋은 것은 아니다. 너무 작은 컴포넌트가 흩어져 있으면 흐름을 읽기 어려워진다.

컴포넌트를 나눌 때는 다음 기준을 사용한다.

  • UI의 의미 단위가 분명한가?
  • 독립적으로 테스트하거나 story를 만들 가치가 있는가?
  • 상태나 이벤트 책임이 분리되는가?
  • 반복해서 사용되는가?
  • 이름을 붙였을 때 도메인 의미가 생기는가?

AI가 만든 코드는 종종 한 파일에 모든 것을 몰아넣는다. 반대로 “컴포넌트를 나눠줘“라고 요청하면 의미 없는 작은 컴포넌트를 많이 만들기도 한다. 사람은 분리의 기준을 줘야 한다. “폼 필드, 결과 목록, 빈 상태, 페이지네이션을 의미 단위로 나누되, 단순 wrapper는 만들지 마“처럼 말하는 편이 좋다.

컨테이너와 프레젠테이션 분리

컨테이너와 프레젠테이션 분리는 오래된 패턴이지만 여전히 유용하다. 핵심은 데이터를 가져오고 상태를 관리하는 부분과, props를 받아 화면을 그리는 부분을 나누는 것이다.

예를 들어 상품 목록 화면에서 컨테이너는 URL 상태를 읽고, query를 호출하고, mutation을 연결한다. 프레젠테이션 컴포넌트는 items, isLoading, error, onFavoriteToggle, onPageChange 같은 props를 받아 화면을 그린다.

이 분리의 장점은 테스트와 이해 가능성이다. 프레젠테이션 컴포넌트는 서버 상태 도구나 라우터 없이도 테스트할 수 있다. 컨테이너는 데이터 흐름에 집중한다.

하지만 모든 컴포넌트를 억지로 container/presentational로 나눌 필요는 없다. 작은 컴포넌트에서는 오히려 파일만 늘어난다. 이 패턴은 데이터 흐름과 UI 표현이 섞여서 읽기 어려워질 때 도입한다.

커스텀 훅과 composable

React의 커스텀 훅, Vue의 composable은 로직 재사용과 책임 분리에 유용하다. 그러나 훅은 단순히 코드를 함수로 빼는 도구가 아니다. 훅은 상태와 side effect를 가진 작은 모듈이다.

좋은 훅은 이름만 봐도 역할을 알 수 있다.

  • useProductSearchParams: URL 검색 조건을 읽고 쓴다.
  • useProductListQuery: 상품 목록 서버 상태를 관리한다.
  • useFavoriteMutation: 즐겨찾기 변경과 캐시 업데이트를 처리한다.

나쁜 훅은 너무 많은 일을 한다.

useProductPageEverything()

이런 훅은 URL, API, toast, analytics, modal 상태를 모두 품을 가능성이 있다. 처음에는 컨테이너를 깨끗하게 만드는 것처럼 보이지만, 복잡도를 다른 파일로 숨긴 것뿐이다.

AI에게 훅을 만들게 할 때는 훅의 책임을 한 문장으로 제한한다. “URL query parameter 파싱과 업데이트만 담당하는 훅을 만들어줘“처럼 요청하면 좋다.

Adapter: 외부 세계와 내부 모델 사이

외부 API 응답은 내부 UI 모델과 다를 수 있다. 서버는 snake_case를 쓰고, 가격을 문자열로 내려주고, optional 필드를 포함하고, UI에는 필요 없는 메타데이터를 함께 보낼 수 있다. 이 값을 컴포넌트 곳곳에서 직접 해석하면 외부 계약이 UI 전체로 퍼진다.

Adapter 패턴은 외부 데이터를 내부 모델로 바꾸는 경계를 만든다.

type ProductApiResponse = {
  product_id: string;
  display_name: string;
  price_cents: number;
  image_url?: string;
};

type ProductCardModel = {
  id: string;
  name: string;
  priceText: string;
  imageUrl: string;
};

function toProductCardModel(input: ProductApiResponse): ProductCardModel {
  return {
    id: input.product_id,
    name: input.display_name,
    priceText: formatPrice(input.price_cents),
    imageUrl: input.image_url ?? "/images/product-placeholder.png",
  };
}

이렇게 하면 UI 컴포넌트는 API의 세부 구조를 몰라도 된다. API가 바뀌면 adapter를 중심으로 수정한다. AI가 만든 코드가 API 응답을 컴포넌트에서 직접 해체하고 있다면 adapter를 고려해야 한다.

Facade: 복잡한 도구를 단순한 인터페이스로

Facade는 복잡한 내부 구현을 단순한 인터페이스 뒤에 숨기는 패턴이다. 예를 들어 analytics SDK, feature flag SDK, 결제 SDK, 지도 SDK는 직접 컴포넌트에서 호출하기보다 내부 wrapper를 두는 편이 좋다.

trackProductClicked({ productId, source: "search_result" });

컴포넌트는 analytics SDK의 구체적인 API를 몰라도 된다. 나중에 SDK가 바뀌거나 이벤트 필드가 바뀌어도 wrapper를 수정하면 된다.

AI가 기능을 만들 때 외부 SDK 호출을 컴포넌트 곳곳에 직접 넣는 경우가 있다. 이런 코드는 빠르게 만들어지지만 교체가 어렵다. 외부 세계와 만나는 지점에는 facade나 adapter가 필요할 수 있다.

State machine: 흐름이 복잡할 때

상태가 많아지고 전이가 복잡해지면 boolean 몇 개로는 부족하다. 예를 들어 결제 흐름은 idle, validating, submitting, requires_auth, success, failed, cancelled 같은 상태를 가질 수 있다. 이 상태들은 서로 배타적이며, 어떤 상태에서 어떤 이벤트가 가능한지도 중요하다.

이럴 때 state machine이 도움이 된다. 반드시 XState 같은 라이브러리를 써야 한다는 뜻은 아니다. discriminated union과 reducer만으로도 상태 전이를 명시할 수 있다.

type CheckoutState =
  | { status: "idle" }
  | { status: "submitting" }
  | { status: "requires_auth"; redirectUrl: string }
  | { status: "success"; orderId: string }
  | { status: "failed"; message: string };

이 타입은 isLoading, isError, isSuccess, needsAuth 같은 boolean이 동시에 모순된 값을 갖는 것을 막는다. AI가 복잡한 폼이나 결제 흐름을 boolean 여러 개로 만들었다면 상태 기계를 고려해야 한다.

Command와 Strategy

Command 패턴은 사용자의 동작을 명령 객체나 함수로 표현할 때 유용하다. 예를 들어 에디터에서 undo/redo가 필요하거나, 여러 액션을 큐에 넣고 실행해야 하거나, 같은 동작을 나중에 재시도해야 할 때 도움이 된다.

Strategy 패턴은 알고리즘이나 정책이 바뀌는 지점을 분리한다. 예를 들어 정렬 방식, 가격 계산 방식, 권한 체크 방식, 검색 ranking 방식이 상황에 따라 달라진다면 strategy로 나눌 수 있다.

프론트엔드에서 이런 패턴은 거창하게 보일 수 있다. 그러나 복잡한 SaaS, 대시보드, 에디터, 커머스에서는 자주 등장한다. 중요한 것은 패턴 이름이 아니라 변경 지점이다. “여기가 바뀔 수 있는 정책인가“를 물으면 strategy가 보이고, “이 동작을 기록하거나 되돌려야 하는가“를 물으면 command가 보인다.

패턴을 도입하지 말아야 할 때

패턴은 복잡도를 줄이기 위해 존재한다. 따라서 다음 상황에서는 도입을 미룬다.

  • 요구사항이 아직 불안정하고 무엇이 반복될지 모른다.
  • 컴포넌트가 작고 읽기 쉽다.
  • 추상화 이름이 도메인 의미를 갖지 못한다.
  • 분리 후 파일을 오가야만 이해할 수 있다.
  • 테스트나 변경 비용이 실제로 줄지 않는다.

AI는 “더 깔끔하게 리팩터링해줘“라는 요청에 과도한 추상화를 만들 수 있다. 좋은 리팩터링 요청은 “이 컴포넌트에서 API 응답 변환만 adapter로 분리해줘“처럼 문제와 경계를 지정한다.

AI 생성 코드 리뷰 질문

디자인 패턴 관점에서 AI가 만든 코드를 리뷰할 때는 다음 질문을 던진다.

  • 컴포넌트 분리가 의미 단위로 되어 있는가?
  • 한 컴포넌트가 UI, 데이터 요청, 상태, analytics, 변환 로직을 모두 맡고 있지 않은가?
  • 커스텀 훅의 책임이 한 문장으로 설명되는가?
  • 외부 API 응답이 UI 컴포넌트 곳곳에 직접 퍼져 있지 않은가?
  • 외부 SDK 호출이 facade 없이 흩어져 있지 않은가?
  • boolean 조합으로 모순 가능한 상태를 만들지 않았는가?
  • 패턴이 실제 변경 비용을 줄이는가, 아니면 이름만 늘렸는가?
  • 새 추상화가 테스트를 더 쉽게 만드는가?

패턴은 개발자의 어휘다. 어휘가 많으면 문제를 더 정확히 설명할 수 있다. 하지만 좋은 문장은 어려운 단어를 많이 쓰는 문장이 아니라, 필요한 단어를 정확히 쓰는 문장이다. 코드도 같다.

9. 타입, 계약, 경계

TypeScript는 AI 시대의 프론트엔드에서 더 중요해졌다. 타입은 사람이 읽는 문서이면서 AI가 코드를 생성할 때 참고하는 구조적 힌트다. 명확한 타입은 잘못된 제안을 줄이고, 리팩터링을 안전하게 만들며, 코드베이스의 의도를 보존한다.

하지만 타입을 많이 쓴다고 자동으로 좋은 코드가 되는 것은 아니다. any로 가득한 코드도 TypeScript 파일일 수 있고, 모든 필드를 optional로 둔 타입은 거의 계약이 아니다. 타입은 값의 모양을 설명하는 데서 출발하지만, 좋은 타입은 도메인의 의미와 경계를 표현한다.

타입은 계약이다

계약은 양쪽의 기대를 명확히 한다. API 응답 타입은 서버와 클라이언트 사이의 계약이다. 컴포넌트 props는 부모와 자식 사이의 계약이다. 이벤트 payload는 사용자 행동을 처리하는 코드 사이의 계약이다. route params와 query params도 계약이다.

계약이 명확하면 AI가 코드를 생성할 때도 더 좋은 결과를 낸다. 예를 들어 Product 타입이 명확하고, ProductSearchParams 타입이 있고, ProductSearchResult 타입이 있으면 AI는 그 구조 안에서 코드를 만들 수 있다. 반대로 데이터가 object나 any로 흘러다니면 AI는 추측한다.

개발자가 해야 할 일은 AI에게 추측할 여지를 줄이는 것이다. 타입은 그 여지를 줄이는 가장 좋은 방법 중 하나다.

API 타입과 UI 타입은 다를 수 있다

외부 API 응답 타입을 UI 타입으로 그대로 쓰면 편하다. 하지만 항상 좋은 선택은 아니다. 서버의 데이터 구조는 서버의 이유로 만들어진다. UI는 사용자의 행동과 화면 표현을 중심으로 값을 필요로 한다.

예를 들어 서버 응답은 다음과 같을 수 있다.

type ProductResponse = {
  id: string;
  name: string;
  price_cents: number;
  image_url: string | null;
  flags?: string[];
};

하지만 상품 카드 UI에는 다음 모델이 더 적합할 수 있다.

type ProductCardViewModel = {
  id: string;
  title: string;
  priceText: string;
  imageUrl: string;
  isSoldOut: boolean;
};

이 둘을 구분하면 UI 컴포넌트가 서버의 세부 표현에 묶이지 않는다. API가 price_cents를 price 객체로 바꾸더라도 adapter만 수정하면 된다. AI가 서버 응답 타입을 컴포넌트 props로 그대로 퍼뜨리고 있다면 경계가 사라지고 있는 것이다.

optional의 의미

TypeScript에서 ?는 편리하지만 남용하기 쉽다. optional은 “없을 수 있다“는 뜻이다. 하지만 왜 없을 수 있는지는 여러 가지다.

  • 서버가 필드를 보내지 않을 수 있다.
  • 아직 로딩 중이라 값이 없다.
  • 권한이 없어 볼 수 없다.
  • 사용자가 입력하지 않았다.
  • feature flag가 꺼져 있다.

이 모든 상황을 name?: string 하나로 표현하면 의미가 사라진다. 예를 들어 사용자 프로필이 아직 로딩 중인 상황과, 서버에 프로필 이미지가 없는 상황은 다르다.

더 나은 타입은 상태를 분리한다.

type ProfileState =
  | { status: "loading" }
  | { status: "ready"; profile: Profile }
  | { status: "error"; message: string };

이렇게 하면 값이 없는 이유를 코드가 설명한다. AI가 optional을 많이 추가했다면, 그 optional이 실제 도메인 의미인지 아니면 로딩/에러 상태를 흐리고 있는지 확인해야 한다.

discriminated union

discriminated union은 상태를 명확히 표현하는 강력한 도구다. 특히 비동기 상태, 폼 상태, 권한 상태, 결제 상태처럼 서로 배타적인 경우에 유용하다.

type SaveState =
  | { status: "idle" }
  | { status: "saving" }
  | { status: "saved"; savedAt: string }
  | { status: "failed"; error: Error };

이 타입은 status에 따라 사용할 수 있는 필드가 달라진다. status가 failed일 때만 error가 있고, saved일 때만 savedAt이 있다. boolean 여러 개보다 안전하고 읽기 쉽다.

AI가 isLoading, isSaved, isError, error, savedAt 같은 상태를 모두 따로 만들었다면 union type으로 바꿀 수 있는지 검토한다. 타입은 모순을 표현할 수 없게 만들 때 가장 강력하다.

exhaustive check

union type을 사용할 때는 모든 경우를 처리했는지 확인하는 것이 중요하다. TypeScript에서는 never를 이용해 exhaustive check를 만들 수 있다.

function assertNever(value: never): never {
  throw new Error(`Unexpected value: ${value}`);
}

function getSaveMessage(state: SaveState): string {
  switch (state.status) {
    case "idle":
      return "변경 사항이 없습니다.";
    case "saving":
      return "저장 중입니다.";
    case "saved":
      return `${state.savedAt}에 저장되었습니다.`;
    case "failed":
      return state.error.message;
    default:
      return assertNever(state);
  }
}

새 상태가 추가되면 TypeScript가 처리 누락을 알려줄 수 있다. AI가 union에 새 variant를 추가했는데 UI 처리를 빠뜨리는 문제를 줄일 수 있다.

런타임 검증

TypeScript 타입은 빌드 타임에만 존재한다. 브라우저에서 실제 API 응답이 타입과 다르게 와도 TypeScript가 자동으로 막아주지는 않는다. 외부 입력, API 응답, localStorage 데이터, URL query parameter는 런타임 검증이 필요할 수 있다.

Zod 같은 schema validation 도구는 이런 경계에서 유용하다.

import { z } from "zod";

const ProductSchema = z.object({
  id: z.string(),
  name: z.string(),
  priceCents: z.number().int().nonnegative(),
});

type Product = z.infer<typeof ProductSchema>;

모든 데이터에 schema를 붙일 필요는 없다. 하지만 외부에서 들어오는 데이터, 저장소에서 복원하는 데이터, 보안이나 결제와 연결된 데이터는 런타임 검증을 고려해야 한다.

AI가 “타입을 정의했으니 안전하다“고 말하더라도, 그 타입이 실제 런타임 값을 검증하는 것은 아니다. 이 경계를 혼동하면 위험하다.

generated types와 API 계약

OpenAPI, GraphQL, tRPC 같은 도구는 서버 계약에서 타입을 생성할 수 있다. generated types는 중복을 줄이고 서버와 클라이언트의 계약을 맞추는 데 도움이 된다. AI 시대에는 특히 유용하다. 명확한 generated type이 있으면 AI가 API 구조를 추측할 필요가 줄어든다.

하지만 generated type이 UI 설계를 대신하지는 않는다. 서버 타입은 서버 계약이고, UI 타입은 화면 계약이다. 둘 사이에는 adapter나 mapper가 필요할 수 있다. generated type을 그대로 모든 컴포넌트 props에 퍼뜨리면 API 변경이 UI 전체로 번진다.

좋은 구조는 다음과 같다.

  • API layer: generated type을 사용한다.
  • adapter layer: API 응답을 도메인/UI 모델로 바꾼다.
  • UI layer: 안정적인 view model을 받는다.

이 경계는 AI가 코드를 수정할 때도 안전장치가 된다.

컴포넌트 props 계약

컴포넌트 props도 계약이다. props가 너무 많으면 컴포넌트의 책임이 넓다는 신호일 수 있다. boolean props가 많이 늘어나면 가능한 조합이 폭발한다.

type ButtonProps = {
  isPrimary?: boolean;
  isDanger?: boolean;
  isGhost?: boolean;
};

이 타입은 isPrimary와 isDanger가 동시에 true일 수 있다. 더 나은 방식은 variant를 하나의 값으로 표현하는 것이다.

type ButtonVariant = "primary" | "danger" | "ghost";

type ButtonProps = {
  variant: ButtonVariant;
};

가능한 상태를 타입으로 제한하면 AI가 잘못된 조합을 만들 가능성도 줄어든다. 프론트엔드 타입 설계의 목표는 “가능한 값“이 아니라 “허용되는 의미“를 표현하는 것이다.

이벤트 payload와 도메인 언어

이벤트 payload도 타입으로 명확히 표현해야 한다. 특히 analytics, form submit, command action, domain event는 Record<string, unknown>처럼 두면 금방 흐려진다.

type ProductClickedEvent = {
  type: "product_clicked";
  productId: string;
  source: "search_result" | "recommendation" | "favorite_list";
};

이런 타입은 도메인 언어를 보존한다. AI에게 analytics 코드를 만들게 할 때도 이벤트 이름과 payload 타입이 명확하면 임의의 필드를 만들 가능성이 줄어든다.

AI 생성 코드 리뷰 질문

타입과 계약 관점에서 AI가 만든 코드를 리뷰할 때는 다음 질문을 던진다.

  • any, unknown, 지나치게 넓은 object 타입이 불필요하게 쓰이지 않았는가?
  • optional 필드가 값이 없는 이유를 흐리고 있지 않은가?
  • API 응답 타입과 UI view model이 무분별하게 섞이지 않았는가?
  • boolean 여러 개로 모순 가능한 상태를 만들지 않았는가?
  • union type과 exhaustive check로 상태를 더 명확히 만들 수 있는가?
  • 외부 입력과 API 응답에 런타임 검증이 필요한가?
  • generated type이 UI 전체로 직접 퍼지고 있지 않은가?
  • 컴포넌트 props가 가능한 조합을 타입으로 제한하는가?
  • 이벤트 payload가 도메인 언어를 보존하는가?

타입은 AI 시대의 방어선이다. 좋은 타입은 AI가 더 나은 코드를 만들게 하고, 사람이 더 빠르게 리뷰하게 하며, 잘못된 변경을 더 일찍 드러낸다. 타입은 코드의 모양이 아니라 시스템의 약속을 표현해야 한다.

10. 코드베이스를 읽는 방법

AI 시대에 더 중요해지는 능력 중 하나는 코드를 읽는 능력이다. 코드를 직접 덜 쓰게 될수록, 생성된 코드와 기존 코드의 관계를 읽어내는 능력이 개발자의 실력을 가른다. AI가 만든 코드를 제대로 리뷰하려면 먼저 기존 코드베이스의 언어를 읽을 수 있어야 한다.

코드 읽기는 수동적인 활동이 아니다. 파일을 열고 문장을 따라가는 것만으로는 충분하지 않다. 좋은 독자는 질문을 만든다. 이 값은 어디서 오는가. 이 컴포넌트는 왜 이 props를 받는가. 이 조건은 비즈니스 규칙인가 UI 편의인가. 이 함수는 재사용을 위해 나뉜 것인가 테스트를 위해 나뉜 것인가. 이런 질문이 코드의 구조를 드러낸다.

코드를 읽기 전에 목적을 정한다

코드베이스 전체를 한 번에 이해하려는 시도는 실패하기 쉽다. 큰 애플리케이션은 한 사람이 한 번에 모두 담기 어렵다. 따라서 코드를 읽기 전에 목적을 정해야 한다.

  • 버그 원인을 찾기 위해 읽는가?
  • 새 기능을 추가하기 위해 읽는가?
  • 리팩터링 범위를 잡기 위해 읽는가?
  • AI가 만든 변경을 리뷰하기 위해 읽는가?
  • 온보딩을 위해 전체 구조를 파악하는가?

목적에 따라 읽는 순서가 달라진다. 버그를 찾는다면 재현 경로와 관련 파일을 좁혀야 한다. 새 기능을 추가한다면 비슷한 기능의 구조를 찾아야 한다. AI 생성 코드를 리뷰한다면 변경된 파일과 기존 패턴의 차이를 봐야 한다.

AI에게 코드 읽기를 맡길 때도 목적을 명시해야 한다. “이 프로젝트 설명해줘“보다 “상품 검색 필터 기능을 수정하려고 한다. 관련 진입점, 상태 위치, API 호출, 테스트 파일을 찾아줘“가 훨씬 낫다.

첫 번째 지도: 진입점과 라우트

코드베이스를 읽을 때 첫 번째 지도는 앱의 진입점과 라우트다. 프론트엔드 애플리케이션은 사용자가 URL이나 화면 흐름을 통해 만난다. 따라서 라우트 구조를 알면 코드의 큰 뼈대를 이해할 수 있다.

확인할 것은 다음과 같다.

  • 앱의 entry file은 어디인가?
  • router 설정은 어디에 있는가?
  • layout과 page의 계층은 어떻게 나뉘는가?
  • 인증이 필요한 route는 어디에서 보호되는가?
  • error boundary나 loading UI는 어디에 있는가?
  • feature folder와 route folder는 어떻게 연결되는가?

Next.js App Router라면 app/ 아래 layout, page, route handler 구조를 본다. React Router 기반 앱이라면 route config와 layout route를 본다. Vue/Nuxt, SvelteKit, Angular도 각자 라우팅 관습이 있다. 중요한 것은 “사용자의 경로가 어떤 파일로 이어지는가“를 파악하는 것이다.

AI가 만든 변경이 route 경계를 넘는다면 특히 조심해야 한다. route는 화면의 구조이면서 데이터 로딩, 인증, 캐시, 메타데이터, 에러 처리와 연결된다.

두 번째 지도: 데이터가 들어오는 곳

프론트엔드 코드는 데이터를 표시한다. 따라서 데이터가 어디에서 들어오고 어떻게 변환되는지 읽어야 한다.

찾아야 할 지점은 다음과 같다.

  • API client 또는 fetch wrapper
  • query/mutation 훅
  • data loader 또는 server action
  • mock 또는 fixture
  • adapter나 mapper
  • schema validation
  • error handling layer

데이터 흐름을 읽을 때는 타입을 함께 본다. API 응답 타입과 UI props 타입이 같은지, 중간에 변환이 있는지, nullable 값이 어디에서 처리되는지 확인한다.

AI가 만든 코드는 기존 API client를 우회하고 직접 fetch를 추가할 수 있다. 또는 기존 error handling을 거치지 않고 컴포넌트 안에서 임시로 처리할 수 있다. 이런 변경은 작동하더라도 시스템의 일관성을 깨뜨린다.

세 번째 지도: 상태의 위치

상태의 위치는 코드베이스의 성격을 보여준다. 어떤 앱은 서버 상태 도구를 적극적으로 사용한다. 어떤 앱은 route loader 중심이다. 어떤 앱은 전역 store가 강하고, 어떤 앱은 feature별 상태를 선호한다.

상태를 읽을 때는 다음을 확인한다.

  • 서버 상태는 어디에서 관리되는가?
  • URL 상태는 어떻게 파싱하고 갱신하는가?
  • 전역 store는 어떤 값만 담는가?
  • form 상태는 어떤 라이브러리나 패턴을 쓰는가?
  • 파생 상태를 state로 저장하는 관습이 있는가?
  • optimistic update는 어디에서 처리하는가?

상태 위치를 모르면 변경 범위를 제대로 잡을 수 없다. 예를 들어 필터 UI를 고치면서 컴포넌트 state만 보면 URL 동기화를 놓칠 수 있다. 즐겨찾기 버튼을 고치면서 mutation만 보면 캐시 무효화를 놓칠 수 있다.

네 번째 지도: 도메인 언어와 타입

코드베이스에는 팀의 도메인 언어가 숨어 있다. user, member, account, organization, workspace, tenant가 모두 비슷해 보여도 실제 의미는 다를 수 있다. 이 차이를 모르면 AI가 만든 타입 이름이 그럴듯해도 틀릴 수 있다.

도메인 언어를 읽는 방법은 다음과 같다.

  • 주요 타입과 interface 이름을 모은다.
  • 같은 개념을 가리키는 다른 이름이 있는지 확인한다.
  • API 타입과 UI 타입의 경계를 본다.
  • enum이나 union type이 표현하는 상태를 읽는다.
  • route params, query params, event payload 이름을 확인한다.

AI에게 코드를 만들게 하기 전에 도메인 용어를 알려주는 것이 좋다. “이 프로젝트에서 workspace는 사용자가 속한 작업 공간이고, organization은 billing 단위다“처럼 맥락을 주면 잘못된 이름과 구조를 줄일 수 있다.

다섯 번째 지도: 테스트와 Storybook

테스트와 Storybook은 코드의 사용 설명서다. 테스트는 무엇이 깨지면 안 되는지 보여준다. Storybook은 컴포넌트가 어떤 상태로 쓰이는지 보여준다. 둘 다 없다면 코드를 읽기가 더 어렵다.

테스트를 읽을 때는 다음을 본다.

  • 테스트가 사용자 행동 중심으로 작성되었는가?
  • 성공 케이스만 있는가, 실패 케이스도 있는가?
  • mock 데이터가 실제 API와 비슷한가?
  • 접근성 query를 사용하는가?
  • E2E 테스트가 핵심 흐름을 덮는가?

AI가 만든 변경이 테스트를 통과하더라도 충분하지 않을 수 있다. 테스트가 원래 약하면 약한 보증만 제공한다. 반대로 좋은 테스트가 있다면 AI 생성 코드 리뷰가 훨씬 쉬워진다.

여섯 번째 지도: 커밋과 이슈

코드는 현재 상태만 보여준다. 왜 그렇게 되었는지는 커밋과 이슈에 남아 있을 때가 많다. 어떤 이상한 조건문은 과거 장애의 흔적일 수 있다. 어떤 중복처럼 보이는 코드는 서로 다른 사용자 권한을 구분하기 위해 남겨졌을 수 있다.

리팩터링이나 AI 생성 코드 적용 전에 커밋 히스토리를 보는 습관은 유용하다.

git log -- path/to/file
git blame path/to/file

이 명령은 책임을 묻기 위한 도구가 아니다. 의도를 찾기 위한 도구다. AI는 히스토리를 자동으로 추론하지 못한다. 사람이 과거의 이유를 찾아야 할 때가 있다.

변경 가능한 지도 만들기

코드 읽기의 목표는 요약을 얻는 것이 아니라 변경 가능한 지도를 만드는 것이다. 지도는 다음처럼 간단해도 된다.

기능: 상품 검색
진입점: /products route
데이터: useProductListQuery -> productApi.search
URL 상태: useProductSearchParams
지역 상태: FilterPanel open state
주요 타입: ProductSearchParams, ProductCardViewModel
테스트: ProductListPage.test.tsx, products.spec.ts
주의점: 필터 변경 시 page=1 초기화, 빈 결과와 API 실패 구분

이 지도는 AI에게도 줄 수 있다. “이 지도에 맞춰 변경해줘“라고 요청하면 기존 구조를 깨뜨릴 가능성이 줄어든다.

AI와 함께 코드 읽기

AI는 코드 읽기를 도울 수 있다. 관련 파일을 찾고, 데이터 흐름을 요약하고, 변경 위험을 나열하게 할 수 있다. 그러나 AI의 설명을 그대로 믿으면 안 된다. 설명은 파일, 타입, 테스트, 실행 결과로 검증해야 한다.

좋은 요청은 다음과 같다.

아직 코드는 수정하지 말고, 이 기능의 데이터 흐름을 찾아줘.
진입점, 상태 위치, API 호출, 주요 타입, 테스트 파일을 표로 정리해줘.
확실하지 않은 부분은 추측이라고 표시해줘.

마지막 문장이 중요하다. AI가 확실하지 않은 것을 확실한 것처럼 말하지 않게 해야 한다. 개발자도 AI의 설명에서 추측과 증거를 구분해야 한다.

11. 자동생성 코드를 리뷰하는 방법

AI가 만든 코드는 초안이다. 초안은 유용하지만 완성본은 아니다. 자동생성 코드를 리뷰할 때는 “문법이 맞는가“보다 “이 변경이 시스템에 들어와도 되는가“를 물어야 한다. 사용자는 코드가 누가 썼는지 모른다. 장애가 나면 AI가 아니라 팀이 고친다.

자동생성 코드 리뷰는 AI를 불신하는 절차가 아니다. 오히려 AI를 실무에 안전하게 들여오기 위한 협업 절차다. 빠른 초안을 받아들이되, 사람이 의미와 구조와 위험을 점검한다. 이 루프가 있어야 AI는 생산성 도구가 되고, 없으면 기술 부채 생성기가 된다.

리뷰의 첫 질문: 요구사항을 만족하는가

가장 먼저 볼 것은 요구사항이다. AI가 만든 코드는 그럴듯한 UI와 함수 이름을 만들 수 있지만, 실제 요구사항의 미묘한 조건을 놓칠 수 있다.

예를 들어 “검색 필터를 추가한다“는 요구사항에는 다음 조건이 숨어 있을 수 있다.

  • 검색 조건이 URL에 남아야 한다.
  • 뒤로 가기로 이전 조건이 복원되어야 한다.
  • 필터 변경 시 페이지는 1로 돌아가야 한다.
  • 빈 결과와 서버 오류는 다르게 보여야 한다.
  • 권한이 없는 사용자는 일부 필터를 볼 수 없어야 한다.

AI가 단순히 입력창과 select를 추가했다면 기능은 있어 보이지만 요구사항은 부족하다. 리뷰어는 “화면이 생겼는가“가 아니라 “사용자 흐름이 맞는가“를 봐야 한다.

기존 패턴과 맞는가

자동생성 코드는 종종 기존 코드베이스의 관습을 우회한다. 기존에는 apiClient를 사용하고 있는데 직접 fetch를 호출한다. 기존에는 Button 컴포넌트를 쓰는데 새 버튼 스타일을 만든다. 기존에는 TanStack Query를 쓰는데 useEffect와 useState로 서버 상태를 직접 관리한다.

이런 코드는 작동할 수 있다. 하지만 팀의 유지보수성을 해친다. 코드베이스의 패턴은 단순한 취향이 아니라 축적된 운영 방식이다. AI가 만든 코드가 더 짧거나 새로워 보여도 기존 질서를 깨뜨린다면 비용이 생긴다.

리뷰할 때는 다음을 확인한다.

  • 기존 공용 컴포넌트를 재사용했는가?
  • API 호출 방식이 기존과 같은가?
  • 에러 처리와 toast 방식이 일관적인가?
  • 파일 위치와 이름이 프로젝트 관습을 따르는가?
  • 테스트 스타일이 기존 테스트와 맞는가?

좋은 리뷰는 “이 코드 별로야“가 아니라 “이 프로젝트에서는 서버 상태를 query 훅으로 다루므로, 이 useEffect fetch는 기존 패턴과 맞지 않는다“처럼 말한다.

새 의존성은 정말 필요한가

AI는 문제를 풀기 위해 새 라이브러리를 제안할 수 있다. date formatting, form validation, animation, chart, drag and drop, utility 함수에 새 의존성을 추가하려고 할 수 있다. 하지만 의존성은 공짜가 아니다.

새 의존성을 검토할 때는 다음을 본다.

  • 이미 프로젝트에 비슷한 도구가 있는가?
  • 번들 크기에 영향이 큰가?
  • 보안과 유지보수 상태는 괜찮은가?
  • tree shaking이 되는가?
  • 팀이 학습해야 하는 API가 늘어나는가?
  • 작은 유틸리티로 충분한 문제인가?

AI가 “이 라이브러리를 설치하면 됩니다“라고 말해도 바로 받아들이지 않는다. 특히 프론트엔드에서는 번들 크기와 장기 유지보수가 중요하다.

상태와 데이터 흐름을 망치지 않았는가

자동생성 코드에서 가장 자주 보는 문제는 상태 중복이다. 서버에서 온 데이터를 지역 state에 복사하고, URL에 있어야 할 값을 컴포넌트 state에 두고, 파생 값을 별도 state로 저장한다.

리뷰어는 다음을 확인한다.

  • 서버 상태를 클라이언트 state로 복사하지 않았는가?
  • URL 상태와 form 상태가 불필요하게 중복되지 않는가?
  • 파생 상태를 저장해 동기화 문제가 생기지 않는가?
  • query key가 모든 입력 상태를 포함하는가?
  • mutation 후 캐시 업데이트가 있는가?

AI가 만든 코드가 “작동하는 데모“에서 “운영 가능한 기능“으로 올라가려면 상태 흐름이 정리되어야 한다.

실패 모드를 다루는가

AI 생성 코드는 성공 경로에 강하고 실패 경로에 약한 경우가 많다. 프론트엔드에서는 실패 경로가 사용자 경험의 큰 부분이다.

검토할 실패 모드는 다음과 같다.

  • 로딩 중 사용자가 다시 클릭하는 경우
  • API 요청 실패
  • 인증 만료
  • 권한 없음
  • 빈 데이터
  • 느린 네트워크
  • 오래된 응답
  • 컴포넌트 unmount 후 응답
  • optimistic update 실패
  • validation error

실패 모드를 다루는 코드는 보통 더 길다. AI가 만든 코드가 너무 짧고 매끈하다면 실패 경로가 빠져 있을 가능성이 있다.

접근성과 키보드 조작

AI가 만든 UI는 시각적으로 그럴듯할 수 있지만 접근성을 놓치기 쉽다. 버튼처럼 보이는 div, label 없는 input, focus trap 없는 modal, 키보드로 열 수 없는 dropdown, 스크린 리더 이름이 없는 icon button은 흔한 문제다.

리뷰할 때는 다음을 본다.

  • 클릭 가능한 요소가 실제 button이나 anchor인가?
  • input에 label이 연결되어 있는가?
  • icon button에 accessible name이 있는가?
  • modal과 dropdown의 focus 흐름이 맞는가?
  • 키보드만으로 주요 동작을 수행할 수 있는가?
  • ARIA를 과하게 쓰지 않았는가?

접근성은 마지막에 장식처럼 붙이는 것이 아니다. 컴포넌트 구조를 결정하는 기준이다.

보안과 개인정보

자동생성 코드는 보안 맥락을 모른 채 편리한 코드를 제안할 수 있다. 토큰을 localStorage에 저장하거나, 사용자 입력을 raw HTML로 렌더링하거나, 에러 객체 전체를 화면에 보여주거나, 클라이언트 번들에 secret을 넣는 식이다.

리뷰어는 최소한 다음을 확인해야 한다.

  • 민감한 값이 브라우저 저장소나 번들에 들어가지 않는가?
  • 사용자 입력과 외부 HTML이 안전하게 처리되는가?
  • 인증/권한 실패가 적절히 처리되는가?
  • 로그와 에러 메시지가 개인정보를 노출하지 않는가?
  • CORS나 CSRF 문제를 위험한 방식으로 우회하지 않는가?

프론트엔드 보안은 백엔드만의 일이 아니다. 브라우저에 들어가는 코드는 사용자가 볼 수 있고 조작할 수 있다.

테스트는 무엇을 보장하는가

AI에게 테스트를 만들게 하면 성공 케이스 중심의 테스트가 나올 수 있다. 리뷰어는 테스트가 있다는 사실보다 무엇을 보장하는지 봐야 한다.

좋은 테스트는 다음을 포함한다.

  • 사용자 행동 중심의 상호작용
  • 실패 상태
  • 빈 상태
  • 권한 또는 인증 상태
  • 비동기 race condition이 중요한 경우의 방어
  • 접근성 query
  • 회귀 가능성이 높은 비즈니스 규칙

테스트가 구현 세부 사항에 너무 강하게 묶여 있으면 리팩터링 때 쉽게 깨진다. 반대로 너무 표면적인 테스트는 중요한 버그를 놓친다. AI가 만든 테스트도 리뷰 대상이다.

리뷰 결과를 다시 AI에게 돌려보내기

자동생성 코드 리뷰의 장점은 리뷰 결과를 다시 AI에게 줄 수 있다는 것이다. 사람 리뷰어가 모든 수정을 직접 할 필요는 없다. 중요한 것은 피드백이 구체적이어야 한다.

나쁜 피드백:

더 깔끔하게 고쳐줘.

좋은 피드백:

이 변경에서 서버 상태를 지역 state로 복사한 부분을 제거해줘.
기존 `useProductListQuery` 패턴을 사용하고,
빈 결과와 API 실패 상태를 분리하며,
필터 변경 시 page query parameter가 1로 초기화되게 해줘.

AI는 구체적인 구조 피드백에 더 잘 반응한다. 리뷰어의 역할은 막연한 불만을 구조적 요청으로 바꾸는 것이다.

병합 전 최종 질문

자동생성 코드를 병합하기 전에 마지막으로 다음을 묻는다.

  • 이 코드를 내가 설명할 수 있는가?
  • 이 변경이 왜 필요한지 리뷰어가 이해할 수 있는가?
  • 실패했을 때 어디를 봐야 하는지 알 수 있는가?
  • 기존 코드베이스의 질서를 지키는가?
  • 검증하지 못한 위험을 알고 있는가?

“이해하지 못한 코드는 병합하지 않는다“는 원칙은 AI 시대에 더 중요하다. 코드를 적게 썼더라도 책임은 줄어들지 않는다.

12. 테스트와 검증 루프

AI 시대의 테스트는 더 중요해진다. 코드 생성 속도가 빨라질수록 검증 루프가 느슨하면 결함도 빠르게 누적된다. 테스트는 AI를 불신해서 만드는 것이 아니라, AI와 사람이 더 빠르게 협업하기 위한 안전장치다.

검증 없는 AI 개발은 빠른 초안 만들기에 머문다. 반대로 검증 루프가 있는 팀은 AI가 만든 변경을 더 자신 있게 받아들일 수 있다. 중요한 것은 모든 것을 테스트하는 것이 아니라, 어떤 위험을 어떤 검증으로 잡을지 아는 것이다.

검증은 여러 층이다

프론트엔드 검증은 하나의 도구로 끝나지 않는다. 여러 층이 서로 다른 종류의 위험을 잡는다.

  • 타입 검사: 값의 형태와 계약을 확인한다.
  • 린트: 코드 스타일, 훅 규칙, 잠재적 실수를 잡는다.
  • 단위 테스트: 순수 함수와 작은 로직을 확인한다.
  • 컴포넌트 테스트: 렌더링과 사용자 이벤트를 확인한다.
  • E2E 테스트: 실제 브라우저에서 주요 흐름을 확인한다.
  • 시각적 회귀 테스트: 레이아웃과 스타일 변화를 확인한다.
  • 접근성 검사: label, role, keyboard, contrast를 확인한다.
  • 성능 측정: 로딩, 상호작용, 번들 비용을 확인한다.

각 층은 한계가 있다. 타입 검사는 런타임 API 응답이 진짜 맞는지 보장하지 않는다. 단위 테스트는 브라우저에서 focus가 제대로 움직이는지 충분히 보지 못할 수 있다. E2E 테스트는 강력하지만 느리고 관리 비용이 크다. 따라서 검증 전략은 균형의 문제다.

타입 검사와 린트

타입 검사와 린트는 가장 빠른 방어선이다. AI가 만든 코드에서 잘못된 props, 누락된 return, 잘못된 union 처리, hook dependency 문제를 빠르게 잡을 수 있다.

하지만 타입과 린트가 통과한다고 기능이 맞는 것은 아니다. 타입은 코드의 모양을 확인하고, 린트는 규칙 위반을 찾는다. 사용자 경험과 비즈니스 요구사항은 별도의 테스트와 검토가 필요하다.

AI에게 코드를 만들게 할 때는 “타입 에러 없이“만 요구하지 말고 “타입이 도메인 상태를 명확히 표현하게” 요구해야 한다. 린트 경고를 없애기 위해 dependency array를 비우거나 eslint disable을 추가하는 식의 변경은 특히 조심해야 한다.

단위 테스트

단위 테스트는 순수 함수, adapter, parser, formatter, reducer, validation rule처럼 입력과 출력이 명확한 코드에 적합하다. 프론트엔드에서는 UI보다 주변 로직에 단위 테스트가 큰 효과를 낼 때가 많다.

예를 들어 URL query parameter 파싱 함수는 단위 테스트하기 좋다.

expect(parseProductSearchParams("?q=react&page=2")).toEqual({
  q: "react",
  page: 2,
  sort: "latest",
});

이런 테스트는 AI가 코드를 수정해도 핵심 계약을 지켜준다. 특히 adapter와 schema validation은 단위 테스트의 좋은 대상이다. 외부 API와 내부 모델 사이의 경계가 깨지면 많은 UI가 영향을 받기 때문이다.

컴포넌트 테스트

컴포넌트 테스트는 사용자가 보는 것과 하는 일을 기준으로 작성해야 한다. 내부 state 이름이나 함수 호출보다, 화면에 무엇이 보이고 사용자가 어떤 행동을 할 수 있는지를 확인한다.

좋은 컴포넌트 테스트는 다음과 같은 질문에 답한다.

  • 사용자가 필요한 정보를 볼 수 있는가?
  • 버튼이나 입력을 조작하면 기대한 변화가 생기는가?
  • 로딩, 에러, 빈 상태가 표시되는가?
  • 접근 가능한 이름과 role로 요소를 찾을 수 있는가?

Testing Library의 철학은 구현 세부 사항보다 사용자 관점에 가깝게 테스트하는 것이다. AI에게 테스트를 작성하게 할 때도 “state가 바뀌는지 확인해줘“보다 “사용자가 검색어를 입력하고 검색 결과를 보는 흐름을 테스트해줘“라고 요청하는 편이 낫다.

E2E 테스트

E2E 테스트는 실제 브라우저에서 사용자의 핵심 흐름을 검증한다. 로그인, 상품 검색, 결제, 설정 저장, 권한별 화면 접근 같은 흐름은 E2E 테스트의 좋은 후보이다.

E2E 테스트는 강력하지만 모든 것을 E2E로 검증하면 느려지고 불안정해진다. 따라서 핵심 사용자 여정과 회귀 비용이 큰 기능에 집중한다.

AI 시대의 E2E 테스트는 특히 중요하다. AI가 여러 파일을 빠르게 수정할 때 작은 단위 테스트가 통과해도 실제 브라우저 흐름이 깨질 수 있다. Playwright 같은 도구는 브라우저에서 실제 클릭, 입력, navigation, network mocking을 검증할 수 있다.

실패 케이스를 요청하기

AI에게 테스트를 작성하게 하면 성공 케이스만 나오는 경우가 많다. 사람은 실패 케이스를 명시해야 한다.

예를 들어 상품 검색 테스트를 요청한다면 다음을 포함한다.

  • 검색 결과가 있을 때 목록을 보여준다.
  • 결과가 없을 때 빈 상태를 보여준다.
  • API가 실패하면 에러와 재시도 버튼을 보여준다.
  • 필터를 바꾸면 page가 1로 초기화된다.
  • 뒤로 가기로 이전 검색 조건이 복원된다.
  • 빠른 입력에서도 오래된 응답이 최신 결과를 덮어쓰지 않는다.

테스트는 요구사항을 코드로 바꾸는 작업이다. AI에게 테스트를 맡기더라도 요구사항의 실패 모드는 사람이 제공해야 한다.

시각적 회귀와 접근성

프론트엔드에서는 “기능은 되지만 화면이 깨진” 문제가 자주 생긴다. 시각적 회귀 테스트는 레이아웃, 간격, 색, 겹침, responsive breakpoint 문제를 잡는 데 도움을 준다. 모든 화면에 도입하기 어렵다면 핵심 컴포넌트와 자주 깨지는 화면부터 시작할 수 있다.

접근성 검증도 자동화할 수 있는 부분과 수동 확인이 필요한 부분이 있다. 자동 도구는 label 누락, role 문제, contrast 일부를 잡아준다. 하지만 실제 키보드 흐름, 스크린 리더 경험, focus management는 사람이 확인해야 한다.

AI가 만든 UI를 검증할 때는 최소한 다음을 본다.

  • 키보드만으로 주요 흐름을 사용할 수 있는가?
  • focus가 예측 가능하게 이동하는가?
  • icon button에 accessible name이 있는가?
  • modal이 열렸을 때 배경으로 focus가 새지 않는가?
  • 색만으로 상태를 전달하지 않는가?

성능 검증

성능은 느낌만으로 판단하기 어렵다. 빠른 개발 환경에서는 괜찮아 보여도 실제 사용자의 기기와 네트워크에서는 느릴 수 있다. 성능 검증은 번들 크기, 렌더링 비용, network waterfall, Core Web Vitals, interaction delay를 함께 본다.

AI가 만든 코드에서 성능 위험이 큰 지점은 다음과 같다.

  • 큰 라이브러리 추가
  • 렌더링마다 비싼 계산 수행
  • 이미지 크기와 lazy loading 누락
  • 불필요한 클라이언트 컴포넌트 증가
  • debounce/throttle 없는 빈번한 이벤트 처리
  • key가 불안정한 목록 렌더링

성능 테스트는 항상 완벽할 수 없다. 하지만 변경이 성능에 영향을 줄 가능성이 있다면 최소한 위험을 언급하고 측정 방법을 제안해야 한다.

검증 결과를 기록하기

검증은 실행하는 것만큼 기록하는 것도 중요하다. 코드 리뷰나 PR 설명에는 다음이 남아야 한다.

검증:
- 타입 검사 통과
- 컴포넌트 테스트 통과
- 상품 검색 E2E 통과
- 브라우저에서 필터/뒤로 가기/빈 상태 확인

미검증:
- 모바일 Safari 수동 확인은 하지 못함
- 실제 운영 API의 500 응답은 mock으로만 확인

미검증 항목을 기록하는 것은 약점이 아니라 정직함이다. 모든 것을 확인하지 못했다면 무엇을 확인하지 못했는지 알아야 한다.

AI 생성 테스트 리뷰 질문

AI가 만든 테스트를 리뷰할 때는 다음 질문을 던진다.

  • 테스트가 사용자의 행동과 결과를 중심으로 작성되었는가?
  • 성공 케이스뿐 아니라 실패 케이스도 있는가?
  • mock 데이터가 실제 계약과 맞는가?
  • 구현 세부 사항에 과하게 묶여 있지 않은가?
  • 비동기 흐름을 안정적으로 기다리는가?
  • 접근성 query를 사용할 수 있는 곳에서 사용했는가?
  • 테스트 이름이 요구사항을 설명하는가?
  • 중요한 회귀 위험을 실제로 잡을 수 있는가?

좋은 검증 루프는 개발자를 느리게 만들지 않는다. 오히려 AI가 만든 변경을 더 빨리 받아들일 수 있게 한다.

13. 성능, 접근성, 보안의 최소 기준

프론트엔드 품질은 기능이 작동하는지로 끝나지 않는다. 빠르게 로드되는가, 키보드와 스크린 리더로 사용할 수 있는가, 민감한 정보를 안전하게 다루는가도 품질의 일부다. AI가 만든 UI는 시각적으로 그럴듯해도 이 세 기준을 놓치는 경우가 많다.

이 장은 성능, 접근성, 보안을 전문가 수준으로 깊게 다루지는 않는다. 대신 프론트엔드 개발자가 자동생성 코드를 받아들일 때 반드시 지켜야 할 최소 기준을 정리한다. 최소 기준은 팀이 반복해서 적용할 수 있어야 한다. 체크리스트, 자동 검사, 리뷰 질문, 브라우저 확인으로 연결되어야 한다.

최소 기준이 필요한 이유

AI가 만든 코드는 빠르게 늘어난다. 빠르게 늘어나는 코드에 매번 깊은 전문가 리뷰를 붙일 수는 없다. 그래서 팀에는 최소 기준이 필요하다. 최소 기준은 “이 정도는 항상 확인한다“는 약속이다.

예를 들어 다음 같은 기준을 둘 수 있다.

  • 새 이미지 UI는 width/height 또는 aspect ratio를 가져야 한다.
  • 새 interactive element는 keyboard로 조작 가능해야 한다.
  • 새 form input은 label과 error message 연결을 가져야 한다.
  • 새 API 호출은 로딩/에러/빈 상태를 구분해야 한다.
  • 새 외부 의존성은 번들 크기와 대체재를 검토해야 한다.
  • 클라이언트 번들에는 secret이 들어가면 안 된다.

이 기준은 완벽하지 않지만 반복 가능하다. 반복 가능한 기준이 있어야 AI 생성 코드도 팀의 품질 체계 안에 들어온다.

성능 최소 기준

성능은 사용자 경험이다. 사용자는 “React가 느리다“거나 “hydration 비용이 크다“고 말하지 않는다. 그냥 화면이 늦게 뜨고, 클릭이 늦게 반응하고, 스크롤이 끊긴다고 느낀다.

프론트엔드에서 최소한 확인해야 할 성능 기준은 다음과 같다.

  • 이미지와 비디오의 크기, 비율, lazy loading이 적절한가?
  • 큰 라이브러리를 새로 추가하지 않았는가?
  • 렌더링마다 비싼 계산을 반복하지 않는가?
  • 목록 rendering에 안정적인 key를 사용하는가?
  • 검색, 스크롤, resize 이벤트에 debounce/throttle 또는 적절한 최적화가 있는가?
  • 클라이언트에서 실행할 필요 없는 코드를 브라우저로 보내지 않는가?
  • LCP, INP, CLS에 영향을 줄 수 있는 변경인지 검토했는가?

AI가 만든 UI는 종종 데모 데이터에서는 빠르다. 하지만 실제 데이터가 100개, 1,000개로 늘어나면 달라진다. 성능 리뷰는 “지금 내 컴퓨터에서 괜찮다“보다 “사용자 조건에서 어떤 비용이 생기는가“를 묻는다.

접근성 최소 기준

접근성은 별도의 기능이 아니다. UI가 제대로 만들어졌는지를 보는 기준이다. HTML의 의미를 올바르게 사용하면 접근성뿐 아니라 테스트와 유지보수도 좋아진다.

최소 기준은 다음과 같다.

  • 클릭 가능한 것은 button 또는 적절한 a 요소를 사용한다.
  • form input에는 label이 있다.
  • error message는 관련 input과 연결된다.
  • icon-only button에는 접근 가능한 이름이 있다.
  • modal, dropdown, popover는 focus 흐름을 고려한다.
  • 키보드만으로 주요 기능을 사용할 수 있다.
  • 색상만으로 상태를 전달하지 않는다.
  • ARIA는 native HTML로 해결할 수 없을 때만 신중히 사용한다.

AI는 때때로 div에 onClick을 붙여 버튼처럼 만든다. 시각적으로는 같아 보여도 키보드와 스크린 리더 경험은 다르다. 리뷰어는 눈에 보이는 모양뿐 아니라 상호작용의 의미를 봐야 한다.

보안 최소 기준

프론트엔드 보안의 최소 기준은 “브라우저에 들어가는 것은 사용자가 볼 수 있고 조작할 수 있다“는 사실에서 출발한다.

확인할 항목은 다음과 같다.

  • secret, private token, 서버 전용 API key가 클라이언트 번들에 들어가지 않는가?
  • 인증 정보를 localStorage에 저장하는 결정을 무심코 하지 않았는가?
  • 사용자 입력이나 외부 HTML을 안전하게 렌더링하는가?
  • dangerouslySetInnerHTML 또는 raw HTML 렌더링에 sanitize가 있는가?
  • URL을 만들 때 위험한 scheme을 허용하지 않는가?
  • 에러 메시지와 로그가 개인정보를 노출하지 않는가?
  • dependency 추가 시 보안과 유지보수 상태를 확인했는가?

AI가 보안에 취약한 코드를 악의적으로 만드는 것은 아니다. 다만 맥락을 모른 채 가장 흔한 예제를 제안할 수 있다. 흔한 예제가 안전한 예제라는 뜻은 아니다.

성능, 접근성, 보안은 서로 연결된다

이 세 기준은 따로 떨어져 있지 않다. 의미 있는 HTML은 접근성을 높이고 테스트를 쉽게 하며, JavaScript 의존을 줄여 성능에도 도움이 될 수 있다. 이미지 크기와 layout 안정성은 성능 지표뿐 아니라 사용자의 조작 안정성과도 연결된다. 안전한 데이터 처리는 보안뿐 아니라 에러 UX와 운영 신뢰성에도 영향을 준다.

예를 들어 모달을 생각해보자. 성능 관점에서는 불필요한 렌더링과 무거운 content loading을 피해야 한다. 접근성 관점에서는 focus trap, aria 속성, 키보드 닫기, 배경 inert 처리를 봐야 한다. 보안 관점에서는 모달 안에 렌더링하는 외부 HTML이나 사용자 입력을 확인해야 한다. 하나의 UI에도 세 기준이 함께 들어 있다.

AI에게 품질 검토를 요청하는 법

AI에게도 품질 검토를 시킬 수 있다. 다만 “문제 찾아줘“보다 기준을 주는 편이 좋다.

이 컴포넌트를 성능, 접근성, 보안 관점에서 검토해줘.
특히 이미지 layout shift, keyboard navigation, icon button label,
사용자 입력 렌더링, 클라이언트 번들 secret 노출 가능성을 확인해줘.
확실하지 않은 부분은 실제 브라우저나 도구로 확인해야 한다고 표시해줘.

AI의 검토는 1차 점검이다. 실제 접근성은 키보드로 조작해보고, 실제 성능은 브라우저 도구로 측정하고, 실제 보안은 팀의 보안 기준과 함께 확인해야 한다.

팀 체크리스트로 만들기

최소 기준은 개인의 기억에만 의존하면 사라진다. 팀은 이를 체크리스트와 자동화로 만들어야 한다.

  • PR template에 성능/접근성/보안 확인 항목을 넣는다.
  • lint와 type check를 CI에 넣는다.
  • 핵심 UI에 컴포넌트 테스트와 E2E 테스트를 둔다.
  • 접근성 자동 검사를 가능한 범위에서 추가한다.
  • 번들 분석 또는 dependency review를 주기적으로 한다.
  • 보안상 민감한 저장소 사용 규칙을 문서화한다.

AI 시대에는 이런 운영 장치가 더 중요하다. 사람이 모든 줄을 직접 쓰지 않는다면, 시스템이 최소 품질을 반복해서 확인해야 한다.

병합 전 최소 질문

성능, 접근성, 보안 관점에서 병합 전 다음 질문을 던진다.

  • 이 변경이 사용자에게 느려짐으로 느껴질 수 있는가?
  • 주요 상호작용을 키보드로 수행할 수 있는가?
  • 스크린 리더가 이해할 수 있는 이름과 구조가 있는가?
  • 브라우저에 저장하거나 노출하는 값이 안전한가?
  • 외부 입력을 HTML이나 URL로 사용할 때 검증하는가?
  • 새 의존성은 정말 필요한가?
  • 자동 검사로 잡지 못하는 부분을 수동으로 확인했는가?

이 질문은 AI가 만든 코드에만 적용되는 것이 아니다. 모든 프론트엔드 코드에 적용된다. 다만 AI가 코드 생성 속도를 높였기 때문에, 이 질문을 더 자주 더 체계적으로 던져야 한다.

14. 에이전트와 함께 일하는 법

AI 에이전트는 단순한 자동완성보다 강력하다. 파일을 읽고, 변경하고, 테스트를 실행하고, 결과를 요약할 수 있다. 어떤 에이전트는 브라우저를 열어 동작을 확인하고, 어떤 에이전트는 터미널 명령을 실행하며, 어떤 에이전트는 여러 파일에 걸친 리팩터링을 수행한다. 이 능력은 프론트엔드 개발자의 작업 방식을 크게 바꾼다.

하지만 강력한 도구일수록 작업 단위와 경계를 분명히 해야 한다. 에이전트는 넓은 범위를 빠르게 건드릴 수 있기 때문에, 좋은 지시와 좋은 검증이 없으면 잘못된 변경도 빠르게 확산된다. 에이전트와 잘 일한다는 것은 프롬프트를 멋지게 쓰는 일이 아니라, 개발 작업을 명확한 목표와 경계와 완료 조건으로 운영하는 일이다.

에이전트는 동료가 아니라 실행자다

에이전트를 사람 동료처럼 대화할 수는 있지만, 실제 운영에서는 실행자로 보는 편이 안전하다. 동료는 제품 맥락, 팀의 역사, 암묵적 제약, 정치적 판단, 운영 사고의 기억을 갖고 있다. 에이전트는 주어진 컨텍스트와 도구 안에서 추론하고 실행한다. 이 차이를 잊으면 위험하다.

에이전트에게 맡기기 좋은 일은 다음과 같다.

  • 관련 파일 탐색
  • 기존 패턴 요약
  • 작은 기능 초안 구현
  • 반복적인 테스트 추가
  • 타입 정리
  • 리팩터링 후보 제안
  • 변경 요약 작성

반대로 사람이 붙잡아야 하는 일은 다음과 같다.

  • 제품 요구사항의 우선순위
  • 사용자 경험의 최종 판단
  • 보안과 개인정보 결정
  • 새 아키텍처 경계 도입
  • 새 의존성 승인
  • 배포와 운영 리스크 판단

에이전트는 일을 실행할 수 있다. 하지만 어떤 일이 맞는지, 어떤 위험을 감수할지, 어떤 수준이면 충분한지는 사람이 결정해야 한다.

작업을 헌장으로 시작하기

에이전트 작업은 작은 작업 헌장으로 시작하는 것이 좋다. 작업 헌장은 길 필요가 없지만 다음을 포함해야 한다.

목표:
사용자 또는 제품 관점에서 무엇이 가능해져야 하는가?

범위:
수정해도 되는 화면, 컴포넌트, 파일, API는 무엇인가?

기존 패턴:
따라야 할 컴포넌트, 훅, 타입, 테스트 스타일은 무엇인가?

완료 조건:
어떤 동작, 테스트, 문서가 확인되어야 끝났다고 볼 수 있는가?

제약:
건드리지 말아야 할 파일, 추가하면 안 되는 의존성, 바꾸면 안 되는 API는 무엇인가?

보고:
마지막에 어떤 형식으로 변경과 검증 결과를 보고해야 하는가?

이 헌장은 에이전트를 위한 지시이면서 개발자 자신을 위한 정리다. 이 정도도 쓰기 어렵다면 작업이 아직 흐릿한 것이다.

탐색, 계획, 구현, 검증을 나눈다

큰 작업을 한 번에 시키면 에이전트는 추측을 많이 한다. 추측이 많을수록 변경 위험이 커진다. 그래서 탐색, 계획, 구현, 검증을 나누는 것이 좋다.

탐색 단계의 요청:

아직 파일을 수정하지 말고 관련 코드를 찾아줘.
진입점, 데이터 흐름, 상태 위치, 주요 타입, 테스트 파일을 정리하고,
확실하지 않은 부분은 추측이라고 표시해줘.

계획 단계의 요청:

찾은 코드 기준으로 변경 계획을 제안해줘.
변경할 파일, 변경 이유, 위험, 필요한 테스트를 나눠서 써줘.
아직 코드는 수정하지 마.

구현 단계의 요청:

계획의 1-2번만 먼저 구현해줘.
기존 패턴을 유지하고 새 의존성은 추가하지 마.

검증 단계의 요청:

변경된 코드의 타입, 테스트, 접근성, 상태 흐름을 점검해줘.
실행한 검증과 실행하지 못한 검증을 구분해서 보고해줘.

이렇게 나누면 개발자는 중간에 방향을 바로잡을 수 있다. 에이전트도 더 적은 추측으로 작업한다.

좋은 요청은 경계를 가진다

에이전트에게 가장 나쁜 요청은 “전체를 좋게 고쳐줘“다. 이 요청은 목표도 범위도 완료 조건도 없다. 에이전트는 넓게 바꿀 수밖에 없고, 개발자는 결과를 검토하기 어려워진다.

좋은 요청은 다음처럼 경계를 가진다.

상품 검색 화면에서 URL 상태와 폼 상태가 중복되는 문제를 정리해줘.
`useProductSearchParams`와 `ProductFilterPanel`만 우선 확인하고,
검색어/카테고리/정렬/페이지는 URL query parameter를 단일 출처로 삼아줘.
필터 변경 시 page는 1로 초기화되어야 한다.
기존 테스트 스타일에 맞춰 실패할 수 있는 테스트를 추가해줘.

이 요청에는 목표, 범위, 상태 규칙, 완료 조건이 들어 있다. 에이전트는 더 정확히 일할 수 있고, 사람은 결과를 더 쉽게 평가할 수 있다.

에이전트의 요약을 믿지 말고 diff를 본다

에이전트는 작업이 끝나면 요약을 제공한다. 이 요약은 유용하지만 증거는 아니다. 실제 증거는 diff, 테스트 결과, 브라우저 동작, 타입 검사 결과다.

요약을 읽을 때는 다음을 확인한다.

  • 실제로 변경된 파일과 요약이 일치하는가?
  • 의도하지 않은 파일이 수정되지 않았는가?
  • 새 의존성이나 설정 변경이 숨어 있지 않은가?
  • 검증했다고 말한 명령이 실제로 실행되었는가?
  • 실패하거나 생략한 검증이 보고되었는가?

에이전트가 틀렸다는 전제를 둘 필요는 없다. 다만 에이전트의 보고를 코드 리뷰의 출발점으로 삼아야지, 리뷰의 대체물로 삼으면 안 된다.

실패했을 때 더 작게 돌아간다

에이전트가 잘못된 방향으로 가면 많은 사람이 더 긴 설명을 추가한다. 때로는 필요하지만, 더 좋은 첫 대응은 작업을 더 작게 나누는 것이다.

예를 들어 에이전트가 검색 화면 전체를 잘못 고쳤다면 다음처럼 되돌아간다.

전체 구현은 멈추고, 먼저 URL query parameter 파싱 함수만 작성해줘.
입력과 출력 예시는 다음과 같아.
테스트도 이 함수에 대해서만 추가해줘.

작은 작업은 실패 비용이 낮고 검토가 쉽다. AI 시대의 개발자는 큰 요청을 잘 쓰는 사람보다, 큰 문제를 좋은 작은 작업으로 나누는 사람에 가깝다.

팀에서 에이전트를 운영하는 규칙

개인 생산성을 넘어 팀에서 에이전트를 쓰려면 운영 규칙이 필요하다.

  • 에이전트가 만든 코드도 동일한 리뷰 기준을 통과해야 한다.
  • 에이전트가 실행한 검증과 생략한 검증을 PR에 기록한다.
  • 새 의존성, 보안, 개인정보 관련 변경은 사람이 명시적으로 승인한다.
  • 대규모 리팩터링은 탐색/계획 단계의 산출물을 먼저 리뷰한다.
  • 에이전트에게 제공할 프로젝트 지식베이스를 관리한다.
  • 반복되는 작업은 템플릿과 체크리스트로 만든다.

팀의 목표는 “AI를 많이 쓰는 팀“이 아니라 “AI가 들어와도 코드 품질과 운영 신뢰성을 유지하는 팀“이다.

프로젝트 지식베이스 만들기

에이전트는 컨텍스트가 좋을수록 더 잘 일한다. 프로젝트 지식베이스에는 다음 내용을 담을 수 있다.

  • 폴더 구조와 feature 경계
  • API client 사용 규칙
  • 서버 상태와 URL 상태 관리 방식
  • 디자인 시스템 컴포넌트 사용 규칙
  • 테스트 작성 패턴
  • 접근성 최소 기준
  • 새 의존성 추가 기준
  • 배포와 검증 명령
  • 자주 발생한 실패 모드

이 지식베이스는 사람에게도 유용하다. 좋은 AI 컨텍스트는 좋은 온보딩 문서와 거의 같다. 결국 AI 친화적인 팀은 사람 친화적인 팀이다.

에이전트 작업의 완료 보고

에이전트 작업은 다음 형식으로 마무리하면 좋다.

변경:
- 무엇을 바꿨는가

이유:
- 왜 이렇게 바꿨는가

검증:
- 실행한 명령과 확인한 사용자 흐름

미검증:
- 확인하지 못한 환경, 브라우저, 실패 케이스

남은 위험:
- 리뷰어가 특히 봐야 할 지점

이 형식은 에이전트뿐 아니라 사람 개발자에게도 좋다. 코드 리뷰어가 빠르게 맥락을 이해하고, 나중에 문제가 생겼을 때 변경의 의도를 추적할 수 있다.

15. AI 시대의 일정 산정

프론트엔드 개발에서 일정 산정은 늘 어려웠다. 화면은 사용자가 직접 만지는 제품의 끝단에 있고, 백엔드 API, 디자인, 기획, 데이터, 인증, 브라우저, 배포 일정의 영향을 모두 받는다. 그래서 프론트엔드 작업은 종종 최종 병목이 되었다. 앞단의 의사결정이 늦어져도 마지막에는 화면이 완성되어야 하고, 일정은 프론트엔드에서 압축되기 쉬웠다.

이런 환경에서는 정확한 일정 산정이 큰 의미를 가졌다. 언제 끝나는지 알아야 출시일을 잡을 수 있고, 디자인과 API와 QA와 배포를 맞출 수 있었다. 일정 산정은 단순한 관리 행위가 아니라, 프로젝트의 마지막 병목을 다루는 생존 기술이었다.

AI 시대에는 이 의미가 사라지는 것이 아니라 바뀐다. 코드 초안 작성 속도는 빨라질 수 있다. 하지만 제품이 완성되는 속도 전체가 같은 비율로 빨라진다고 보기는 어렵다. 일정 산정의 중심은 “코드를 쓰는 데 며칠 걸리는가“에서 “불확실성을 줄이고, 생성된 결과를 검증하고, 시스템 안에 안전하게 통합하는 데 얼마나 걸리는가“로 이동한다.

예전의 병목: 끝단 구현

전통적인 프론트엔드 일정에서 큰 병목은 구현량이었다. 디자인이 확정되고 API가 준비되면, 프론트엔드 개발자는 화면을 만들고 상태를 연결하고 예외를 처리하고 테스트하고 QA 이슈를 고쳤다. 특히 마감이 가까울수록 프론트엔드가 모든 변경의 접점이 되었다.

이때 일정 산정은 주로 다음을 묻는 일이었다.

  • 화면이 몇 개인가?
  • 컴포넌트가 얼마나 복잡한가?
  • API가 준비되어 있는가?
  • 상태와 폼이 얼마나 많은가?
  • QA와 브라우저 대응이 얼마나 걸릴 것인가?
  • 디자인 변경이 얼마나 남아 있는가?

물론 이 질문들은 여전히 중요하다. 하지만 AI가 컴포넌트 초안, 테스트 초안, 타입 정의, 반복 스타일링을 빠르게 만들어주기 시작하면서 “순수 작성 시간“의 비중은 줄어들 수 있다. 문제는 줄어든 시간이 프로젝트 전체 일정에서 그대로 절약되느냐는 것이다.

새 병목: 판단과 통합

AI 시대의 새 병목은 판단과 통합이다. AI가 만든 코드는 빠르게 나온다. 하지만 그 코드가 요구사항을 정확히 이해했는지, 기존 코드베이스의 패턴을 지키는지, 상태를 올바른 위치에 두었는지, 접근성과 성능과 보안을 해치지 않는지 확인해야 한다.

예를 들어 검색 필터 UI를 AI가 하루가 아니라 한 시간 안에 만들 수 있다고 하자. 그래도 다음 작업은 남아 있다.

  • 검색 조건이 URL에 남아야 하는지 판단한다.
  • 서버 상태와 지역 상태를 분리한다.
  • API 실패, 빈 결과, 권한 없음, 느린 네트워크를 확인한다.
  • 디자인 시스템 컴포넌트와 맞춘다.
  • 접근성 label과 키보드 흐름을 확인한다.
  • 테스트를 추가하고 실제 브라우저에서 검증한다.
  • 기존 코드와 충돌하는 패턴을 정리한다.
  • 리뷰어가 이해할 수 있게 변경 이유를 설명한다.

AI가 초안을 빠르게 만든다고 해서 이 작업들이 사라지지는 않는다. 오히려 생성 속도가 빨라질수록 검증과 통합을 체계화하지 않으면 품질 문제가 더 빠르게 쌓인다.

일정 산정의 단위가 바뀐다

AI 이전의 일정 산정은 구현 항목을 기준으로 하는 경우가 많았다.

상품 목록 화면: 2일
필터 UI: 1일
상세 모달: 1일
테스트와 QA: 1일

AI 시대에는 이보다 작업의 성격을 나누는 산정이 더 유용하다.

요구사항 정리와 불확실성 제거: 0.5일
기존 코드 탐색과 변경 계획: 0.5일
AI 초안 생성과 1차 구현: 0.5일
상태/타입/계약 정리: 1일
검증과 실패 케이스 보강: 1일
리뷰 반영과 통합: 0.5일

여기서 중요한 변화는 구현 시간이 전체의 전부가 아니라는 점이다. AI는 “초안 생성“을 줄일 수 있지만, 요구사항 정리, 기존 코드 탐색, 검증, 리뷰, 통합은 여전히 별도 항목으로 산정해야 한다.

일정은 약속이 아니라 불확실성 모델이다

일정 산정을 약속으로만 보면 개발자는 방어적으로 변한다. “언제 끝나요?“라는 질문에 정확한 날짜를 말해야 한다고 느끼면, 불확실성을 숨기거나 과도한 버퍼를 넣게 된다. 하지만 좋은 일정 산정은 약속 이전에 불확실성 모델이다.

프론트엔드 작업의 불확실성은 보통 다음에서 나온다.

  • 요구사항이 아직 모호하다.
  • 디자인이 확정되지 않았다.
  • API 계약이 바뀔 수 있다.
  • 기존 코드 구조를 아직 모른다.
  • 브라우저나 접근성 이슈가 숨어 있다.
  • AI가 만든 초안의 품질을 아직 모른다.
  • 테스트 환경이나 mock 데이터가 부족하다.
  • 리뷰와 QA에서 어떤 문제가 나올지 모른다.

따라서 일정 산정은 “3일입니다“보다 “현재 불확실성 기준으로 3-5일이며, API 계약과 빈 상태 정책이 확정되면 범위를 줄일 수 있습니다“에 가까워져야 한다. AI 시대에는 특히 그렇다. 초안은 빠르게 만들 수 있지만, 초안 이후의 위험은 작업을 열어보기 전까지 알기 어렵다.

AI가 줄이는 시간과 줄이지 못하는 시간

AI가 줄일 수 있는 시간은 대체로 다음과 같다.

  • 반복적인 코드 작성
  • boilerplate 생성
  • 비슷한 테스트 케이스 초안
  • 타입 정의 초안
  • 문서와 체크리스트 초안
  • 기존 코드 요약
  • 리팩터링 후보 제안

AI가 줄이기 어려운 시간은 다음과 같다.

  • 요구사항의 우선순위 결정
  • 사용자 경험의 판단
  • 팀의 기존 구조와 히스토리 이해
  • 보안과 개인정보 판단
  • 실패 모드의 실제 검증
  • 브라우저와 기기별 확인
  • 조직 간 의사결정
  • 리뷰와 합의

일정 산정은 이 둘을 구분해야 한다. “AI를 쓰면 절반으로 줄어들까요?“라는 질문에는 “어떤 시간이 줄어드는가“로 답해야 한다. 작성 시간이 50% 줄어도 검증과 통합 시간이 그대로라면 전체 일정은 50% 줄지 않는다.

산정 전에 AI로 할 수 있는 일

AI는 일정 산정 자체에도 도움을 줄 수 있다. 다만 AI에게 “이거 며칠 걸려?“라고 묻는 것은 좋지 않다. AI는 팀의 코드베이스, 개발자의 숙련도, 리뷰 속도, 조직의 의사결정, QA 환경을 충분히 알지 못한다.

더 좋은 요청은 산정에 필요한 정보를 정리하게 하는 것이다.

아직 구현하지 말고 이 기능의 일정 산정을 돕기 위해 관련 코드를 탐색해줘.
변경이 필요한 파일, 상태/타입/API 위험, 테스트 필요 항목,
확정되지 않은 요구사항, 예상 실패 모드를 표로 정리해줘.
확실하지 않은 것은 추측이라고 표시해줘.

이 요청은 AI에게 날짜를 맞히게 하지 않는다. 대신 산정에 필요한 재료를 모으게 한다. 일정 산정의 품질은 답변의 숫자가 아니라, 드러난 불확실성의 품질에서 나온다.

이 템플릿의 핵심은 “AI가 빠르게 해줄 부분“과 “사람이 책임져야 할 부분“을 분리하는 것이다.

일정 재산정의 주기가 짧아진다

AI 시대에는 첫 산정이 더 정확해지는 것이 아니라, 재산정 주기가 짧아질 수 있다. 에이전트가 빠르게 탐색하고 초안을 만들면, 우리는 더 이른 시점에 실제 복잡도를 볼 수 있다. 예전에는 이틀을 구현하고 나서야 알던 문제를, 이제는 몇 시간 안에 발견할 수 있다.

따라서 좋은 운영 방식은 초기 산정 하나에 집착하지 않고, 산정 체크포인트를 명시하는 것이다.

  • 탐색 후 재산정
  • AI 초안 후 재산정
  • 테스트 실패 모드 확인 후 재산정
  • 첫 리뷰 후 재산정
  • QA 첫 라운드 후 재산정

이렇게 하면 일정 산정은 예측 게임이 아니라 학습 루프가 된다. AI는 이 학습 루프의 속도를 높인다.

프론트엔드 병목은 사라지는가

AI가 코드를 빠르게 만들어도 프론트엔드 병목이 완전히 사라지지는 않는다. 병목의 위치가 바뀐다.

예전의 병목:

  • 화면 구현량
  • 스타일링 반복
  • 상태 연결
  • 테스트 작성 속도

새로운 병목:

  • 요구사항과 디자인의 모호성
  • API와 UI 계약의 불일치
  • 자동생성 코드의 리뷰와 검증
  • 접근성, 성능, 보안의 품질 기준
  • 여러 팀 사이의 의사결정
  • 출시 전 통합과 QA

즉, 프론트엔드는 여전히 끝단이다. 다만 끝단에서 손으로 코드를 쓰는 시간이 줄고, 끝단에서 시스템 전체의 모순을 드러내고 정리하는 시간이 더 중요해진다.

팀과 대화하는 방식도 바뀐다

AI 시대의 일정 대화는 “AI 쓰면 빨리 되죠?“라는 기대와 자주 부딪힌다. 이때 개발자는 방어적으로 “그래도 오래 걸립니다“라고만 말하기보다, 시간이 어디에 쓰이는지 분해해서 보여줘야 한다.

예를 들어 이렇게 말할 수 있다.

AI로 화면 초안은 빠르게 만들 수 있습니다.
다만 이 기능은 검색 조건을 URL과 동기화해야 하고,
API 실패/빈 결과/권한 없음 상태를 분리해야 하며,
기존 서버 상태 캐시와 연결해야 합니다.
초안은 반나절 안에 만들 수 있지만,
검증 가능한 제품 코드로 통합하는 데 2-3일이 필요합니다.

이 설명은 AI를 부정하지 않는다. 동시에 AI가 줄이지 못하는 책임을 드러낸다. 일정 산정은 개발자의 방어 수단이 아니라, 제품 리스크를 함께 보는 언어가 되어야 한다.

일정 산정의 새로운 의미

AI 이전의 일정 산정은 종종 “끝단 구현을 언제 끝낼 수 있는가“에 가까웠다. AI 시대의 일정 산정은 “어떤 불확실성을 언제 줄이고, 어떤 수준의 검증을 거쳐, 언제 책임질 수 있는 코드로 만들 수 있는가“에 가까워진다.

그래서 일정 산정은 더 이상 단순한 날짜 맞히기가 아니다. 일정 산정은 작업을 이해하는 방식이고, 리스크를 드러내는 방식이며, AI가 만든 빠른 초안을 제품 코드로 바꾸는 데 필요한 책임의 언어다.

AI가 구현 속도를 높일수록 일정 산정은 사라지지 않는다. 오히려 더 정교해져야 한다. 빠르게 만들 수 있는 것과 책임질 수 있는 것 사이의 차이를 설명하는 일이 중요해지기 때문이다.

16. 사례 연구: 기능 하나를 끝까지 만들기

이 장은 책의 앞부분에서 다룬 내용을 하나의 기능 개발 흐름에 적용한다. 예제 기능은 “상품 목록에서 검색, 필터, 정렬, 페이지네이션, 즐겨찾기, 빈 상태, 에러 상태를 제공하는 화면“이다. 이 기능은 특별히 화려하지 않지만 대부분의 프론트엔드 애플리케이션에서 반복되는 문제를 포함한다.

이 사례 연구의 목적은 완벽한 예제 앱을 만드는 것이 아니다. AI와 함께 기능을 만들 때 사람이 어떤 순간에 판단해야 하는지 보여주는 것이다. AI가 초안을 만들 수는 있지만, 상태의 경계, API 계약, 접근성, 테스트, 성능, 에러 처리는 사람이 의식적으로 검토해야 한다.

기능 요구사항

사용자는 상품 목록 화면에서 다음 행동을 할 수 있어야 한다.

  • 검색어를 입력해 상품명을 검색한다.
  • 카테고리와 가격 범위로 필터링한다.
  • 최신순, 인기순, 가격순으로 정렬한다.
  • 페이지를 이동한다.
  • 상품을 즐겨찾기에 추가하거나 제거한다.
  • 검색 조건을 URL로 공유할 수 있다.
  • 새로고침하거나 뒤로 가기를 해도 검색 조건이 유지된다.
  • 결과가 없을 때 빈 상태를 본다.
  • API 요청이 실패하면 에러 상태와 재시도 버튼을 본다.

이 요구사항은 평범해 보이지만 설계할 것이 많다. 검색어는 입력 즉시 API를 호출해야 하는가, 제출 버튼을 눌렀을 때만 호출해야 하는가. 필터 상태는 컴포넌트 안에 둘 것인가, URL에 둘 것인가. 즐겨찾기 상태는 서버 상태인가 클라이언트 상태인가. 페이지 이동 중 이전 결과를 유지할 것인가, 로딩 스피너로 비울 것인가. 이 질문들이 프론트엔드 설계의 실제 난이도다.

상태 분류

먼저 상태를 분류한다.

URL 상태:

  • 검색어
  • 카테고리
  • 가격 범위
  • 정렬 기준
  • 페이지 번호

서버 상태:

  • 상품 목록
  • 총 개수
  • 현재 페이지의 응답 상태
  • 즐겨찾기 반영 결과

지역 UI 상태:

  • 필터 패널 열림 여부
  • 검색 입력 중 임시 값
  • 즐겨찾기 버튼의 optimistic pending 상태

파생 상태:

  • 적용된 필터 개수
  • 빈 상태 표시 여부
  • 다음 페이지 가능 여부

이 분류를 먼저 해두면 AI에게도 더 좋은 요청을 할 수 있다. “필터 UI를 만들어줘“보다 “검색 조건은 URL 상태로 관리하고, 서버 데이터는 query cache로 관리하며, 필터 패널 열림 여부만 지역 상태로 둬“라고 말하는 편이 훨씬 낫다.

API 계약

예제 API 계약은 다음과 같이 둘 수 있다.

type ProductSort = "latest" | "popular" | "price_asc" | "price_desc";

type ProductSearchParams = {
  q?: string;
  category?: string;
  minPrice?: number;
  maxPrice?: number;
  sort: ProductSort;
  page: number;
};

type Product = {
  id: string;
  name: string;
  category: string;
  price: number;
  imageUrl: string;
  isFavorite: boolean;
};

type ProductSearchResult = {
  items: Product[];
  totalCount: number;
  page: number;
  pageSize: number;
};

이 타입은 단순한 코드가 아니라 계약이다. AI가 컴포넌트를 만들 때 이 계약을 기준으로 삼게 해야 한다. 특히 API 응답과 UI 모델이 항상 같은 것은 아니라는 점을 기억해야 한다. 서버가 내려주는 값이 화면에 바로 쓰기 불편하다면 adapter를 두고 변환한다.

AI에게 줄 작업 헌장

이 기능을 AI와 함께 만든다면 첫 요청은 다음과 같이 작성할 수 있다.

목표:
상품 목록 화면에 검색, 필터, 정렬, 페이지네이션을 추가한다.

상태 규칙:
검색어, 카테고리, 가격 범위, 정렬, 페이지는 URL query parameter로 관리한다.
서버 응답은 TanStack Query로 관리한다.
필터 패널 열림 여부만 지역 상태로 둔다.

기존 패턴:
기존 `src/features/products` 폴더의 컴포넌트 구조를 따른다.
공용 버튼, 입력, select 컴포넌트를 새로 만들지 말고 재사용한다.

완료 조건:
새로고침 후 검색 조건이 유지된다.
뒤로 가기로 이전 검색 조건이 복원된다.
빈 결과와 API 실패 상태가 표시된다.
주요 흐름 테스트가 추가된다.

제약:
새 상태 관리 라이브러리를 추가하지 않는다.
API 계약은 변경하지 않는다.

이 요청은 AI에게 단순히 코드를 만들라고 하지 않는다. 설계의 중요한 결정을 사람이 먼저 잡아준다. AI는 그 안에서 구현을 돕는다.

예상되는 AI의 실수

이 기능에서 AI가 자주 만들 수 있는 실수는 다음과 같다.

  • 검색 조건을 URL과 컴포넌트 state에 동시에 저장한다.
  • 페이지 번호를 필터 변경 후 1로 초기화하지 않는다.
  • 빈 문자열을 query parameter에 남긴다.
  • API 실패와 빈 결과를 같은 화면으로 처리한다.
  • 오래된 요청 응답이 최신 결과를 덮어쓰게 둔다.
  • 즐겨찾기 optimistic update 실패를 되돌리지 않는다.
  • 버튼에 접근 가능한 이름을 제공하지 않는다.
  • 이미지 크기를 지정하지 않아 레이아웃 이동을 만든다.
  • 테스트가 성공 케이스만 확인한다.

이 목록은 AI를 불신하기 위한 것이 아니다. 리뷰 기준을 만들기 위한 것이다. 사람이 예상 실패 모드를 알고 있으면 AI에게 더 구체적인 보완을 요청할 수 있다.

구현 순서

실제 구현은 다음 순서로 나누는 것이 좋다.

  1. URL query parameter 파싱과 직렬화 함수를 만든다.
  2. ProductSearchParams 타입과 기본값을 정의한다.
  3. 상품 검색 query 함수를 만든다.
  4. 목록 화면에서 URL 상태를 읽어 query를 호출한다.
  5. 필터 패널과 정렬 컨트롤을 연결한다.
  6. 빈 상태, 로딩 상태, 에러 상태를 추가한다.
  7. 페이지네이션을 추가한다.
  8. 즐겨찾기 mutation과 optimistic update를 추가한다.
  9. 테스트를 추가한다.
  10. 브라우저에서 새로고침, 뒤로 가기, 느린 네트워크를 확인한다.

AI에게도 이 순서대로 요청하는 편이 좋다. 한 번에 전체를 만들게 하면 상태와 UI가 섞이기 쉽다. 작은 단계로 나누면 각 단계의 diff를 사람이 이해할 수 있다.

검증 체크리스트

기능이 완성되었다고 보기 전에 다음을 확인한다.

  • 검색 조건이 URL에 반영되는가?
  • 새로고침 후 같은 결과가 보이는가?
  • 뒤로 가기로 이전 조건이 복원되는가?
  • 필터 변경 시 페이지가 1로 돌아가는가?
  • 빈 결과와 API 실패가 구분되는가?
  • 로딩 중 중복 클릭이나 중복 요청이 문제가 되지 않는가?
  • 키보드로 필터와 정렬을 조작할 수 있는가?
  • 즐겨찾기 실패 시 UI가 되돌아오는가?
  • 이미지 영역에 안정적인 크기가 있는가?
  • 테스트가 실패 모드를 포함하는가?

이 체크리스트는 기능 하나를 위한 것이지만, 다른 기능에도 그대로 응용할 수 있다. AI 시대의 개발자는 코드를 빨리 만드는 것보다 검증 가능한 기능 단위를 만드는 데 익숙해야 한다.

사례 연구의 핵심 교훈

이 사례에서 가장 중요한 교훈은 상태의 위치를 먼저 정해야 한다는 것이다. 상태의 위치가 정리되면 컴포넌트 구조, API 호출, 테스트, 접근성 검증이 훨씬 단순해진다. 반대로 상태가 뒤섞이면 AI가 만든 코드는 점점 더 복잡해지고, 사람도 어디서부터 고쳐야 할지 알기 어려워진다.

AI는 구현 속도를 높여준다. 하지만 설계의 순서를 대신 정해주지는 않는다. 좋은 프론트엔드 개발자는 AI에게 코드를 쓰게 하기 전에, 코드가 들어갈 구조를 만든다.

17. 프론트엔드 개발자의 학습 로드맵

AI 시대의 학습 로드맵은 “더 많은 도구를 따라잡기“가 아니다. 오히려 도구가 바뀌어도 남는 층을 먼저 세우고, 그 위에 프레임워크와 AI 워크플로를 얹어야 한다. 프론트엔드 생태계는 빠르게 바뀌지만, 브라우저와 네트워크와 사용자 경험의 기본 문제는 오래 남는다.

AI는 학습의 방식도 바꾼다. 모르는 것을 검색하고 예제를 찾는 속도는 빨라졌다. 하지만 빠른 답은 깊은 이해를 보장하지 않는다. AI에게 물어서 해결한 문제를 다시 설명할 수 없다면, 그 지식은 아직 내 것이 아니다.

네 개의 학습 층

프론트엔드 개발자의 학습은 네 개의 층으로 볼 수 있다.

첫째, 실행 환경이다. HTML, CSS, JavaScript, TypeScript, 브라우저 렌더링, 이벤트 루프, HTTP, 캐시, 보안, 스토리지 같은 주제다. 이 층은 프레임워크보다 아래에 있다. 이 층이 약하면 프레임워크가 감춰주는 부분을 벗어났을 때 문제를 설명하지 못한다.

둘째, 제품 구현이다. 컴포넌트, 상태, 라우팅, 폼, 데이터 패칭, 접근성, 성능, 에러 처리, 디자인 시스템을 다룬다. 여기서는 사용자 흐름을 작동하는 화면으로 바꾸는 능력이 중요하다.

셋째, 시스템 설계다. feature 경계, 상태 모델, 타입 계약, 테스트 전략, 배포와 관찰, 팀의 코드 규칙을 다룬다. 미드레벨 이후의 성장에서 이 층이 중요해진다. 코드 한 조각보다 코드가 들어갈 구조를 보는 능력이다.

넷째, AI 협업이다. 요구사항 분해, 작업 헌장 작성, 코드 읽기, 자동생성 코드 리뷰, 검증 루프, 에이전트 운영, 프로젝트 지식베이스 관리가 포함된다. AI 협업은 별도의 얇은 기술이 아니라 앞의 세 층을 바탕으로 한다.

주니어 개발자의 로드맵

주니어 개발자는 AI를 쓰되 기본 구현 경험을 건너뛰면 안 된다. 직접 작은 기능을 만들어보고, 브라우저에서 깨뜨려보고, 테스트를 작성해보고, 버그를 고쳐보는 경험이 필요하다.

우선순위는 다음과 같다.

  • HTML과 CSS의 기본 의미를 익힌다.
  • JavaScript의 비동기와 이벤트 흐름을 이해한다.
  • TypeScript로 props, 상태, API 응답 타입을 표현한다.
  • React/Vue/Svelte 중 하나를 골라 컴포넌트와 상태를 익힌다.
  • form, list, modal, tab, search 같은 흔한 UI를 직접 구현한다.
  • Testing Library로 사용자 행동 중심 테스트를 작성한다.
  • AI가 만든 코드를 그대로 쓰기보다 한 줄씩 설명해본다.

주니어에게 좋은 AI 사용법은 “답을 복사하기“가 아니라 “내가 쓴 코드를 설명받고 반례를 찾기“다. 예를 들어 “이 코드에서 race condition이 생길 수 있는지 봐줘” 또는 “이 컴포넌트의 접근성 문제를 찾아줘“처럼 검토 도구로 쓰면 좋다.

미드레벨 개발자의 로드맵

미드레벨 개발자는 기능을 만드는 능력에서 코드베이스를 읽고 변경 범위를 통제하는 능력으로 성장해야 한다. 이제 중요한 것은 “작동하게 만들기“만이 아니라 “팀의 구조 안에서 안전하게 바꾸기“다.

우선순위는 다음과 같다.

  • 서버 상태, URL 상태, 지역 상태를 분리해서 설계한다.
  • API 응답과 UI 모델 사이의 adapter를 만든다.
  • 컴포넌트 합성과 커스텀 훅의 책임을 명확히 한다.
  • discriminated union으로 복잡한 상태를 표현한다.
  • 실패 케이스를 포함한 테스트를 작성한다.
  • PR에서 변경 이유와 검증 결과를 설명한다.
  • AI에게 탐색, 계획, 구현, 검증을 나눠 요청한다.

미드레벨에게 AI는 생산성 증폭기다. 하지만 코드베이스 지도를 갖고 있어야 AI를 제대로 쓸 수 있다. 관련 파일을 모르고, 상태 위치를 모르고, 테스트 전략을 모르면 AI의 제안을 평가할 수 없다.

시니어와 리드의 로드맵

시니어와 리드는 개인 생산성을 넘어 팀의 작업 방식을 설계해야 한다. AI 도구를 도입하면 팀의 코드 양, 리뷰 방식, 테스트 중요도, 문서화 방식이 바뀐다. 이 변화를 방치하면 속도는 늘고 품질은 떨어질 수 있다.

우선순위는 다음과 같다.

  • 팀의 AI 사용 원칙을 정한다.
  • 프로젝트 지식베이스와 작업 템플릿을 만든다.
  • 자동생성 코드 리뷰 기준을 문서화한다.
  • 테스트와 CI를 AI 시대의 안전장치로 강화한다.
  • 새 의존성, 보안, 개인정보 관련 승인 기준을 정한다.
  • 대규모 리팩터링을 에이전트와 함께 수행하는 절차를 만든다.
  • 주니어가 AI 답변을 비판적으로 읽도록 교육한다.

시니어의 역할은 AI를 금지하거나 무조건 장려하는 것이 아니다. 팀이 AI를 써도 코드 품질과 제품 책임이 흐려지지 않게 운영 체계를 만드는 것이다.

12주 학습 계획

이 책을 바탕으로 12주 학습 계획을 만든다면 다음과 같이 진행할 수 있다.

1-2주차: 브라우저와 JavaScript 런타임
DOM, CSSOM, 렌더링, 이벤트 루프, Promise, AbortController를 작은 예제로 실습한다.

3-4주차: 프레임워크와 상태
선택한 프레임워크에서 list, form, modal, search 화면을 만들고 지역 상태, URL 상태, 서버 상태를 분리한다.

5-6주차: 타입과 계약
API 응답 타입, UI view model, adapter, schema validation, discriminated union을 연습한다.

7-8주차: 코드 읽기와 리뷰
기존 오픈소스 또는 팀 코드의 진입점, 데이터 흐름, 상태 위치, 테스트를 지도로 정리한다. AI에게도 같은 작업을 시키고 결과를 비교한다.

9-10주차: 테스트와 품질
Testing Library, Playwright, 접근성 점검, 성능 기본 측정을 실습한다. 성공 케이스보다 실패 케이스를 의식적으로 추가한다.

11-12주차: AI 협업 루프
작업 헌장 작성, 에이전트 탐색/계획/구현/검증 분리, 자동생성 코드 리뷰, 결과 보고 형식을 반복한다.

학습의 기준은 설명 가능성이다

AI 시대에는 “해결했다“와 “이해했다“를 구분해야 한다. AI 도움으로 버그를 고쳤더라도 다음 질문에 답할 수 있어야 한다.

  • 원인은 무엇이었는가?
  • 왜 이 수정이 문제를 해결하는가?
  • 다른 해결책은 무엇이었고 왜 선택하지 않았는가?
  • 이 변경이 깨뜨릴 수 있는 것은 무엇인가?
  • 어떤 검증을 했고 무엇을 하지 못했는가?

이 질문에 답할 수 있다면 AI와 함께 배운 것이다. 답할 수 없다면 AI가 해결한 것을 잠시 빌린 것이다.

앞으로 남는 개발자

AI는 프론트엔드 개발자의 역할을 없애기보다 더 명확하게 만든다. 앞으로의 프론트엔드 개발자는 코드를 많이 쓰는 사람보다 좋은 질문을 만들고, 좋은 구조를 유지하고, 빠른 생성과 엄격한 검증 사이의 균형을 잡는 사람이 된다.

프레임워크는 바뀐다. 도구도 바뀐다. 하지만 브라우저는 계속 코드를 실행하고, 사용자는 계속 화면을 사용하며, 팀은 계속 변경 가능한 시스템을 유지해야 한다. 이 사실이 프론트엔드 개발자의 기본기를 오래 살아남게 만든다.

C. 용어집

이 부록은 책에서 반복해서 등장하는 용어를 정리한다.

AI 보조 개발: AI 도구를 사용해 코드 작성, 리팩터링, 테스트, 문서화, 디버깅을 돕는 개발 방식.

에이전트: 사용자의 목표를 받아 파일 탐색, 코드 수정, 명령 실행, 결과 검증 같은 여러 단계를 수행할 수 있는 AI 시스템.

자동생성 코드: AI, 코드 생성기, 스캐폴딩 도구 등이 만든 코드. 사람이 작성하지 않았더라도 사람이 이해하고 책임져야 한다.

작업 헌장: AI나 에이전트에게 일을 맡기기 전에 목표, 범위, 완료 조건, 제약, 검증 방법을 정리한 짧은 명세.

일정 산정: 작업을 완료하는 데 필요한 시간과 불확실성을 추정하는 활동. AI 시대에는 구현 시간뿐 아니라 탐색, 검증, 통합, 리뷰, 책임 가능한 품질 확보 시간을 함께 본다.

재산정: 탐색, 초안, 테스트, 리뷰처럼 불확실성이 줄어드는 지점에서 일정을 다시 계산하는 활동.

검증 가능한 제품 코드: 단순히 동작하는 초안이 아니라 요구사항, 타입, 테스트, 접근성, 성능, 보안, 기존 코드 패턴을 통과해 팀이 책임질 수 있는 코드.

렌더링 파이프라인: 브라우저가 HTML, CSS, JavaScript를 처리해 화면의 픽셀로 만드는 과정.

DOM: HTML 문서의 구조를 객체 모델로 표현한 것.

CSSOM: CSS 규칙을 브라우저가 해석해 만든 객체 모델.

레이아웃: 브라우저가 각 요소의 위치와 크기를 계산하는 단계.

페인트: 요소의 색, 텍스트, 테두리, 이미지 같은 시각적 표현을 그리는 단계.

컴포지팅: 여러 레이어를 합쳐 최종 화면을 만드는 단계.

이벤트 루프: 자바스크립트 실행, 태스크, 마이크로태스크, 렌더링 타이밍을 조율하는 런타임 메커니즘.

마이크로태스크: Promise callback처럼 현재 태스크가 끝난 뒤 렌더링 전 처리되는 비동기 작업 큐.

Race condition: 여러 비동기 작업의 완료 순서에 따라 결과가 달라지는 문제.

Stale closure: 콜백이 만들어진 시점의 오래된 값을 참조해 최신 상태와 어긋나는 문제.

AbortController: fetch 같은 비동기 작업을 취소하기 위해 사용하는 Web API.

서버 상태: 서버에서 가져온 데이터와 그 캐시, 동기화 상태. 클라이언트 UI 상태와 구분해서 다루는 것이 좋다.

지역 상태: 특정 컴포넌트나 좁은 UI 범위 안에서만 의미 있는 상태.

URL 상태: 검색 조건, 페이지, 탭처럼 주소에 남아 공유나 새로고침 후 복원이 필요한 상태.

공유 UI 상태: 서버 데이터는 아니지만 여러 컴포넌트가 함께 읽거나 쓰는 화면 상태.

파생 상태: 다른 상태로부터 계산할 수 있는 값. 불필요하게 저장하면 동기화 문제가 생긴다.

계약: API, 타입, 스키마, 컴포넌트 props처럼 코드 사이의 기대와 책임을 표현하는 경계.

Adapter: 외부 API 응답이나 SDK의 데이터를 내부 도메인 또는 UI 모델로 변환하는 경계.

Facade: 복잡한 외부 도구나 내부 구현을 단순한 인터페이스 뒤에 감추는 패턴.

State machine: 가능한 상태와 상태 전이를 명시적으로 표현하는 모델.

Discriminated union: 공통 식별 필드를 기준으로 여러 상태나 형태를 안전하게 구분하는 TypeScript 타입 패턴.

Exhaustive check: union type의 모든 경우를 처리했는지 컴파일 타임에 확인하는 기법.

런타임 검증: TypeScript 타입과 별개로 실제 실행 중 들어온 데이터가 기대한 형태인지 확인하는 과정.

회귀: 이전에 작동하던 기능이나 품질이 변경 이후 깨지는 현상.

Core Web Vitals: 사용자의 로딩, 상호작용, 시각적 안정성을 측정하기 위한 웹 성능 지표 집합.

LCP: 주요 콘텐츠가 화면에 표시되는 시점을 측정하는 성능 지표.

INP: 사용자 상호작용에 대한 응답성을 측정하는 성능 지표.

CLS: 화면 요소가 예상치 못하게 이동하는 정도를 측정하는 성능 지표.

XSS: 사용자 입력이나 외부 데이터를 안전하지 않게 렌더링해 스크립트 실행이 가능해지는 공격.

CSRF: 사용자가 인증된 상태를 악용해 원치 않는 요청을 보내게 만드는 공격.

Accessible name: 스크린 리더 등 보조 기술이 버튼, 링크, 입력 요소의 이름으로 인식하는 텍스트.
