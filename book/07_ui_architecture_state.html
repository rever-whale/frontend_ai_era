AI 시대의 프론트엔드 개발자

7. UI 상태와 데이터 흐름 설계

복잡한 프론트엔드 애플리케이션은 상태 관리에서 무너진다. 상태가 너무 멀리 있거나, 너무 가까이 있거나, 서로 다른 출처의 상태가 섞이면 작은 변경도 위험해진다. AI가 코드를 생성할 때도 상태의 경계를 정확히 설명하지 않으면 중복 상태와 임시 플래그가 빠르게 늘어난다.

상태 관리는 라이브러리 선택 문제가 아니다. Redux를 쓰면 해결되는 문제도 있고, TanStack Query를 쓰면 해결되는 문제도 있고, 단순한 useState 하나가 가장 좋은 답인 문제도 있다. 하지만 어떤 상태가 어떤 성격인지 구분하지 못하면 어떤 도구를 써도 복잡도는 줄지 않는다.

상태를 분류하는 이유

상태를 분류하는 이유는 저장 위치를 결정하기 위해서다. 모든 값을 전역 store에 넣으면 접근은 쉬워지지만 변경 이유를 추적하기 어려워진다. 모든 값을 지역 state로 두면 컴포넌트는 독립적으로 보이지만 화면 간 동기화가 깨진다. 서버에서 온 데이터를 클라이언트 state로 복사하면 캐시, 재검증, stale data 문제가 생긴다.

AI가 만든 UI 코드에서 흔한 문제는 “필요해 보이는 값을 일단 state로 만든다“는 것이다. 예를 들어 필터된 목록, 선택된 개수, 다음 페이지 가능 여부처럼 기존 데이터로 계산할 수 있는 값을 별도 state로 저장한다. 처음에는 편해 보이지만, 원본 상태가 바뀔 때 파생 상태도 함께 갱신해야 하므로 버그가 생긴다.

상태를 분류하면 이런 질문을 던질 수 있다.

  • 이 값은 사용자가 직접 바꾸는가?
  • 서버에서 온 값인가?
  • URL에 남아야 하는가?
  • 여러 화면에서 공유되어야 하는가?
  • 기존 값으로 계산할 수 있는가?
  • 시간이 지나면 자동으로 stale해지는가?

이 질문이 상태 설계의 출발점이다.

지역 상태

지역 상태는 특정 컴포넌트 안에서만 의미 있는 값이다. 예를 들어 dropdown이 열렸는지, 탭 내부의 임시 선택값, hover된 항목, 모달 안의 아직 제출하지 않은 입력값이 여기에 해당한다.

지역 상태의 장점은 단순함이다. 가까운 곳에 있으니 이해하기 쉽고, 컴포넌트가 사라지면 상태도 함께 사라진다. 문제는 지역 상태가 여러 컴포넌트에서 필요해질 때다. 이때 무작정 부모로 끌어올리거나 전역 store에 넣기 전에, 그 상태가 정말 공유되어야 하는지 확인해야 한다.

예를 들어 상품 카드의 hover 상태는 전역 상태가 될 이유가 없다. 반면 사이드바의 열림 여부가 여러 레이아웃 컴포넌트와 연결되어 있고 사용자 설정으로 저장되어야 한다면 더 넓은 범위의 상태가 될 수 있다.

AI에게 지역 상태가 충분한 작업을 맡길 때는 “이 상태는 이 컴포넌트 밖으로 노출하지 말아줘“라고 명시하는 것도 좋다. 작은 상태를 불필요하게 전역화하면 나중에 변경 비용이 커진다.

공유 UI 상태

공유 UI 상태는 서버 데이터는 아니지만 여러 컴포넌트가 함께 봐야 하는 화면 상태다. 예를 들어 현재 선택된 organization, 열려 있는 command palette, 전역 toast, layout density, theme, side panel 상태가 있다.

공유 UI 상태는 Context, store, route layout, framework의 전역 상태 도구로 관리할 수 있다. 중요한 것은 범위다. 전체 앱에서 필요한 상태인지, 특정 feature 안에서만 필요한 상태인지 구분해야 한다.

AI가 생성한 코드는 종종 편의를 위해 Context를 새로 만들거나 전역 store에 값을 추가한다. 하지만 전역 상태는 쉽게 늘어나고 어렵게 줄어든다. 공유 UI 상태를 만들기 전에 다음 질문을 한다.

  • 이 상태를 읽는 컴포넌트는 몇 개인가?
  • 이 상태를 변경하는 주체는 몇 개인가?
  • 이 상태가 앱을 벗어나 URL이나 저장소에 남아야 하는가?
  • feature 단위 store로 충분한가?
  • 전역화했을 때 테스트와 디버깅이 쉬워지는가?

공유 상태는 편의가 아니라 책임이다.

서버 상태

서버 상태는 서버에서 온 데이터와 그 데이터의 캐시, stale 여부, 재검증, mutation 결과를 포함한다. 상품 목록, 사용자 프로필, 알림 목록, 권한 정보, 결제 상태 같은 값은 서버 상태다.

서버 상태를 일반 클라이언트 상태처럼 다루면 문제가 생긴다. 서버 데이터는 시간이 지나면 낡는다. 여러 화면에서 같은 데이터를 사용할 수 있다. mutation 후에는 관련 데이터를 다시 가져오거나 캐시를 업데이트해야 한다. 네트워크 실패, 재시도, optimistic update도 필요할 수 있다.

그래서 TanStack Query, SWR 같은 서버 상태 도구가 유용하다. 이 도구들은 단순히 fetch를 감싸는 라이브러리가 아니다. 캐시 키, stale time, refetch, mutation, optimistic update, error state 같은 서버 상태의 문제를 모델링한다.

AI에게 서버 데이터를 다루는 코드를 요청할 때는 다음을 명시해야 한다.

  • query key에 어떤 값이 포함되어야 하는가?
  • 언제 stale하다고 볼 것인가?
  • mutation 후 어떤 query를 무효화할 것인가?
  • optimistic update를 할 것인가?
  • 실패 시 rollback이 필요한가?

서버 상태는 “가져온 뒤 화면에 뿌리는 값“이 아니라 동기화 대상이다.

URL 상태

URL 상태는 주소에 남아야 하는 상태다. 검색어, 필터, 정렬, 페이지 번호, 선택된 탭, 공유 가능한 view mode가 여기에 해당한다. URL 상태의 기준은 간단하다. 사용자가 새로고침하거나 링크를 공유했을 때 유지되어야 하는가?

검색 화면에서 필터를 컴포넌트 state로만 관리하면 새로고침 때 조건이 사라진다. 뒤로 가기도 기대대로 동작하지 않는다. 반대로 모든 UI 상태를 URL에 넣으면 주소가 불필요하게 복잡해진다.

URL 상태는 다음 원칙으로 다룬다.

  • 공유 가능한 검색 조건은 URL에 둔다.
  • 기본값은 query parameter에서 생략한다.
  • 빈 문자열과 undefined를 구분한다.
  • 필터가 바뀌면 페이지 번호를 초기화한다.
  • URL 파싱과 직렬화는 한 곳에 모은다.

AI가 필터 화면을 만들 때 가장 자주 놓치는 것이 URL 상태다. “필터는 새로고침과 뒤로 가기에서도 유지되어야 한다“는 요구를 명시하지 않으면 단순한 지역 state로 구현할 가능성이 높다.

파생 상태

파생 상태는 기존 값으로 계산할 수 있는 값이다. 예를 들어 장바구니 총액, 선택된 항목 개수, 필터 적용 여부, 다음 페이지 가능 여부, 에러 배너 표시 여부가 있다. 파생 상태는 가능하면 저장하지 않고 계산한다.

나쁜 예는 다음과 같다.

const [items, setItems] = useState<Product[]>([]);
const [filteredItems, setFilteredItems] = useState<Product[]>([]);
const [count, setCount] = useState(0);

filteredItems와 count는 items와 필터 조건으로 계산할 수 있다. 별도 state로 저장하면 items가 바뀔 때마다 동기화해야 한다. 동기화가 빠지는 순간 UI는 거짓말을 한다.

더 나은 방식은 다음과 같다.

const filteredItems = useMemo(
  () => items.filter((item) => matchesFilter(item, filter)),
  [items, filter],
);

const count = filteredItems.length;

물론 모든 계산에 useMemo가 필요한 것은 아니다. 중요한 것은 “이 값이 원본인가, 계산 결과인가“를 구분하는 것이다. AI가 state를 많이 만들수록 파생 상태가 섞였는지 확인해야 한다.

상태 위치를 결정하는 절차

새 기능을 만들 때 다음 절차를 사용할 수 있다.

  1. 화면에서 보이는 값을 모두 적는다.
  2. 각 값이 사용자 입력인지, 서버 데이터인지, URL 상태인지, 파생 상태인지 표시한다.
  3. 서버 데이터는 서버 상태 도구나 data loader에 연결한다.
  4. 공유 가능한 검색/탭/페이지 상태는 URL로 보낸다.
  5. 컴포넌트 안에서만 필요한 값은 지역 상태로 둔다.
  6. 기존 값으로 계산 가능한 것은 저장하지 않는다.
  7. 여러 컴포넌트가 읽고 쓰는 UI 상태만 제한적으로 공유한다.

이 절차는 AI에게도 그대로 줄 수 있다. “상태를 먼저 분류하고 구현 계획을 제안해줘“라고 요청하면, 코드를 만들기 전에 설계 대화를 할 수 있다.

데이터 흐름은 한 방향으로 읽혀야 한다

복잡한 UI에서 데이터 흐름은 가능한 한 한 방향으로 읽혀야 한다. 사용자의 행동이 상태를 바꾸고, 상태가 화면을 바꾸고, 필요한 경우 서버와 동기화한다. 이 흐름이 뒤엉키면 디버깅이 어려워진다.

예를 들어 자식 컴포넌트가 부모의 상태를 암묵적으로 바꾸고, 동시에 URL도 바꾸고, 서버 요청도 직접 실행한다면 변경 이유를 추적하기 어렵다. 반대로 이벤트는 위로 올리고, 상태 변경은 정해진 위치에서 일어나고, 서버 동기화는 query/mutation 계층에서 처리되면 코드가 읽힌다.

AI가 만든 컴포넌트가 너무 많은 책임을 갖고 있다면 다음처럼 나눌 수 있다.

  • UI 컴포넌트: props를 받아 표시하고 사용자 이벤트를 알린다.
  • 상태 훅: URL 상태와 지역 상태를 관리한다.
  • 데이터 훅: 서버 query와 mutation을 관리한다.
  • adapter: API 응답을 UI 모델로 변환한다.

이 분리는 재사용보다 이해 가능성을 위해 한다.

AI 생성 코드 리뷰 질문

상태와 데이터 흐름 관점에서 AI가 만든 코드를 리뷰할 때는 다음 질문을 던진다.

  • 서버 상태를 클라이언트 state로 불필요하게 복사하지 않았는가?
  • URL에 남아야 할 검색 조건을 지역 state로만 관리하지 않는가?
  • 기존 값으로 계산 가능한 파생 상태를 별도 state로 저장하지 않는가?
  • 전역 상태가 정말 전역이어야 하는가?
  • query key가 상태의 모든 입력을 포함하는가?
  • mutation 후 캐시 무효화나 업데이트가 명확한가?
  • 상태 변경이 한 방향으로 읽히는가?
  • 컴포넌트 하나가 UI, 데이터 요청, URL 동기화, 비즈니스 규칙을 모두 맡고 있지 않은가?

상태 설계는 프론트엔드의 중심이다. AI가 코드를 빠르게 만들수록 상태의 위치를 먼저 결정하는 습관이 더 중요해진다.
