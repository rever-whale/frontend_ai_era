AI 시대의 프론트엔드 개발자

2. 2020년대 중반의 프레임워크 지형도

프론트엔드 프레임워크를 배울 때 가장 흔한 질문은 “무엇이 가장 인기 있는가“다. 이 질문이 완전히 틀린 것은 아니다. 인기는 생태계, 채용 시장, 문서, 라이브러리, 커뮤니티 규모와 연결된다. 하지만 이 질문만으로는 오래 버티는 판단을 만들 수 없다. 더 좋은 질문은 “이 도구는 어떤 문제를 해결하려고 하는가“다.

프레임워크는 개발자가 브라우저 위에서 애플리케이션을 만들 때 반복해서 마주치는 문제에 대한 답이다. UI를 어떻게 조각낼 것인가, 상태를 어떻게 변경할 것인가, 서버에서 데이터를 언제 가져올 것인가, HTML을 어디서 만들 것인가, 사용자가 상호작용하기 전에 무엇을 미리 준비할 것인가, 팀이 같은 규칙으로 일하게 하려면 어떤 제약을 둘 것인가. 프레임워크의 차이는 결국 이 질문들에 대한 답의 차이다.

AI 시대에는 이 관점이 더 중요하다. AI는 특정 프레임워크의 문법을 빠르게 흉내 낼 수 있다. 하지만 어떤 렌더링 모델이 현재 제품에 맞는지, 어떤 상태 관리 방식이 변경을 견디는지, 어떤 아키텍처가 팀의 속도와 안정성을 동시에 지키는지는 여전히 사람이 판단해야 한다.

React와 Next.js: 생태계와 풀스택 경계

React는 오랫동안 프론트엔드 생태계의 중심에 있었다. React의 힘은 단순히 컴포넌트 문법이 아니라 거대한 생태계, 채용 시장, 디자인 시스템과 상태 관리 도구, 테스트 도구, 문서와 사례의 축적에서 나온다.

Next.js는 React를 기반으로 라우팅, 서버 렌더링, 정적 생성, API route, 서버 컴포넌트, streaming 같은 기능을 통합해 애플리케이션 프레임워크로 확장했다. 이 흐름은 프론트엔드와 백엔드의 경계를 다시 생각하게 만든다. 화면은 브라우저에서만 만들어지는 것이 아니라 서버에서도 만들어지고, 데이터 패칭은 클라이언트 effect 안에서만 일어나지 않는다.

React와 Next.js를 배울 때 중요한 질문은 다음과 같다.

  • 이 컴포넌트는 서버에서 렌더링되어도 되는가?
  • 클라이언트 상태가 필요한 부분은 어디인가?
  • 데이터 패칭은 route 단위인가, 컴포넌트 단위인가?
  • hydration 비용은 어느 정도인가?
  • 서버와 클라이언트 경계에서 어떤 값이 직렬화되는가?

AI에게 Next.js 코드를 생성하게 할 때는 이 경계를 명확히 설명해야 한다. 그렇지 않으면 불필요하게 클라이언트 컴포넌트를 만들거나, 서버에서만 가능한 작업을 브라우저 코드에 넣거나, 캐싱 정책을 놓칠 수 있다.

Vue와 Nuxt: 접근성과 균형

Vue는 템플릿 기반 문법과 반응성 시스템을 통해 많은 개발자에게 접근성이 좋은 프레임워크로 자리 잡았다. Vue의 장점은 HTML과 가까운 템플릿, 명확한 single-file component 구조, 점진적 도입 가능성이다. Nuxt는 Vue 생태계에서 서버 렌더링과 파일 기반 라우팅, 데이터 패칭, 모듈 시스템을 제공한다.

Vue와 Nuxt는 팀이 명확한 구조를 선호하면서도 지나치게 무거운 규율을 원하지 않을 때 좋은 선택지가 될 수 있다. 특히 템플릿, 스타일, 로직이 한 파일 안에서 분명하게 나뉘는 구조는 AI가 코드를 읽고 수정할 때도 장점이 될 수 있다.

다만 어떤 도구든 마찬가지로, 접근성이 좋다는 것이 설계가 자동으로 좋아진다는 뜻은 아니다. Vue에서도 서버 상태와 UI 상태를 섞을 수 있고, 컴포넌트를 지나치게 크게 만들 수 있으며, composable이 무분별하게 늘어날 수 있다.

Svelte와 SvelteKit: 컴파일러 중심 사고

Svelte는 런타임보다 컴파일러에 더 많은 일을 맡기는 접근으로 주목받았다. 개발자는 비교적 간결한 문법으로 컴포넌트를 작성하고, Svelte는 이를 빌드 시점에 효율적인 JavaScript로 변환한다. SvelteKit은 여기에 라우팅, 서버 렌더링, 데이터 로딩, form action 같은 애플리케이션 기능을 제공한다.

Svelte 계열을 이해할 때 중요한 것은 “프레임워크가 브라우저에서 얼마나 많은 일을 하는가“라는 질문이다. 모든 프레임워크는 추상화를 제공하지만, 그 추상화가 런타임에서 작동하는지 빌드 시점에 정리되는지에 따라 성능과 디버깅 경험이 달라진다.

AI 시대에는 간결한 문법이 초안 생성에 유리할 수 있다. 그러나 컴파일러가 제공하는 반응성 규칙을 제대로 이해하지 못하면 생성된 코드가 미묘하게 잘못될 수 있다. 문법이 짧을수록 규칙을 더 정확히 알아야 한다.

Astro와 아일랜드 아키텍처

Astro는 콘텐츠 중심 사이트와 정적 사이트, 문서, 블로그, 마케팅 페이지에서 강점을 보인다. 핵심 아이디어는 모든 페이지를 거대한 클라이언트 애플리케이션으로 만들지 않고, 필요한 상호작용 영역만 선택적으로 hydration하는 것이다. 이를 흔히 아일랜드 아키텍처라고 부른다.

Astro가 던지는 질문은 중요하다. “이 페이지 전체가 정말 JavaScript 애플리케이션이어야 하는가?” 많은 웹 페이지는 대부분 정적 콘텐츠이고, 일부만 상호작용이 필요하다. 이런 경우 불필요한 JavaScript를 줄이는 것이 성능과 사용자 경험에 직접적인 이득을 준다.

프론트엔드 개발자는 SPA를 기본값으로 생각하기 쉽다. 하지만 AI 시대에는 더 많은 코드가 더 빨리 만들어지기 때문에, 오히려 “이 코드는 필요 없는 것 아닌가“라는 질문이 중요해진다.

Angular, Solid, 그리고 다른 선택지

Angular는 명확한 프레임워크 규율과 큰 조직에서의 운영성을 강점으로 한다. 라우팅, 폼, HTTP, 의존성 주입, 빌드 도구 등 많은 요소가 프레임워크 안에 들어 있다. 이는 자유도를 줄이는 대신 팀 단위 일관성을 높인다.

Solid는 세밀한 반응성과 효율적인 업데이트 모델을 강조한다. React와 비슷한 JSX 형태를 사용하면서도 내부 모델은 다르다. 이런 도구는 프론트엔드 생태계가 단일한 방향으로만 가지 않는다는 사실을 보여준다.

이 밖에도 Qwik, Remix, htmx 같은 도구들은 각자 다른 문제의식을 갖고 있다. 모든 도구를 깊이 배울 필요는 없지만, 이들이 던지는 질문은 배울 가치가 있다. hydration을 줄일 수 있는가, 서버와 클라이언트의 협업을 단순화할 수 있는가, HTML의 힘을 더 활용할 수 있는가, 복잡한 SPA가 정말 필요한가.

프레임워크를 비교하는 여섯 가지 축

프레임워크 선택은 취향만으로 할 수 없다. 다음 여섯 가지 축으로 비교하면 더 차분한 판단을 할 수 있다.

첫째, 렌더링 모델이다. CSR, SSR, SSG, ISR, streaming, partial hydration 중 어떤 방식을 지원하고, 기본값은 무엇인가.

둘째, 데이터 패칭 모델이다. 데이터를 route에서 가져오는가, 컴포넌트에서 가져오는가, 서버에서 가져오는가, 클라이언트에서 가져오는가. 캐시와 재검증은 어떻게 표현하는가.

셋째, 상태 관리 모델이다. 지역 상태, 서버 상태, URL 상태, 전역 상태를 어떻게 나누는가. 프레임워크가 제공하는 기본 도구가 있는가.

넷째, 타입 친화성이다. TypeScript와 자연스럽게 결합되는가. 타입 추론이 좋은가. API와 route, form의 계약을 표현하기 쉬운가.

다섯째, 팀 운영성이다. 문서, 채용, 생태계, 업그레이드 비용, 린트와 테스트 도구, 디자인 시스템과의 결합을 고려해야 한다.

여섯째, AI 친화성이다. 코드 패턴이 일관적인가. 타입과 테스트가 충분한가. 파일 구조가 예측 가능한가. AI가 변경할 때 실수해도 검증 루프가 잡아낼 수 있는가.

유행보다 중요한 것

프레임워크는 계속 바뀐다. 하지만 프레임워크가 해결하려는 문제는 오래 남는다. 렌더링을 어디서 할 것인가, 데이터를 언제 가져올 것인가, 상태를 어디에 둘 것인가, 사용자가 상호작용하기 전까지 어떤 비용을 지불할 것인가, 팀이 같은 방식으로 코드를 읽게 하려면 어떤 규칙이 필요한가.

AI 시대의 프론트엔드 개발자는 프레임워크 이름보다 이 질문들을 익혀야 한다. 그래야 React를 쓰든 Vue를 쓰든 Svelte를 쓰든, 새로운 도구가 나오든, 도구의 표면 아래에 있는 설계 선택을 읽을 수 있다.
