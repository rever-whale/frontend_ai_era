AI 시대의 프론트엔드 개발자

15. AI 시대의 일정 산정

프론트엔드 개발에서 일정 산정은 늘 어려웠다. 화면은 사용자가 직접 만지는 제품의 끝단에 있고, 백엔드 API, 디자인, 기획, 데이터, 인증, 브라우저, 배포 일정의 영향을 모두 받는다. 그래서 프론트엔드 작업은 종종 최종 병목이 되었다. 앞단의 의사결정이 늦어져도 마지막에는 화면이 완성되어야 하고, 일정은 프론트엔드에서 압축되기 쉬웠다.

이런 환경에서는 정확한 일정 산정이 큰 의미를 가졌다. 언제 끝나는지 알아야 출시일을 잡을 수 있고, 디자인과 API와 QA와 배포를 맞출 수 있었다. 일정 산정은 단순한 관리 행위가 아니라, 프로젝트의 마지막 병목을 다루는 생존 기술이었다.

AI 시대에는 이 의미가 사라지는 것이 아니라 바뀐다. 코드 초안 작성 속도는 빨라질 수 있다. 하지만 제품이 완성되는 속도 전체가 같은 비율로 빨라진다고 보기는 어렵다. 일정 산정의 중심은 “코드를 쓰는 데 며칠 걸리는가“에서 “불확실성을 줄이고, 생성된 결과를 검증하고, 시스템 안에 안전하게 통합하는 데 얼마나 걸리는가“로 이동한다.

예전의 병목: 끝단 구현

전통적인 프론트엔드 일정에서 큰 병목은 구현량이었다. 디자인이 확정되고 API가 준비되면, 프론트엔드 개발자는 화면을 만들고 상태를 연결하고 예외를 처리하고 테스트하고 QA 이슈를 고쳤다. 특히 마감이 가까울수록 프론트엔드가 모든 변경의 접점이 되었다.

이때 일정 산정은 주로 다음을 묻는 일이었다.

  • 화면이 몇 개인가?
  • 컴포넌트가 얼마나 복잡한가?
  • API가 준비되어 있는가?
  • 상태와 폼이 얼마나 많은가?
  • QA와 브라우저 대응이 얼마나 걸릴 것인가?
  • 디자인 변경이 얼마나 남아 있는가?

물론 이 질문들은 여전히 중요하다. 하지만 AI가 컴포넌트 초안, 테스트 초안, 타입 정의, 반복 스타일링을 빠르게 만들어주기 시작하면서 “순수 작성 시간“의 비중은 줄어들 수 있다. 문제는 줄어든 시간이 프로젝트 전체 일정에서 그대로 절약되느냐는 것이다.

새 병목: 판단과 통합

AI 시대의 새 병목은 판단과 통합이다. AI가 만든 코드는 빠르게 나온다. 하지만 그 코드가 요구사항을 정확히 이해했는지, 기존 코드베이스의 패턴을 지키는지, 상태를 올바른 위치에 두었는지, 접근성과 성능과 보안을 해치지 않는지 확인해야 한다.

예를 들어 검색 필터 UI를 AI가 하루가 아니라 한 시간 안에 만들 수 있다고 하자. 그래도 다음 작업은 남아 있다.

  • 검색 조건이 URL에 남아야 하는지 판단한다.
  • 서버 상태와 지역 상태를 분리한다.
  • API 실패, 빈 결과, 권한 없음, 느린 네트워크를 확인한다.
  • 디자인 시스템 컴포넌트와 맞춘다.
  • 접근성 label과 키보드 흐름을 확인한다.
  • 테스트를 추가하고 실제 브라우저에서 검증한다.
  • 기존 코드와 충돌하는 패턴을 정리한다.
  • 리뷰어가 이해할 수 있게 변경 이유를 설명한다.

AI가 초안을 빠르게 만든다고 해서 이 작업들이 사라지지는 않는다. 오히려 생성 속도가 빨라질수록 검증과 통합을 체계화하지 않으면 품질 문제가 더 빠르게 쌓인다.

일정 산정의 단위가 바뀐다

AI 이전의 일정 산정은 구현 항목을 기준으로 하는 경우가 많았다.

상품 목록 화면: 2일
필터 UI: 1일
상세 모달: 1일
테스트와 QA: 1일

AI 시대에는 이보다 작업의 성격을 나누는 산정이 더 유용하다.

요구사항 정리와 불확실성 제거: 0.5일
기존 코드 탐색과 변경 계획: 0.5일
AI 초안 생성과 1차 구현: 0.5일
상태/타입/계약 정리: 1일
검증과 실패 케이스 보강: 1일
리뷰 반영과 통합: 0.5일

여기서 중요한 변화는 구현 시간이 전체의 전부가 아니라는 점이다. AI는 “초안 생성“을 줄일 수 있지만, 요구사항 정리, 기존 코드 탐색, 검증, 리뷰, 통합은 여전히 별도 항목으로 산정해야 한다.

일정은 약속이 아니라 불확실성 모델이다

일정 산정을 약속으로만 보면 개발자는 방어적으로 변한다. “언제 끝나요?“라는 질문에 정확한 날짜를 말해야 한다고 느끼면, 불확실성을 숨기거나 과도한 버퍼를 넣게 된다. 하지만 좋은 일정 산정은 약속 이전에 불확실성 모델이다.

프론트엔드 작업의 불확실성은 보통 다음에서 나온다.

  • 요구사항이 아직 모호하다.
  • 디자인이 확정되지 않았다.
  • API 계약이 바뀔 수 있다.
  • 기존 코드 구조를 아직 모른다.
  • 브라우저나 접근성 이슈가 숨어 있다.
  • AI가 만든 초안의 품질을 아직 모른다.
  • 테스트 환경이나 mock 데이터가 부족하다.
  • 리뷰와 QA에서 어떤 문제가 나올지 모른다.

따라서 일정 산정은 “3일입니다“보다 “현재 불확실성 기준으로 3-5일이며, API 계약과 빈 상태 정책이 확정되면 범위를 줄일 수 있습니다“에 가까워져야 한다. AI 시대에는 특히 그렇다. 초안은 빠르게 만들 수 있지만, 초안 이후의 위험은 작업을 열어보기 전까지 알기 어렵다.

AI가 줄이는 시간과 줄이지 못하는 시간

AI가 줄일 수 있는 시간은 대체로 다음과 같다.

  • 반복적인 코드 작성
  • boilerplate 생성
  • 비슷한 테스트 케이스 초안
  • 타입 정의 초안
  • 문서와 체크리스트 초안
  • 기존 코드 요약
  • 리팩터링 후보 제안

AI가 줄이기 어려운 시간은 다음과 같다.

  • 요구사항의 우선순위 결정
  • 사용자 경험의 판단
  • 팀의 기존 구조와 히스토리 이해
  • 보안과 개인정보 판단
  • 실패 모드의 실제 검증
  • 브라우저와 기기별 확인
  • 조직 간 의사결정
  • 리뷰와 합의

일정 산정은 이 둘을 구분해야 한다. “AI를 쓰면 절반으로 줄어들까요?“라는 질문에는 “어떤 시간이 줄어드는가“로 답해야 한다. 작성 시간이 50% 줄어도 검증과 통합 시간이 그대로라면 전체 일정은 50% 줄지 않는다.

산정 전에 AI로 할 수 있는 일

AI는 일정 산정 자체에도 도움을 줄 수 있다. 다만 AI에게 “이거 며칠 걸려?“라고 묻는 것은 좋지 않다. AI는 팀의 코드베이스, 개발자의 숙련도, 리뷰 속도, 조직의 의사결정, QA 환경을 충분히 알지 못한다.

더 좋은 요청은 산정에 필요한 정보를 정리하게 하는 것이다.

아직 구현하지 말고 이 기능의 일정 산정을 돕기 위해 관련 코드를 탐색해줘.
변경이 필요한 파일, 상태/타입/API 위험, 테스트 필요 항목,
확정되지 않은 요구사항, 예상 실패 모드를 표로 정리해줘.
확실하지 않은 것은 추측이라고 표시해줘.

이 요청은 AI에게 날짜를 맞히게 하지 않는다. 대신 산정에 필요한 재료를 모으게 한다. 일정 산정의 품질은 답변의 숫자가 아니라, 드러난 불확실성의 품질에서 나온다.

새로운 일정 산정 템플릿

AI 시대의 프론트엔드 일정 산정은 다음 템플릿으로 정리할 수 있다.

작업 목표:

사용자에게 보이는 완료 상태:

AI로 단축 가능한 작업:
- 초안 생성:
- 반복 코드:
- 테스트 초안:
- 문서화:

사람이 판단해야 하는 작업:
- 요구사항:
- UX:
- 상태/아키텍처:
- 보안/개인정보:
- 리뷰/합의:

불확실성:
- 요구사항:
- 디자인:
- API:
- 기존 코드:
- 테스트/QA:

검증 범위:
- 타입/린트:
- 단위/컴포넌트 테스트:
- E2E:
- 접근성:
- 성능:
- 브라우저 수동 확인:

예상 일정:
- 탐색:
- 초안:
- 통합:
- 검증:
- 리뷰 반영:

범위 축소 옵션:

일정을 다시 산정할 시점:

이 템플릿의 핵심은 “AI가 빠르게 해줄 부분“과 “사람이 책임져야 할 부분“을 분리하는 것이다.

일정 재산정의 주기가 짧아진다

AI 시대에는 첫 산정이 더 정확해지는 것이 아니라, 재산정 주기가 짧아질 수 있다. 에이전트가 빠르게 탐색하고 초안을 만들면, 우리는 더 이른 시점에 실제 복잡도를 볼 수 있다. 예전에는 이틀을 구현하고 나서야 알던 문제를, 이제는 몇 시간 안에 발견할 수 있다.

따라서 좋은 운영 방식은 초기 산정 하나에 집착하지 않고, 산정 체크포인트를 명시하는 것이다.

  • 탐색 후 재산정
  • AI 초안 후 재산정
  • 테스트 실패 모드 확인 후 재산정
  • 첫 리뷰 후 재산정
  • QA 첫 라운드 후 재산정

이렇게 하면 일정 산정은 예측 게임이 아니라 학습 루프가 된다. AI는 이 학습 루프의 속도를 높인다.

프론트엔드 병목은 사라지는가

AI가 코드를 빠르게 만들어도 프론트엔드 병목이 완전히 사라지지는 않는다. 병목의 위치가 바뀐다.

예전의 병목:

  • 화면 구현량
  • 스타일링 반복
  • 상태 연결
  • 테스트 작성 속도

새로운 병목:

  • 요구사항과 디자인의 모호성
  • API와 UI 계약의 불일치
  • 자동생성 코드의 리뷰와 검증
  • 접근성, 성능, 보안의 품질 기준
  • 여러 팀 사이의 의사결정
  • 출시 전 통합과 QA

즉, 프론트엔드는 여전히 끝단이다. 다만 끝단에서 손으로 코드를 쓰는 시간이 줄고, 끝단에서 시스템 전체의 모순을 드러내고 정리하는 시간이 더 중요해진다.

팀과 대화하는 방식도 바뀐다

AI 시대의 일정 대화는 “AI 쓰면 빨리 되죠?“라는 기대와 자주 부딪힌다. 이때 개발자는 방어적으로 “그래도 오래 걸립니다“라고만 말하기보다, 시간이 어디에 쓰이는지 분해해서 보여줘야 한다.

예를 들어 이렇게 말할 수 있다.

AI로 화면 초안은 빠르게 만들 수 있습니다.
다만 이 기능은 검색 조건을 URL과 동기화해야 하고,
API 실패/빈 결과/권한 없음 상태를 분리해야 하며,
기존 서버 상태 캐시와 연결해야 합니다.
초안은 반나절 안에 만들 수 있지만,
검증 가능한 제품 코드로 통합하는 데 2-3일이 필요합니다.

이 설명은 AI를 부정하지 않는다. 동시에 AI가 줄이지 못하는 책임을 드러낸다. 일정 산정은 개발자의 방어 수단이 아니라, 제품 리스크를 함께 보는 언어가 되어야 한다.

일정 산정의 새로운 의미

AI 이전의 일정 산정은 종종 “끝단 구현을 언제 끝낼 수 있는가“에 가까웠다. AI 시대의 일정 산정은 “어떤 불확실성을 언제 줄이고, 어떤 수준의 검증을 거쳐, 언제 책임질 수 있는 코드로 만들 수 있는가“에 가까워진다.

그래서 일정 산정은 더 이상 단순한 날짜 맞히기가 아니다. 일정 산정은 작업을 이해하는 방식이고, 리스크를 드러내는 방식이며, AI가 만든 빠른 초안을 제품 코드로 바꾸는 데 필요한 책임의 언어다.

AI가 구현 속도를 높일수록 일정 산정은 사라지지 않는다. 오히려 더 정교해져야 한다. 빠르게 만들 수 있는 것과 책임질 수 있는 것 사이의 차이를 설명하는 일이 중요해지기 때문이다.
