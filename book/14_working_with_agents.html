AI 시대의 프론트엔드 개발자

14. 에이전트와 함께 일하는 법

AI 에이전트는 단순한 자동완성보다 강력하다. 파일을 읽고, 변경하고, 테스트를 실행하고, 결과를 요약할 수 있다. 어떤 에이전트는 브라우저를 열어 동작을 확인하고, 어떤 에이전트는 터미널 명령을 실행하며, 어떤 에이전트는 여러 파일에 걸친 리팩터링을 수행한다. 이 능력은 프론트엔드 개발자의 작업 방식을 크게 바꾼다.

하지만 강력한 도구일수록 작업 단위와 경계를 분명히 해야 한다. 에이전트는 넓은 범위를 빠르게 건드릴 수 있기 때문에, 좋은 지시와 좋은 검증이 없으면 잘못된 변경도 빠르게 확산된다. 에이전트와 잘 일한다는 것은 프롬프트를 멋지게 쓰는 일이 아니라, 개발 작업을 명확한 목표와 경계와 완료 조건으로 운영하는 일이다.

에이전트는 동료가 아니라 실행자다

에이전트를 사람 동료처럼 대화할 수는 있지만, 실제 운영에서는 실행자로 보는 편이 안전하다. 동료는 제품 맥락, 팀의 역사, 암묵적 제약, 정치적 판단, 운영 사고의 기억을 갖고 있다. 에이전트는 주어진 컨텍스트와 도구 안에서 추론하고 실행한다. 이 차이를 잊으면 위험하다.

에이전트에게 맡기기 좋은 일은 다음과 같다.

  • 관련 파일 탐색
  • 기존 패턴 요약
  • 작은 기능 초안 구현
  • 반복적인 테스트 추가
  • 타입 정리
  • 리팩터링 후보 제안
  • 변경 요약 작성

반대로 사람이 붙잡아야 하는 일은 다음과 같다.

  • 제품 요구사항의 우선순위
  • 사용자 경험의 최종 판단
  • 보안과 개인정보 결정
  • 새 아키텍처 경계 도입
  • 새 의존성 승인
  • 배포와 운영 리스크 판단

에이전트는 일을 실행할 수 있다. 하지만 어떤 일이 맞는지, 어떤 위험을 감수할지, 어떤 수준이면 충분한지는 사람이 결정해야 한다.

작업을 헌장으로 시작하기

에이전트 작업은 작은 작업 헌장으로 시작하는 것이 좋다. 작업 헌장은 길 필요가 없지만 다음을 포함해야 한다.

목표:
사용자 또는 제품 관점에서 무엇이 가능해져야 하는가?

범위:
수정해도 되는 화면, 컴포넌트, 파일, API는 무엇인가?

기존 패턴:
따라야 할 컴포넌트, 훅, 타입, 테스트 스타일은 무엇인가?

완료 조건:
어떤 동작, 테스트, 문서가 확인되어야 끝났다고 볼 수 있는가?

제약:
건드리지 말아야 할 파일, 추가하면 안 되는 의존성, 바꾸면 안 되는 API는 무엇인가?

보고:
마지막에 어떤 형식으로 변경과 검증 결과를 보고해야 하는가?

이 헌장은 에이전트를 위한 지시이면서 개발자 자신을 위한 정리다. 이 정도도 쓰기 어렵다면 작업이 아직 흐릿한 것이다.

탐색, 계획, 구현, 검증을 나눈다

큰 작업을 한 번에 시키면 에이전트는 추측을 많이 한다. 추측이 많을수록 변경 위험이 커진다. 그래서 탐색, 계획, 구현, 검증을 나누는 것이 좋다.

탐색 단계의 요청:

아직 파일을 수정하지 말고 관련 코드를 찾아줘.
진입점, 데이터 흐름, 상태 위치, 주요 타입, 테스트 파일을 정리하고,
확실하지 않은 부분은 추측이라고 표시해줘.

계획 단계의 요청:

찾은 코드 기준으로 변경 계획을 제안해줘.
변경할 파일, 변경 이유, 위험, 필요한 테스트를 나눠서 써줘.
아직 코드는 수정하지 마.

구현 단계의 요청:

계획의 1-2번만 먼저 구현해줘.
기존 패턴을 유지하고 새 의존성은 추가하지 마.

검증 단계의 요청:

변경된 코드의 타입, 테스트, 접근성, 상태 흐름을 점검해줘.
실행한 검증과 실행하지 못한 검증을 구분해서 보고해줘.

이렇게 나누면 개발자는 중간에 방향을 바로잡을 수 있다. 에이전트도 더 적은 추측으로 작업한다.

좋은 요청은 경계를 가진다

에이전트에게 가장 나쁜 요청은 “전체를 좋게 고쳐줘“다. 이 요청은 목표도 범위도 완료 조건도 없다. 에이전트는 넓게 바꿀 수밖에 없고, 개발자는 결과를 검토하기 어려워진다.

좋은 요청은 다음처럼 경계를 가진다.

상품 검색 화면에서 URL 상태와 폼 상태가 중복되는 문제를 정리해줘.
`useProductSearchParams`와 `ProductFilterPanel`만 우선 확인하고,
검색어/카테고리/정렬/페이지는 URL query parameter를 단일 출처로 삼아줘.
필터 변경 시 page는 1로 초기화되어야 한다.
기존 테스트 스타일에 맞춰 실패할 수 있는 테스트를 추가해줘.

이 요청에는 목표, 범위, 상태 규칙, 완료 조건이 들어 있다. 에이전트는 더 정확히 일할 수 있고, 사람은 결과를 더 쉽게 평가할 수 있다.

에이전트의 요약을 믿지 말고 diff를 본다

에이전트는 작업이 끝나면 요약을 제공한다. 이 요약은 유용하지만 증거는 아니다. 실제 증거는 diff, 테스트 결과, 브라우저 동작, 타입 검사 결과다.

요약을 읽을 때는 다음을 확인한다.

  • 실제로 변경된 파일과 요약이 일치하는가?
  • 의도하지 않은 파일이 수정되지 않았는가?
  • 새 의존성이나 설정 변경이 숨어 있지 않은가?
  • 검증했다고 말한 명령이 실제로 실행되었는가?
  • 실패하거나 생략한 검증이 보고되었는가?

에이전트가 틀렸다는 전제를 둘 필요는 없다. 다만 에이전트의 보고를 코드 리뷰의 출발점으로 삼아야지, 리뷰의 대체물로 삼으면 안 된다.

실패했을 때 더 작게 돌아간다

에이전트가 잘못된 방향으로 가면 많은 사람이 더 긴 설명을 추가한다. 때로는 필요하지만, 더 좋은 첫 대응은 작업을 더 작게 나누는 것이다.

예를 들어 에이전트가 검색 화면 전체를 잘못 고쳤다면 다음처럼 되돌아간다.

전체 구현은 멈추고, 먼저 URL query parameter 파싱 함수만 작성해줘.
입력과 출력 예시는 다음과 같아.
테스트도 이 함수에 대해서만 추가해줘.

작은 작업은 실패 비용이 낮고 검토가 쉽다. AI 시대의 개발자는 큰 요청을 잘 쓰는 사람보다, 큰 문제를 좋은 작은 작업으로 나누는 사람에 가깝다.

팀에서 에이전트를 운영하는 규칙

개인 생산성을 넘어 팀에서 에이전트를 쓰려면 운영 규칙이 필요하다.

  • 에이전트가 만든 코드도 동일한 리뷰 기준을 통과해야 한다.
  • 에이전트가 실행한 검증과 생략한 검증을 PR에 기록한다.
  • 새 의존성, 보안, 개인정보 관련 변경은 사람이 명시적으로 승인한다.
  • 대규모 리팩터링은 탐색/계획 단계의 산출물을 먼저 리뷰한다.
  • 에이전트에게 제공할 프로젝트 지식베이스를 관리한다.
  • 반복되는 작업은 템플릿과 체크리스트로 만든다.

팀의 목표는 “AI를 많이 쓰는 팀“이 아니라 “AI가 들어와도 코드 품질과 운영 신뢰성을 유지하는 팀“이다.

프로젝트 지식베이스 만들기

에이전트는 컨텍스트가 좋을수록 더 잘 일한다. 프로젝트 지식베이스에는 다음 내용을 담을 수 있다.

  • 폴더 구조와 feature 경계
  • API client 사용 규칙
  • 서버 상태와 URL 상태 관리 방식
  • 디자인 시스템 컴포넌트 사용 규칙
  • 테스트 작성 패턴
  • 접근성 최소 기준
  • 새 의존성 추가 기준
  • 배포와 검증 명령
  • 자주 발생한 실패 모드

이 지식베이스는 사람에게도 유용하다. 좋은 AI 컨텍스트는 좋은 온보딩 문서와 거의 같다. 결국 AI 친화적인 팀은 사람 친화적인 팀이다.

에이전트 작업의 완료 보고

에이전트 작업은 다음 형식으로 마무리하면 좋다.

변경:
- 무엇을 바꿨는가

이유:
- 왜 이렇게 바꿨는가

검증:
- 실행한 명령과 확인한 사용자 흐름

미검증:
- 확인하지 못한 환경, 브라우저, 실패 케이스

남은 위험:
- 리뷰어가 특히 봐야 할 지점

이 형식은 에이전트뿐 아니라 사람 개발자에게도 좋다. 코드 리뷰어가 빠르게 맥락을 이해하고, 나중에 문제가 생겼을 때 변경의 의도를 추적할 수 있다.
