AI 시대의 프론트엔드 개발자

4. 브라우저 렌더링 파이프라인

프론트엔드 개발자가 브라우저를 모르면 화면 버그를 설명할 수 없다. 프레임워크는 개발자에게 편리한 추상화를 제공하지만, 실제 화면은 여전히 브라우저가 만든다. React 컴포넌트든 Vue 템플릿이든 Svelte 컴포넌트든, 마지막에는 HTML, CSS, JavaScript가 브라우저 안에서 해석되고 픽셀로 바뀐다.

AI가 만든 코드도 이 규칙을 벗어나지 않는다. AI는 그럴듯한 컴포넌트를 빠르게 만들 수 있지만, 그 컴포넌트가 레이아웃을 얼마나 자주 다시 계산하게 하는지, 이미지를 로드하면서 화면을 얼마나 밀어내는지, 애니메이션이 메인 스레드를 막는지는 자동으로 보장하지 않는다. 화면이 느리거나 흔들리거나 깜박이는 문제를 해결하려면 브라우저의 렌더링 파이프라인을 알아야 한다.

HTML에서 픽셀까지

브라우저가 화면을 만드는 과정은 단순히 HTML을 위에서 아래로 그리는 일이 아니다. 대략 다음 흐름을 거친다.

  1. HTML을 파싱해 DOM을 만든다.
  2. CSS를 파싱해 CSSOM을 만든다.
  3. DOM과 CSSOM을 결합해 렌더 트리를 만든다.
  4. 각 요소의 위치와 크기를 계산한다.
  5. 요소의 색, 글자, 테두리, 그림자 등을 페인트한다.
  6. 여러 레이어를 합성해 최종 화면을 만든다.

이 흐름을 정확한 내부 구현 수준까지 외울 필요는 없다. 하지만 어떤 변경이 어느 단계에 영향을 주는지는 알아야 한다. 예를 들어 텍스트 색을 바꾸는 것은 대체로 레이아웃을 다시 계산할 필요가 없지만, 요소의 너비를 바꾸면 주변 요소의 위치까지 다시 계산해야 할 수 있다. transform으로 이동시키는 애니메이션과 top이나 left를 계속 바꾸는 애니메이션은 브라우저가 부담하는 비용이 다르다.

DOM과 CSSOM

DOM은 HTML 문서의 구조를 표현한다. CSSOM은 CSS 규칙을 구조화한 것이다. 브라우저는 이 둘을 함께 사용해 실제로 보이는 요소를 결정한다. display: none인 요소는 DOM에는 있을 수 있지만 렌더 트리에는 포함되지 않는다. 반대로 visibility: hidden인 요소는 보이지 않지만 공간은 차지한다.

이 차이는 UI 버그를 설명할 때 중요하다. 예를 들어 AI가 모달을 만들면서 닫힌 상태의 모달을 단순히 투명하게만 처리했다면, 화면에는 보이지 않아도 포커스나 클릭 흐름에 영향을 줄 수 있다. 접근성과 레이아웃을 함께 고려해야 한다.

CSS도 단순한 장식이 아니다. CSS 선택자의 범위, cascade, specificity, layout model, stacking context는 모두 실제 화면의 동작에 영향을 준다. AI가 만든 스타일이 기존 디자인 시스템의 규칙을 덮어쓰거나, 의도치 않은 stacking context를 만들어 dropdown이 뒤에 깔리는 문제도 자주 생긴다.

레이아웃: 위치와 크기를 계산하는 비용

레이아웃은 각 요소가 어디에 있고 얼마나 큰지 계산하는 단계다. 이 단계는 비쌀 수 있다. 특히 어떤 JavaScript 코드가 DOM의 크기를 읽고 곧바로 스타일을 쓰는 일을 반복하면 브라우저는 최신 값을 알기 위해 레이아웃을 강제로 계산해야 할 수 있다.

예를 들어 다음과 같은 패턴은 조심해야 한다.

items.forEach((item) => {
  const height = item.getBoundingClientRect().height;
  item.style.marginTop = `${height / 2}px`;
});

이 코드는 읽기와 쓰기가 반복된다. 브라우저는 스타일 변경이 다음 측정값에 영향을 줄 수 있기 때문에 불필요한 레이아웃 계산을 하게 될 수 있다. 더 나은 방식은 먼저 필요한 값을 모두 읽고, 그다음 스타일을 한꺼번에 쓰는 것이다.

AI가 스크롤 위치 계산, 드래그 앤 드롭, 가상 목록, 툴팁 위치 조정 코드를 만들 때 이런 문제가 생기기 쉽다. 코드가 맞아 보여도 성능이 나쁠 수 있다.

페인트와 컴포지팅

페인트는 요소의 시각적 표현을 그리는 단계다. 배경색, 텍스트, 테두리, 그림자, 이미지 등이 여기에 포함된다. 컴포지팅은 여러 레이어를 합쳐 최종 화면을 만드는 단계다.

애니메이션을 만들 때는 어떤 속성을 바꾸는지가 중요하다. 대체로 transform과 opacity는 레이아웃과 페인트 비용을 줄이기 쉬운 속성이다. 반면 width, height, top, left, box-shadow 같은 속성을 매 프레임 바꾸면 더 많은 비용이 들 수 있다.

물론 “항상 transform만 써라” 같은 절대 규칙은 위험하다. 모든 최적화는 실제 측정과 함께 판단해야 한다. 하지만 최소한 AI가 만든 애니메이션 코드가 레이아웃을 계속 흔드는지, 브라우저가 매 프레임 많은 일을 하게 만드는지는 확인해야 한다.

이미지와 폰트가 만드는 흔들림

프론트엔드에서 흔한 성능 문제 중 하나는 레이아웃 이동이다. 이미지의 크기를 지정하지 않으면 브라우저는 이미지가 로드되기 전까지 공간을 얼마나 확보해야 할지 모른다. 뒤늦게 이미지 크기가 결정되면 아래 콘텐츠가 밀려난다. 사용자는 버튼을 누르려다가 다른 요소를 누를 수 있다.

폰트도 비슷하다. 웹 폰트가 늦게 로드되면 fallback font로 먼저 렌더링한 뒤 글자 폭이 달라져 레이아웃이 바뀔 수 있다. 이런 문제는 “화면은 결국 보인다“는 관점에서는 작아 보이지만, 실제 사용자 경험에서는 매우 거슬린다.

AI가 카드 목록을 만들 때 img 태그에 width와 height를 지정하지 않거나, 반응형 컨테이너의 aspect ratio를 잡지 않는 경우가 있다. 이때 개발자는 단순히 화면이 예쁜지보다 레이아웃이 안정적인지 봐야 한다.

JavaScript와 렌더링의 관계

JavaScript는 브라우저의 메인 스레드에서 많은 일을 한다. 긴 JavaScript 실행은 렌더링과 사용자 입력 처리를 지연시킬 수 있다. 사용자가 버튼을 눌렀는데 반응이 늦다면, 네트워크만 문제가 아닐 수 있다. JavaScript가 너무 오래 실행 중일 수 있다.

프론트엔드 프레임워크도 결국 JavaScript 위에서 동작한다. 상태가 바뀌면 컴포넌트가 다시 계산되고, DOM 변경이 만들어지며, 브라우저가 렌더링을 갱신한다. 불필요한 상태 변경이나 큰 컴포넌트 트리의 잦은 재계산은 사용자 경험에 영향을 준다.

AI가 만든 코드는 종종 편의를 위해 모든 값을 한 컴포넌트에 몰아넣거나, 렌더링마다 비싼 계산을 다시 수행하거나, 배열 전체를 매번 새로 만드는 식의 코드를 낸다. 작은 예제에서는 문제 없어 보이지만, 데이터가 커지면 문제가 된다.

Core Web Vitals를 읽는 이유

Core Web Vitals는 사용자의 경험을 성능 지표로 바라보려는 시도다. LCP는 주요 콘텐츠가 얼마나 빨리 보이는지, INP는 상호작용에 얼마나 빨리 반응하는지, CLS는 화면이 얼마나 안정적인지를 본다.

이 지표를 숫자 암기 대상으로 볼 필요는 없다. 중요한 것은 각 지표가 어떤 사용자 경험을 나타내는지 이해하는 것이다.

  • LCP가 나쁘다면 주요 이미지, 서버 응답, 렌더링 차단 리소스, 클라이언트 렌더링 비용을 의심한다.
  • INP가 나쁘다면 긴 JavaScript 작업, 무거운 이벤트 핸들러, 불필요한 렌더링을 의심한다.
  • CLS가 나쁘다면 이미지 크기, 광고/동적 콘텐츠 삽입, 폰트 로딩, layout placeholder를 의심한다.

AI에게 “성능을 개선해줘“라고 요청하는 것보다 “이 화면의 LCP를 나쁘게 만들 수 있는 요소를 찾아줘“라고 요청하는 편이 낫다. 성능도 구체적인 질문으로 바꿔야 다룰 수 있다.

AI 생성 코드 리뷰 질문

브라우저 렌더링 관점에서 AI가 만든 UI를 리뷰할 때는 다음 질문을 던진다.

  • 이미지나 비디오의 크기, 비율, placeholder가 안정적으로 잡혀 있는가?
  • 애니메이션이 layout을 계속 바꾸는 속성을 사용하지 않는가?
  • 스크롤, resize, mousemove 같은 빈번한 이벤트에서 비싼 작업을 하지 않는가?
  • DOM 측정과 스타일 쓰기가 반복적으로 섞여 있지 않은가?
  • 닫힌 modal, dropdown, tooltip이 포커스와 stacking context를 망치지 않는가?
  • 렌더링마다 큰 배열 변환이나 비싼 계산을 다시 하지 않는가?
  • 실제 브라우저 성능 도구로 확인할 수 있는가?

이 질문들은 프론트엔드 개발자에게 방어력을 준다. AI가 코드를 빠르게 만들더라도, 브라우저가 그 코드를 어떻게 실행하는지 설명할 수 있으면 통제권을 잃지 않는다.
