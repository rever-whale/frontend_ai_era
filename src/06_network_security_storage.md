# 6. 네트워크, 보안, 스토리지 기본기

프론트엔드는 더 이상 "화면만 만드는 영역"이 아니다. 인증, 세션, 토큰, 쿠키, 캐시, API 에러, 보안 헤더, 개인정보 노출은 모두 프론트엔드 코드와 연결된다. AI가 만든 코드가 겉으로는 편리해 보여도, 토큰을 `localStorage`에 무심코 저장하거나 민감한 값을 클라이언트 번들에 노출하면 심각한 문제가 된다.

이 장의 목표는 프론트엔드 개발자가 보안 전문가가 되는 것이 아니다. 최소한 어떤 질문을 해야 하는지 아는 것이다. 이 값은 브라우저에 저장해도 되는가. 이 API 응답은 캐시되어도 되는가. 이 요청은 인증 정보를 포함하는가. 사용자가 입력한 값은 안전하게 렌더링되는가. 에러 메시지가 민감한 정보를 노출하지 않는가. 이런 질문이 있어야 AI가 만든 코드를 검토할 수 있다.

## HTTP는 UI의 일부다

프론트엔드 개발자는 HTTP를 화면 밖의 일로 생각하기 쉽다. 하지만 HTTP는 UI의 일부다. API가 느리면 로딩 상태가 필요하다. 401 응답은 로그인 흐름과 연결된다. 403 응답은 권한 안내와 연결된다. 404 응답은 빈 상태와 다르다. 500 응답은 재시도와 장애 안내가 필요하다.

상태 코드를 단순히 성공과 실패로만 나누면 사용자 경험이 거칠어진다.

- 200: 정상 응답
- 201: 생성 완료
- 204: 응답 본문 없는 성공
- 400: 잘못된 요청
- 401: 인증 필요
- 403: 권한 없음
- 404: 리소스 없음
- 409: 충돌
- 422: 유효성 검증 실패
- 500: 서버 오류

AI가 만든 API 호출 코드는 종종 `catch`에서 "오류가 발생했습니다"만 보여준다. 실제 제품에서는 응답의 의미를 구분해야 한다. 특히 폼에서는 422 응답이 필드별 에러로 연결되어야 하고, 401은 로그인 갱신이나 재인증으로 이어질 수 있다.

## 캐시는 성능이면서 정확성 문제다

캐시는 빠른 UI를 만드는 데 중요하다. 하지만 잘못된 캐시는 틀린 정보를 보여준다. 프론트엔드 개발자는 브라우저 캐시, HTTP cache header, 서비스 워커 캐시, 서버 상태 라이브러리의 메모리 캐시를 구분할 수 있어야 한다.

예를 들어 상품 목록은 짧은 시간 캐시해도 괜찮을 수 있다. 반면 계정 권한, 결제 상태, 개인정보는 오래 캐시하면 위험할 수 있다. 검색 결과는 query parameter와 함께 캐시 키가 만들어져야 한다. 필터가 바뀌었는데 같은 캐시 키를 사용하면 이전 결과가 계속 보일 수 있다.

AI에게 서버 상태 코드를 만들게 할 때는 캐시 키와 재검증 조건을 명시해야 한다.

```text
검색어, 카테고리, 가격 범위, 정렬, 페이지가 모두 query key에 포함되어야 한다.
즐겨찾기 변경 후 현재 목록 query를 무효화하거나 optimistic update를 일관되게 처리해야 한다.
```

캐시는 성능 최적화가 아니라 데이터 정확성의 일부다.

## CORS는 보안 기능이지 장애물이 아니다

CORS는 브라우저가 다른 origin으로 요청할 때 적용하는 보안 모델이다. 개발 중에는 CORS 에러가 귀찮은 장벽처럼 느껴질 수 있다. 그래서 AI에게 물어보면 "서버에서 모든 origin을 허용하세요" 같은 답을 받을 수도 있다. 그러나 실무에서는 위험한 해결책이다.

CORS를 볼 때는 다음을 확인한다.

- 어떤 origin을 허용해야 하는가?
- credential을 포함하는 요청인가?
- preflight 요청이 필요한가?
- 허용할 method와 header는 무엇인가?
- 운영 환경과 개발 환경의 설정이 분리되어 있는가?

프론트엔드 개발자가 CORS 설정을 직접 담당하지 않더라도, 문제를 정확히 설명할 수 있어야 백엔드나 인프라 담당자와 협업할 수 있다. "CORS가 안 돼요"보다 "프론트엔드 origin은 `https://app.example.com`이고, credential 포함 요청에서 preflight가 실패하고 있습니다"가 훨씬 좋은 설명이다.

## 쿠키와 토큰 저장

인증 정보를 어디에 저장할지는 중요한 결정이다. `localStorage`는 사용하기 쉽지만 JavaScript로 접근 가능하다. XSS가 발생하면 저장된 토큰을 읽을 수 있다. `HttpOnly` 쿠키는 JavaScript에서 읽을 수 없기 때문에 토큰 탈취 위험을 줄일 수 있지만, CSRF와 SameSite 정책을 함께 고려해야 한다.

쿠키를 볼 때 중요한 속성은 다음과 같다.

- `HttpOnly`: JavaScript에서 접근할 수 없게 한다.
- `Secure`: HTTPS에서만 전송한다.
- `SameSite`: cross-site 요청에서 쿠키 전송 정책을 제어한다.
- `Path`와 `Domain`: 쿠키가 전송되는 범위를 정한다.
- `Expires`와 `Max-Age`: 만료 시점을 정한다.

AI가 인증 구현을 제안할 때는 저장 위치를 반드시 확인해야 한다. "JWT를 localStorage에 저장하고 요청마다 Authorization header에 넣는다"는 패턴은 많은 예제에서 보이지만, 모든 서비스에 적합한 답은 아니다. 보안 요구사항, 백엔드 구조, 배포 환경, CSRF 방어 전략에 따라 달라진다.

## 브라우저 스토리지의 차이

브라우저에는 여러 저장소가 있다. 각각의 용도가 다르다.

- `localStorage`: 문자열 기반, 동기 API, 탭을 닫아도 유지된다.
- `sessionStorage`: 탭 세션 동안 유지된다.
- IndexedDB: 더 큰 데이터와 구조화된 저장에 적합하다.
- Cookie: 요청과 함께 서버로 전송될 수 있다.
- Cache Storage: 서비스 워커와 함께 리소스 캐싱에 쓰인다.

AI가 "브라우저에 저장하면 된다"고 제안할 때는 무엇을 어디에 저장하는지 물어야 한다. UI 선호도나 임시 draft는 localStorage에 둘 수 있다. 하지만 access token, 주민등록번호, 결제 정보 같은 민감한 값은 그렇게 다루면 안 된다.

또한 localStorage는 동기 API다. 큰 데이터를 자주 읽고 쓰면 메인 스레드에 영향을 줄 수 있다. 저장소 선택은 보안뿐 아니라 성능과 데이터 모델의 문제이기도 하다.

## XSS와 사용자 입력

XSS는 사용자가 입력한 값이나 외부 데이터를 안전하지 않게 HTML로 렌더링할 때 발생할 수 있다. 현대 프레임워크는 기본적으로 문자열을 escape해주기 때문에 많은 위험을 줄여준다. 하지만 `dangerouslySetInnerHTML`, 직접 DOM 조작, markdown 렌더링, 외부 HTML 삽입, URL 생성에서는 여전히 조심해야 한다.

AI는 사용자의 요구가 "HTML을 렌더링해줘"라면 너무 쉽게 raw HTML 삽입 코드를 제안할 수 있다. 이때 개발자는 다음을 확인해야 한다.

- 이 HTML의 출처는 신뢰할 수 있는가?
- sanitize를 거쳤는가?
- 링크의 `target="_blank"`에 `rel="noopener noreferrer"`가 필요한가?
- URL scheme이 `javascript:` 같은 위험한 값이 될 수 있는가?
- markdown 렌더러가 raw HTML을 허용하는가?

사용자 입력은 단순한 문자열이 아니다. 브라우저에서 실행될 수 있는 컨텍스트에 들어가면 보안 문제가 된다.

## CSRF와 credential 요청

CSRF는 사용자가 로그인한 상태를 악용해 원치 않는 요청을 보내게 만드는 공격이다. 쿠키 기반 인증을 사용할 때 특히 고려해야 한다. SameSite 쿠키 정책은 CSRF 위험을 줄이는 데 도움이 되지만, 모든 상황을 자동으로 해결하지는 않는다. 중요한 변경 요청에는 CSRF token, origin 검증, 적절한 SameSite 정책이 함께 필요할 수 있다.

프론트엔드 개발자는 credential 요청을 만들 때 다음을 확인해야 한다.

- 요청에 쿠키가 포함되는가?
- cross-site 요청인가?
- 서버가 어떤 CSRF 방어를 기대하는가?
- CORS 설정과 credential 정책이 일관되는가?

AI가 fetch 코드에 `credentials: "include"`를 추가했다면, 그것이 왜 필요한지 확인해야 한다. 인증이 되게 하려고 넣은 한 줄이 CORS와 CSRF 문제를 함께 끌고 올 수 있다.

## 클라이언트 번들과 환경 변수

프론트엔드 빌드에서 환경 변수는 조심해야 한다. 클라이언트 번들에 들어간 값은 사용자가 볼 수 있다고 생각해야 한다. 이름이 `SECRET`이어도 브라우저에 포함되면 더 이상 secret이 아니다.

프론트엔드에 들어가도 되는 값은 public API base URL, 공개 가능한 feature flag, 분석 도구의 public key처럼 노출을 전제로 한 값이다. 서버 전용 secret, private API key, 관리자 token은 클라이언트 코드에 들어가면 안 된다.

AI가 `.env` 설정이나 SDK 초기화 코드를 만들 때는 이 경계를 확인해야 한다. 특히 "이 API key를 넣어주세요"라는 예제 코드는 브라우저용 public key인지 서버용 secret인지 구분해야 한다.

## AI 생성 코드 리뷰 질문

네트워크, 보안, 스토리지 관점에서 AI가 만든 코드를 리뷰할 때는 다음 질문을 던진다.

- 인증 정보나 민감한 값이 브라우저 저장소에 들어가지 않는가?
- 클라이언트 번들에 secret이 포함되지 않는가?
- HTTP 상태 코드별 사용자 경험이 구분되는가?
- 캐시 키가 모든 검색 조건과 사용자 맥락을 포함하는가?
- mutation 이후 캐시 무효화나 optimistic rollback이 있는가?
- CORS 문제를 무작정 wildcard로 해결하려 하지 않는가?
- credential 요청과 CSRF 방어가 일관되는가?
- 사용자 입력이나 외부 HTML이 안전하게 렌더링되는가?
- 에러 로그나 에러 메시지가 개인정보를 노출하지 않는가?

프론트엔드 개발자는 보안의 최종 책임자가 아닐 수 있다. 하지만 브라우저에서 어떤 값이 저장되고, 어떤 요청이 나가고, 어떤 데이터가 화면에 들어오는지 가장 가까이에서 보는 사람이다. AI 시대에는 이 감각이 더 중요해진다. 자동생성 코드가 편해 보일수록, 브라우저와 네트워크의 경계를 더 분명히 봐야 한다.

## Part 2 실습과 회고

이 Part의 목표는 브라우저가 실제로 무엇을 하는지 눈으로 확인하는 것이다. AI가 설명한 브라우저 동작을 믿는 데서 멈추지 말고, 개발자 도구와 작은 코드로 확인한다.

실습 과제:

- 본인이 만든 페이지나 익숙한 웹앱 한 화면을 고른다.
- DevTools Performance 패널에서 주요 상호작용 하나를 기록하고, layout, paint, scripting 시간이 어디에서 발생하는지 메모한다.
- Network 패널에서 API 요청 하나를 골라 request header, response status, cache header, payload 모양을 확인한다.
- Application 패널에서 cookie, localStorage, sessionStorage에 어떤 값이 저장되는지 확인한다.
- AI에게 이 기록을 바탕으로 "성능, 보안, 스토리지 관점의 리뷰 질문"을 만들게 하고, 질문 중 부정확한 것을 지운다.

회고 질문:

- 내가 성능 문제라고 느낀 것이 실제로 layout, script, network 중 어디에서 발생했는가?
- 브라우저에 저장된 값 중 서버에 있어야 할 값은 없었는가?
- API 실패 상태가 사용자 경험으로 구분되어 있었는가?
- AI가 만든 네트워크/보안 설명 중 직접 확인하지 않으면 위험한 부분은 무엇이었는가?

완료 체크:

- Performance 기록에서 병목 후보를 하나 찾았다.
- Network 요청 하나의 상태 코드와 cache 정책을 설명할 수 있다.
- 브라우저 저장소에 남은 값을 목록으로 적었다.
- 자동생성 코드 리뷰 질문을 브라우저 동작 기준으로 보강했다.
