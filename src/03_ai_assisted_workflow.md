# 3. AI 보조 개발의 실제 워크플로

AI를 개발에 활용한다는 말은 너무 넓다. 어떤 사람에게는 자동완성이고, 어떤 사람에게는 챗봇에게 질문하는 일이며, 어떤 사람에게는 에이전트가 파일을 수정하고 테스트까지 실행하는 일이다. 이 책에서는 AI 보조 개발을 "개발자가 목표와 경계를 정의하고, AI가 탐색과 생성과 수정의 일부를 맡고, 개발자가 결과를 검증해 제품 코드에 통합하는 작업 방식"으로 본다.

중요한 점은 AI가 개발 루프 안으로 들어왔다는 것이다. 예전에는 개발자가 문서를 읽고, 코드를 작성하고, 테스트하고, 리뷰를 받았다. 이제는 그 사이사이에 AI가 들어온다. 요구사항을 정리할 때, 기존 코드를 읽을 때, 초안을 만들 때, 테스트를 추가할 때, 리뷰 체크리스트를 만들 때 AI가 도울 수 있다.

하지만 도구가 들어왔다고 루프의 본질이 바뀌지는 않는다. 좋은 개발은 여전히 문제 정의, 설계, 구현, 검증, 기록의 반복이다. AI는 이 반복을 빠르게 만들 수 있지만, 반복 자체를 없애지는 못한다.

## 나쁜 워크플로: 큰 요청과 무검증 병합

AI를 처음 도입할 때 흔한 방식은 큰 요청을 한 번에 던지는 것이다.

```text
관리자 대시보드를 만들어줘.
```

이런 요청은 데모를 만들 때는 효과적일 수 있다. AI는 카드, 표, 그래프, 필터, 버튼이 있는 화면을 빠르게 만들어낸다. 하지만 실제 코드베이스에서는 문제가 된다. 어떤 디자인 시스템을 써야 하는지, 데이터는 어디서 오는지, 권한은 어떻게 처리하는지, URL 상태가 필요한지, 테스트는 어떤 수준이어야 하는지, 기존 컴포넌트와 어떻게 연결되는지 알 수 없기 때문이다.

더 위험한 것은 생성된 코드를 충분히 읽지 않고 병합하는 것이다. 빠르게 만들어진 코드는 빠르게 기술 부채가 된다. 특히 프론트엔드에서는 UI가 눈에 보이기 때문에 "대충 된 것 같다"는 착각이 쉽다.

## 좋은 워크플로: 작은 작업과 명시적 완료 조건

좋은 AI 보조 개발 워크플로는 작업을 작게 자른다.

```text
상품 목록 페이지의 필터 상태를 URL query parameter와 동기화하려고 한다.
기존 `useProductSearchParams` 훅과 `ProductFilterPanel` 컴포넌트를 먼저 읽고,
현재 중복 상태가 있는지 설명한 뒤,
기존 패턴을 유지하면서 변경안을 제안해줘.
완료 조건은 뒤로 가기/새로고침 후 필터 유지, 빈 값 제거, 기존 테스트 통과다.
```

이 요청은 길지만 좋은 요청이다. 목표, 변경 범위, 읽어야 할 코드, 유지해야 할 패턴, 완료 조건이 들어 있다. AI가 더 좋은 결과를 내기 쉬울 뿐 아니라, 사람이 결과를 검토하기도 쉽다.

좋은 워크플로는 보통 다음 순서를 따른다.

1. 작업 헌장을 쓴다.
2. 기존 코드 탐색을 먼저 시킨다.
3. 변경 계획을 받는다.
4. 작은 단위로 구현한다.
5. 생성된 코드를 사람이 읽는다.
6. 테스트와 브라우저로 검증한다.
7. 변경 요약과 남은 위험을 기록한다.

## 작업 헌장 작성하기

작업 헌장은 AI에게 주는 작은 명세다. 복잡할 필요는 없지만 다음 항목은 들어가면 좋다.

```text
목표:
사용자 관점에서 무엇이 가능해져야 하는가?

변경 범위:
어떤 화면, 컴포넌트, 파일, API를 다룰 수 있는가?

기존 패턴:
따라야 할 컴포넌트, 훅, 타입, 테스트 스타일은 무엇인가?

완료 조건:
어떤 동작이 확인되어야 끝났다고 볼 수 있는가?

검증 방법:
어떤 테스트, 타입 검사, 브라우저 확인을 해야 하는가?

제약:
새 의존성 추가 금지, 특정 파일 수정 금지, API 변경 금지 같은 조건은 무엇인가?
```

작업 헌장은 AI를 위한 문서이면서 사람을 위한 문서다. 작업을 시작하기 전에 이 정도를 쓰지 못한다면, 사실 사람도 아직 문제를 충분히 이해하지 못한 것이다.

## 탐색과 구현을 분리하기

AI에게 바로 코드를 고치게 하기보다 먼저 탐색을 맡기는 것이 좋다. 예를 들어 다음처럼 요청할 수 있다.

```text
아직 코드는 수정하지 말고, 이 기능과 관련된 파일과 데이터 흐름을 찾아줘.
어떤 컴포넌트가 진입점인지, 상태가 어디에 있는지, 테스트가 어디에 있는지 요약해줘.
```

탐색과 구현을 분리하면 두 가지 이점이 있다. 첫째, AI가 잘못된 파일을 수정할 가능성이 줄어든다. 둘째, 개발자가 코드베이스를 함께 이해하게 된다. AI의 요약이 틀렸다면 구현 전에 바로잡을 수 있다.

탐색 결과는 항상 실제 파일과 대조해야 한다. AI가 "이 훅이 서버 상태를 관리한다"고 말했더라도, 코드를 열어보면 단순한 UI 상태일 수 있다. 요약은 출발점이지 증거가 아니다.

## AI에게 초안을 맡기기 좋은 작업

AI에게 맡기기 좋은 작업은 다음과 같다.

- 기존 패턴을 따르는 컴포넌트 초안
- 타입과 인터페이스 정리
- 테스트 케이스 초안
- Storybook story 또는 사용 예시 작성
- 반복적인 스타일 변환
- 간단한 리팩터링
- 문서화와 체크리스트 작성

이 작업들은 사람이 검토하기 쉽고, 실패했을 때 되돌리기도 쉽다. 특히 기존 코드 패턴이 명확하면 AI는 더 안정적인 결과를 낸다. 따라서 AI 친화적인 코드베이스를 만들고 싶다면, 먼저 사람이 읽기 좋은 코드베이스를 만들어야 한다. 명확한 이름, 작은 컴포넌트, 일관된 파일 구조, 좋은 타입, 테스트가 AI에게도 좋은 컨텍스트가 된다.

## 사람이 반드시 붙잡아야 할 작업

반대로 다음 작업은 사람이 최종 판단을 해야 한다.

- 제품 요구사항의 우선순위 결정
- 사용자 경험의 미묘한 판단
- 인증, 권한, 개인정보 처리
- 아키텍처 경계 변경
- 새 의존성 도입
- 성능과 접근성의 최종 승인
- 배포와 운영 리스크 판단

AI는 선택지를 제안할 수 있다. 하지만 "우리 제품에서 이 선택이 맞는가"는 제품과 팀의 맥락을 아는 사람이 판단해야 한다. AI는 저장소의 모든 암묵적 이유를 알지 못한다. 어떤 코드는 과거 장애 때문에 일부러 그렇게 되어 있을 수 있고, 어떤 패턴은 팀의 역량과 운영 방식 때문에 선택되었을 수 있다.

## 검증 루프를 기본값으로 만들기

AI 보조 개발에서 가장 중요한 습관은 검증 루프를 기본값으로 만드는 것이다. 생성이 끝이 아니라 검증의 시작이어야 한다.

프론트엔드 작업의 검증 루프는 보통 다음을 포함한다.

- 타입 검사
- 린트
- 단위 또는 컴포넌트 테스트
- 브라우저에서 실제 사용자 흐름 확인
- 접근성 기본 점검
- 성능에 영향을 줄 수 있는 변경 확인
- diff 리뷰

항상 모든 검증을 다 할 수는 없다. 하지만 어떤 검증을 생략했는지 알고 있어야 한다. "테스트는 돌리지 못했지만 타입 검사는 통과했고, 브라우저에서 주요 흐름은 확인했다"와 "그냥 AI가 해줬다"는 완전히 다르다.

## 결과 보고 형식

AI와 협업할 때는 마지막 요약도 중요하다. 좋은 결과 보고는 다음 내용을 포함한다.

- 무엇을 변경했는가
- 왜 그렇게 변경했는가
- 어떤 검증을 했는가
- 어떤 검증을 하지 못했는가
- 남은 위험은 무엇인가

이 형식은 사람 개발자에게도 좋다. 코드 리뷰어는 변경의 의도를 빠르게 이해할 수 있고, 나중에 문제가 생겼을 때 추적하기 쉽다.
